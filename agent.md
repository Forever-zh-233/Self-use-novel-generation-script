# Agent 工作守则

本文件是给所有参与本项目的 AI 助手的硬性约束。每次对话开始时必须内化这些原则。

## 第一原则：反 Goodhart

代码**绝不**做创意/风格判断。一旦代码对"写得好不好"有牙齿（能触发重写），模型就会为了通过检查而删内容、改风格——头痛砍头。

**有牙齿的代码检查只允许：**
- 客观事实错误（角色名不一致、源文专名污染、内部编号泄露）
- AI腔关键词命中（无歧义的坏词）
- 字数下限（低于硬底线）

**绝对禁止代码检查：**
- 句长/段长/比例/方差/频次 → 这些只能作为数据传给 reviewer 参考
- "转折不够""情绪不对""节奏太平" → 这是创意判断，只有 reviewer/人类能做
- 任何"命中就删/命中就重写"的风格规则

**判断标准：** 如果一条规则被模型钻空子的方式是"删掉内容"或"写得更平庸"，这条规则就不该有牙齿。

## 第二原则：不硬编码穷举

今天是"骡子"，明天是"马子"，后天是"狗子"。不要试图用代码穷举所有可能的错误变体。

正确做法：给 LLM reviewer 足够的上下文（角色卡、beat、设定），让它自己判断。抓不到的个案可以接受，但**不能因为懒得给上下文就说"没办法"——想办法把上下文喂进去。**

## 第三原则：LLM 的已知缺陷

LLM 数不清字数、算不准数学、记不住长距离细节。

- 禁止让 writer 在正文中数字数/笔画（它会数错）
- 禁止让 writer 做精确计算（距离、时间、金额要靠静态参考表）
- 长距离一致性靠结构化台账，不靠模型记忆

## 第四原则：层层递进约束链

```
骨架(人类) → 卷纲规划师(严格遵守骨架)
            → 弧线规划师(严格遵守卷纲)
            → 章节规划师(严格遵守弧线走向)
            → 写手(严格遵守 beat)
```

每一层的权威内容注入为 `critical, non-compressible`。标题带【硬约束】。下层不可违背上层的明确规定，自由度只在上层没提到的细节里。

## 第五原则：Token 预算意识

| 角色 | 上限 | 当前约 |
|------|------|--------|
| writer input | 60,000 | ~19,000 |
| archivist input | 38,000 | ~12,000 |
| archivist output | 8,000 | ~3,000 |
| story_director | 32,000 | ~7,000 |
| reviewer | 28,000 | ~12,000 |

所有动态数据结构必须有界增长（cap/过期/淘汰机制）。新增注入前先算 token 影响。

## 第六原则：数据必须落盘

- 超时 ≠ 放弃。超时要重试（3次，backoff 5/10/15s）。
- 重试全部失败 → 章节停止，但已有文本已保存。
- 下次启动 → `recover_state_from_completed_articles` 自动补台账。
- archivist 输出的每个字段都必须有对应的 merge 路径，否则就是数据丢失。

## 第七原则：不矫枉过正

加规则前想清楚：这条规则会不会让模型写得更差？

- "系统面板出现时只能写一句" → 过度限制，第一次遇到超自然现象多花几个 beat 反应是合理的
- "禁止所有短句独段" → 过度限制，短句独段是强调手段
- "每章必须有转折" → 过度限制，缓冲章可以安静收束

**加规则的正确姿势：** 针对具体的、反复出现的、明确是坏的模式，给具体反面示例，而不是写宽泛的禁令。

## 第八原则：Prompt 措辞的三个陷阱

**陷阱1：数字上限变配额。** "最多2次""不超过25%""大约三到四成"——模型会精确瞄准这个数字，把上限当目标。
→ 修法：改为描述性禁令（"如果读起来像X就是病"），或全禁，不给数字。

**陷阱2："偶尔可以"变"应该有"。** "偶尔点睛可以""不是每章都要，但每3-4章至少一次"——模型会理解为"大部分时候应该做"。
→ 修法：要么全禁（实际效果是偶尔漏出=刚好），要么只描述什么时候不该做。

**陷阱3：示例变模板。** prompt 里给的具体示例（"远处的犬吠""灶膛灭了的烟味"），模型会反复照抄这些具体意象，而不是理解背后的原则。
→ 修法：示例只用于说明"什么是坏的"（反面示例）。正面示例要么不给，要么明确标注"这只是举例，禁止照搬这些具体意象"。描述原则本身（"落在一个纯粹的感官细节上"），不给具体的好答案让它抄。

**安全的 prompt 形态只有两种：**
1. 全禁（禁止X）
2. 描述坏味道（如果读起来像Y就是病）

不安全的形态：数字上限、"偶尔可以"、"大约N%"、"至少N次"、正面具体示例。

## 第九原则：想办法，不是"没办法"

"reviewer 抓不到" → 那就给 reviewer 更多上下文
"代码检测不了" → 那就换 LLM 检测
"LLM 也检测不了" → 那就改 prompt 让 writer 一开始就不犯
"prompt 改了还犯" → 那就加具体反面示例
"加了还犯" → 那才是真的没办法，接受

不到最后一步不说"没办法"。

- config/prompts/设定文档/源文本/全书骨架/卷纲/chunks → 清除脚本不碰
- MIMO_API_KEY → 只在 .env.local，永远不 echo 明文
