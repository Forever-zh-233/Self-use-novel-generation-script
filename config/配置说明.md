# 模型配置说明

双击 `E:\Novel 1\一键自动写小说.bat` 会读取两个配置：

- `config/models.json`：各个角色用哪个 API、哪个模型。
- `config/run.json`：从第几章开始、一次生成几章、是否 dry-run。

## 最少要填哪里

打开 `config/models.json`，优先填这几处：

1. `providers.openai_main.api_key`
   - 使用 OpenAI 官方接口时填这里。
   - `base_url` 保持 `https://api.openai.com/v1`。
   - `type` 保持 `openai_responses`。

2. `providers.openai_compatible.api_key`
   - 使用第三方 OpenAI 兼容服务时填这里。
   - `base_url` 填服务商给你的 `/v1` 地址，例如 `https://example.com/v1`。
   - `type` 保持 `openai_chat`。

3. `providers.anthropic_main.api_key`
   - 使用 Anthropic/Claude 官方接口时填这里。
   - `base_url` 保持 `https://api.anthropic.com`。
   - `type` 保持 `anthropic`。

4. `roles.writer.provider` / `roles.writer.model`
   - 写手建议用最强模型。

5. `roles.beat_planner.provider` / `roles.beat_planner.model`
   - 自动生成章节 beat。可以用便宜一些但结构化输出稳定的模型。

6. `roles.reviewer.provider` / `roles.reviewer.model`
   - 评审可以用便宜一点但判断稳定的模型。

7. `roles.editor.provider` / `roles.editor.model`
   - 修稿手建议接近写手能力。

8. `roles.archivist.provider` / `roles.archivist.model`
   - 记录员可以用便宜模型，但必须稳定遵守格式。

9. `roles.compressor.provider` / `roles.compressor.model`
   - 上下文超过阈值时才调用。建议用便宜但摘要稳定的模型。

`roles.analyst` 是预留给后续重新分析源文本的角色；默认自动写章不会调用它。

## 字段兼容

脚本同时识别以下写法：

- `base_url` 或 `baseUrl`
- `api_key` 或 `apiKey`
- `api_key_env` 或 `apiKeyEnv`
- `anthropic_version` 或 `anthropicVersion`

如果服务商需要额外请求头，可以在 provider 下加：

```json
"headers": {
  "HTTP-Referer": "https://your-site.example",
  "X-Title": "Novel Pipeline"
}
```

如果服务商要求 `max_completion_tokens` 而不是 `max_tokens`，可以在 provider 下加：

```json
"max_tokens_field": "max_completion_tokens"
```

如果服务商需要固定请求体参数，可以加：

```json
"extra_body": {
  "thinking": { "type": "disabled" },
  "stream": false
}
```

## 防止电脑卡死

默认 `config/run.json` 是一次只生成 1 章。

`start_chapter` 默认是 `"auto"`：脚本会查看 `输出/文章/第NNN章.md`，从下一章继续写。想强制重写某一章时，把它改成数字，例如 `1`。

`apply_archivist_updates` 默认是 `true`：记录员报告会非破坏式追加到 `07-动态状态台账.md` 和 `08-期待账本.md`，并保存到 `台账版本/`。想只看报告不改台账时，改成 `false`。

`artifact_retention` 默认是 `"clean"`：章节成功完成后自动删除本章的 prompt/input/raw/draft/review/gate/editor/archive 过程副产物，只保留 `输出/文章/第NNN章.md`、`runtime/` 当前状态、`07/08` 日志、`台账版本/` 快照和 `beats/chapter_N.json`。如果要排错，改成 `"reports"` 保留评审/门禁/记录员报告，或改成 `"debug"` 保留全部过程文件。章节失败时不会自动清理，方便查现场。

`auto_recover_on_start` 默认是 `true`：如果电脑死机或窗口被强关，下一次启动会先检查 `输出/文章/` 是否有已经落盘但 `runtime/state.json` 尚未记录的章节。若有，脚本会先调用/复用 Archivist 补台账，再继续写下一章。

`min_recover_article_chars` 默认是 `1000`：恢复时如果最后一章正文太短，会判定可能是写坏的半章，停止并要求人工检查，不会直接续写。

`default_context_window_tokens` 和 `context_windows` 控制每个角色的上下文预算。`compress_at_ratio` 默认 `0.8`，意思是模型窗口到 80% 时先压缩再发送。例如 1M 上下文模型可填 `1000000`，脚本会在约 800k 触发压缩。

每个角色单独算预算、单独压缩、单独落盘，不会共用一坨聊天上下文。

脚本内部还有硬限制：单次最多 5 章，并且 bat 和 Python 都有锁文件，防止重复双击后并发启动。

锁文件位置：

- `runtime/novel_pipeline.bat.lock`
- `runtime/novel_pipeline.lock`

确认没有脚本在跑时，才可以手动删除锁文件。

## 终端控制

双击 bat 会打开写小说终端：

- 按 `p`：请求暂停；当前 API 调用结束后停在安全点。再次按 `p` 继续。
- 按 `q`：请求停止；当前安全点退出。
- 也可以手动创建 `runtime/pause.request` 暂停，删除该文件继续。
- 当前进度会写入 `runtime/progress.json`。
