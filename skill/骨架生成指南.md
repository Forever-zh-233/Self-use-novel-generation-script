# 全书骨架讨论与生成 Skill

> 本文件供 agent 在与用户讨论/生成全书骨架时参考。骨架是全书的北极星,所有下游职责(卷纲/弧线/beat/写手)都在它的框架内活动。

## 骨架生成的前提

骨架不是脚本能自动决定的,必须和用户讨论后确定。脚本只负责"按骨架执行",不负责"决定骨架"。

## 结局设计原则(反包饺子)

来自头部网文(诡秘之主/夜的命名术/大奉打更人)验证过的规律:

1. **主角必须变强,不能变废物。** 读者跟了几百章,主角最后变弱/变废是背叛读者。代价不能是"失去力量"。
2. **代价应该是"得到了但享受不了"或"赢了但世界没按你想的变"。** 不是拿走东西,是让得到的东西味道变了。
3. **反派不能是纯恶,但也不能洗白。** 灰色是"他的逻辑自洽但你不认同",不是"其实他是好人"。
4. **世界不会因为主角变好,但主角影响过的那一小片会不一样。** 不是拯救世界,是"我路过的地方,有些人过得好了一点"。
5. **结局的情绪应该是"释然"而不是"悲壮"或"圆满"。** 读者合上书的感觉应该是一口气慢慢呼出来,不是哭也不是笑。
6. **绝不包饺子**:主线收了但生活没收,故事是"读者不再跟着看了"不是"结束了"。

## 长生类小说的结局质感

如果主角是长生者,结局应有"长生的轻与重":
- 不写他多痛苦,写别人都老了他还是那样
- 用"听说"体/他人视角写时间流逝的重量
- 最后一幕和第一章呼应:同样的动作,但中间隔了几百章和百年
- 代价不是失去力量,是"失去了想要什么东西的能力"——什么都能做但什么都不特别想做了
- 身边人的离开是自然的(长大/老去/各走各路),不是集中退场的悲剧

## 角色设计原则

### 角色层级
- **核心角色(2-4人)**: 跨全书,有独立弧线(自己的目标/代价/得失),不只是主角的配件
- **卷级角色(每卷2-3人)**: 本卷有重要戏份,可能跨1-2卷,之后退场或降为背景
- **弧级角色**: 某条短线弧的参与者,弧线结束就退场

### 核心角色必须有的东西
每个核心角色需要独立的:
- **想要什么**(目标,且跟主角的目标有时候冲突)
- **为此付出什么**(代价)
- **最终得到/失去什么**(结局)
- 这三样不能是"为了服务主角剧情",必须是"他自己的故事"

### 让配角"活"起来的结构性做法
- 配角有自己的目标,而且这个目标跟主角的目标有时候**冲突**
- 配角的转变是因为**他自己的利益**,不是因为"被主角感化"
- 配角可以做出主角不认同的选择,而且**不被惩罚**(世界不是围着主角转的)

## 角色完整状态模型(动态追踪)

一个角色在长篇网文里需要追踪的完整状态:

### 力量轴
- 境界/修为(含进度和瓶颈)
- 功法/技能(含熟练度:入门→小成→大成→圆满)
- 法宝/武器(含绑定状态/品阶)
- 血脉/体质(可觉醒/升级)
- 坐骑/灵宠(它们自己也有状态)

### 社会轴
- 宗门/势力归属(会变:加入/叛离/被逐)
- 师承关系
- 仇敌/恩人(原因+强度+是否已了结)
- 道侣/情感关系(阶段)
- 名声/声望(分地区/分势力,不同地方名声不同)
- 称号/身份(含隐藏身份)

### 资源轴
- 寿命(剩余/消耗记录)
- 消耗品库存(丹药/符箓/灵石,用了就少)
- 债务与人情(欠谁的/谁欠你的)
- 因果/业力(如果世界有这个设定)

### 知识轴
- 谁知道什么(已知信息)
- 谁知道谁的秘密(信息差)
- 隐藏身份被谁看穿了
- 誓言/契约(约束未来行为)

### 状态轴
- 存活状态(活/死/伤/封印/失踪)
- 当前伤势/诅咒/debuff
- 外貌变化(伤疤/断肢/变化)
- 当前位置 + 同行者

### 心理轴
- 当前短期目标 vs 长期目标
- 心结/创伤(未解开的)
- 性格弧光当前位置(不能倒退)

## 全书结构

### 体量参考
- 800章左右为一个完整故事
- 8卷,每卷约100章
- 每卷有独立的地域/敌人/高潮,同时推进主线

### 层级规划体系
```
全书骨架(手写锁定,800章)
  → 卷纲规划师(每卷~100章,自动)
    → 弧线规划师(3-10章短线弧,自动)
      → beat_planner(单章,自动)
        → writer(写正文)
```

### 配角轮换模式
- 主角是"走路的人",身边人会换
- 2-3个跨卷核心角色(不一直在身边,但反复出现、各自有线)
- 每卷有卷级角色(本卷重要,之后退场或降为背景)
- 用"非物理聚会"机制让分散角色保持在场(书信/传闻/偶遇)

## 使用方式

1. 用户想讨论骨架时,参考本文件的原则引导讨论
2. 用户确认后,将骨架写入 `全书骨架.md`(JSON格式)
3. 骨架确定后,下游职责(卷纲/弧线/beat)自动在框架内工作
4. 骨架可以微调(±5章的位移),但大方向不能改
