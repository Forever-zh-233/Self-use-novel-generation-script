# 设计稿：新事物登场规划 + 跨章措辞/转场质量（待审核）

> 状态：**设计稿待审核**（2026-06-03）。三层方案，按"数据现成度/收益直接度/与现有机制冲突度"排序。
> 起因：DeepSeek 诊断"第11章伤兵凭空出现"——beat_planner 不知道某角色/物件是不是第一次出现，无法规划入场方式。
> 另含其提出的"跨章措辞重复"和"转场质量"两诉求。

## 0. 核实结论（动手前已读代码，纠正 DeepSeek 一处误判）
- **DeepSeek 对**：beat_planner 当前确实看不到"谁是本章首次登场"，无法规划入场方式。症状成立。
- **DeepSeek 错**：它主张"新建 archivist 已出场清单（方向B）"——但这套数据**早就存在**。ledger 每个实体本就有 `first_chapter` 字段（[state.py:46](scripts/pipeline/state.py#L46) 定义、[archivist.py:184](scripts/pipeline/archivist.py#L184) 写入）。系统一直知道谁是新面孔。
- **真正的缺口**：[ledger_context_for_planner:248](scripts/pipeline/context.py#L248) 给 beat_planner 的实体摘要行**没带 first_chapter**——数据在台账里，就差最后一格没注入。**所以这不是"建新清单"的工程，是"补一个注入字段"的小修补。**

## 第一层：注入"本章首次登场"标记（几乎免费，强烈先做）

### 改什么
[ledger_context_for_planner](scripts/pipeline/context.py#L201) 的实体摘要行（248行），对 `first_chapter == 当前章` 的实体加 `【本章首登场】` 标记。

### 为什么够用
beat_planner 已经在读「正典账本摘要」（[planning.py:994](scripts/pipeline/planning.py#L994) 注入），只是那行摘要漏了登场信息。补上后 beat_planner 一眼可辨新面孔，零新机制、零新注入通道、不增 token（只多几个字）。

### 风险
- `first_chapter` 由 archivist 在角色"已经出场后"才登记——即第N章新角色出场→archivist 跑完才写入 first_chapter=N。**所以 beat_planner 规划第N章时，这个新角色还不在 ledger 里**（它是本章才要引入的）。
- 这意味着：第一层标记只能告诉 beat_planner"最近哪些实体是新来的"（用于回响/避免遗忘），**不能预告"本章你要引入一个新角色"**——因为那个角色此刻还不存在于任何台账。
- **真正要规划入场的新角色，信息来源不是 ledger，而是弧线/卷纲**（它们才规定"第N章某角色登场"）。见第二层。

## 第二层：beat_planner 显式规划入场方式（小改，DeepSeek 入场诉求的真解）

### 核实到的现状
- 弧线的 `beat_hint`/`chapter_drift` 是**自由文本**（如"埋疑：周通深夜烧纸，沈安闻到但没看见"），新角色名混在散文里，**没有结构化"本章首登场：X"字段**。beat_planner 得自己从弧线文本认出谁是新的。
- 配合第一层：ledger 实体摘要里**没有的名字** = 台账没记过 = 大概率新面孔。两个信号合起来，beat_planner 能判断"弧线提到的这个角色，台账里查无此人 → 本章首次引入"。

### 改什么（两处）
1. **beat schema 加可选字段 `登场铺垫`**（[beat_planner.md](prompts/beat_planner.md) 字段区）：
   ```
   "登场铺垫": "本章若有新角色/新势力/新地点首次登场，写它通过什么感官信号提前露面（呼吸声/甲片摩擦/血腥气/脚步），不许凭空出现在主角面前。没有新登场写'无'。"
   ```
2. **beat_planner.md 加一条约束**：
   "新角色/新势力/新地点**首次登场**时（弧线文本提到、且正典账本摘要里查无此名），必须在 `登场铺垫` 里规划入场方式——盲人主角靠非视觉感官（声/触/嗅/温度/空间回响）提前半段铺垫其存在，禁止让新人物'竹杖直接戳到'式凭空出现。"

### 为什么这样设计
- 复用第一层的 ledger 标记做"是不是新的"判断，复用弧线文本做"该不该引入"判断，**不新建任何清单**。
- `登场铺垫` 是**可选字段、默认'无'**，符合 beat_planner.md:114"别填满字段"的克制原则——只在真有新登场时填。
- 入场方式落在 beat（规划层），writer 拿到具体铺垫指令就不会凭空写人——**这正是 DeepSeek 说对的"不是 writer 的责任，是 beat 没规划"**。

### 风险/疑虑
- **判断"是不是新的"有边界**：ledger 是 archivist 事后登记，存在"角色第N章登场、N章的 beat 规划时台账还没他"的窗口——但这恰恰是对的：beat_planner 规划第N章时本就该把他当新人引入。唯一误判是"角色在很早章节一闪而过、archivist 没登记，后来再现时被误判为新登场"——概率低，且重复铺垫一次入场的危害远小于凭空出现。
- 与现有 `chunk_人物初登场.md`（writer 的登场手法卡，[context.py:811](scripts/pipeline/context.py#L811) 关键词点亮）**职责互补不冲突**：那张卡教 writer"怎么写登场"，本层管 beat"规划谁登场、靠什么感官铺垫"。两层分工正确。

## 第三层：跨章措辞重复 + 转场质量（有真缺口，但与克制原则冲突，建议暂缓）

DeepSeek 提的两个诉求，核实后都是**真缺口**，但都和"别给 beat 加字段"的克制原则打架。逐个说。

### 3A. 跨章措辞重复（"膝盖软了一下"连续几章）
- **现有机制 `recent_scene_devices_digest`（[planning.py:733](scripts/pipeline/planning.py#L733)）数的是 beat 字段层**（场景类型/具体动作/具体物件标签），**抓不到 writer 写出的正文措辞**——"膝盖软了一下"是 writer 的遣词，beat 里没这个标签。**所以这是真缺口，归属"正文措辞层"，现有机制覆盖不到。**
- **但解法不该是 beat 加"上章关键短语"字段**：beat 是规划层，它不该、也不擅长管 writer 的具体遣词。更对的落点是**给 writer 注入"上一章正文里高频出现的特征短语，本章避开"**——类似 hooks_digest 的做法，但作用于正文 n-gram，受众是 writer 不是 beat_planner。
- **代价**：要新写一个"上章正文特征短语提取"函数（n-gram 统计 + 停用词过滤），注入 writer。中等工作量，且 mimo 在长输入下未必严格遵守"避开某短语"。**ROI 存疑。**

### 3B. 转场质量（"沈安站了一会儿""他往前走"填充过渡）
- **现有 [gates.py:610-648](scripts/pipeline/gates.py#L610) 的转场检查只查穿帮**（夜→昼缺过渡、入睡没醒来），**不查"填充过渡"的质量**。所以也是真缺口。
- **但 DeepSeek 的方案"beat 加转场方式字段"直接撞 beat_planner.md:114 的克制原则**（大部分字段该写'无'，填满=凑数）。再加一个必填转场字段，会让 beat 更臃肿。
- **更对的落点**：转场质量是**文笔问题，归 writer/reviewer**。要么在 writer 的"本章写作要点"里按需点亮一张转场手法卡（已有 `chunk_转场.md`，[context.py:815](scripts/pipeline/context.py#L815) 关键词点亮），要么交给 reviewer 维度。**不该固化成 beat 字段。**

### 第三层结论（我的建议）
- **3A 和 3B 都是真问题，但都不该用"给 beat 加字段"解决**——那违背克制原则、且把文笔问题错放到规划层。
- 建议：**暂缓**。先观察第一+二层落地后，"凭空出现"是否缓解；跨章措辞/转场质量这两个文笔问题，更适合等"换更强 writer 模型"或"reviewer 维度强化"时一并处理，而非现在堆 beat 字段。
- 若一定要现在做 3A：走"上章正文特征短语→注入 writer"，**不碰 beat schema**。3B 走"点亮已有 chunk_转场.md"，同样不碰 beat schema。

## 我的核心疑虑（供你拍板）
1. **范围**：DeepSeek 把三件独立的事打包成"一个问题"，其实优先级差很远。第一+二层数据现成、收益直接、不违背任何原则；第三层是文笔问题、与克制原则冲突、ROI 存疑。我倾向**只做一+二**。
2. **第一层的能力边界**：它标记的是"最近的新面孔"（用于回响/防遗忘），不是"预告本章要引入谁"（那靠弧线文本）。两层配合才完整。要确认你认可这个分工。
3. **第三层要不要现在碰**：我的判断是暂缓，但如果你觉得"凭空出现"和"填充过渡"是当前最痛的，可以只做 3A 的 writer 注入版（不碰 beat）。
