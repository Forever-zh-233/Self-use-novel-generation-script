# Hook 自查改写后方向校验补强计划

## 结论

`hook_self_check` 已经接入并生效，但它改写 `章末钩子/钩子型` 后没有重新经过故事总监方向校验、弧线方向校验或钩子型一致性检查。它的职责是“去重换意象”，但实际能改掉原本承接弧线/总监批注的钩子。

## 当前证据

- `scripts/run_pipeline.py:329-361`：
  - hook 自查输入只看最近钩子和本章钩子。
  - 若返回 JSON，就直接覆盖 `beat["章末钩子"]` 和 `beat["钩子型"]`。
  - 覆盖后直接 `dump_json(beat_path, beat)`，没有再次调用 `beat_direction_check()`。
- 当前 75 章扫描：
  - hook 自查改写次数：43。
- 第23章样例：
  - raw beat 原钩子是码头脚夫提到窑厂路封、巡夜司拦路，能承接 story_director 对 F-034/码头/窑厂方向的要求。
  - hook 自查改成“脚夫小腿旧疤像烙铁烫出来的字”，并标 `钩子型=渴望`。
  - 改写后的内容更像悬念，且从“窑厂路封/巡夜司”转成“旧疤烙字”，方向发生变化。
- 第 96 章同类样本：raw hook 原本承接路/南边雾/回望方向，见 `beats/_debug/第096章/beat_raw.md:19-22`；hook self check 后改成“木牌/娘的面孔”等新对象，见 `beats/_debug/第096章/hook_self_check.md:2-3`、`beats/chapter_96.json:27,29,45`；final/summary/ledger 后续新增 F-233/F-234，见 `输出/文章/第096章.md:199-213`、`runtime/summaries/chapter_096.json:52`、`runtime/active_threads.md:353-354`、`08-期待账本.md:996-997`。说明 hook 自查能替换核心钩子对象，但 `伏笔操作` 仍停在旧计划。

## 影响

- 去重模块可能绕过本章核心方向，把已经对齐的钩子换成另一个新悬念。
- `钩子型` 可能与实际内容不匹配，影响后续 `recent_hooks_digest()` 的去重判断。
- 因为正式 beat 保存的是改写后结果，writer/reviewer 后续只能看到改写后的版本。

## 修复建议

1. 限制 hook 自查的职责边界：
   - 只允许替换意象/表达方式。
   - 不允许替换钩子的核心对象、伏笔编号、方向目标。
2. 改写后增加轻校验：
   - 新旧钩子必须保留同一核心指向，如同一地点/人物/伏笔操作。
   - `钩子型` 必须属于既定枚举。
   - 若故事总监 severity>=2 或本章有明确 arc 指令，改写后重新跑 `beat_direction_check()`。
   - 若新钩子引入新 F-ID、新物件、新关系或新地点，必须同步触发 Beat-Final-Summary/Archivist 对账，而不是只覆盖 `章末钩子`。
3. 保存审计信息：
   - debug 中记录 old_hook/new_hook/change_reason。
   - 若改写被拒，保留原钩子并记录原因。
4. 增加 scenario 测试：
   - 模拟 hook 自查返回偏离原核心对象的新钩子，断言不能覆盖。
   - 模拟仅替换重复意象但保留同一核心对象的新钩子，断言允许覆盖。

## 验收标准

- hook 自查不能把弧线/总监点名的钩子方向替换掉。
- 改写后的 `钩子型` 可校验且与枚举一致。
- debug 留档能还原改写前后差异。
