# 主流程启动与运行态控制契约测试计划

> 状态：待审核。本文件只记录治本方案，未修改代码或运行产物。

## 结论

现有测试多是直接 import 函数或跑局部链路，缺少覆盖 `main()` 真实启动路径的契约测试。`.env.local` 加载、锁、暂停/停止、progress 写入、恢复入口等只在真实 CLI 运行中串起来，因此“函数存在但主流程没接上/顺序错/状态错”的问题容易漏。

## 当前证据

- `scripts/run_pipeline.py:1656-1670`：主入口先加载 `.env.local`，再解析 CLI。
- `scripts/run_pipeline.py:1676-1684`：Analyst 分支独立 acquire/release lock。
- `scripts/run_pipeline.py:1696-1700`：章节主流程 acquire lock 后清理旧 stop 并进入恢复。
- `scripts/pipeline/core.py:67-80`：`.env.local` 加载逻辑在 core 中。
- `scripts/pipeline/core.py:192-197`：progress 写入。
- `scripts/pipeline/core.py:229-241`：pause/stop 控制。
- `scripts/pipeline/core.py:275-297`：主流程锁 acquire/release。
- Turing 审计确认：现有测试主要 direct-call `generate_chapter_final()`、`run_one_chapter()` 或规划子函数，绕开真实 `main()` 启动路径。

## 根因判断

- 测试分层把 `scenario` 定义成角色/模块小链路，没有单独定义“主入口契约测试”。
- CLI 入口、副作用文件、运行控制位和恢复前置动作缺少可替身化 harness。
- 真实启动阶段的失败通常不是某个纯函数错，而是执行顺序和全局状态组合错。

## 影响

- API key 没加载、陈旧锁处理、停止文件残留、progress 状态写错等问题可能只在真实运行时暴露。
- `main()` 分支之间的差异（章节/analyst/outline）不会被测试捕捉。
- 后续新增启动前门禁时，也可能“函数写了但入口没调用”。

## 治本方案

1. 增加主入口 harness。
   - 使用临时 workspace / monkeypatch 路径，模拟 `argv`、env、lock、pause/stop、run config。
   - mock API 调用，禁止生成真实正文。
2. 建立启动阶段契约用例。
   - `.env.local` 注入优先级。
   - acquire lock 必须在恢复/生成前发生。
   - stop.request 的处理策略一致。
   - progress.json 在关键 stage 写入期望字段。
3. 覆盖三条 CLI 分支。
   - 普通章节主流程。
   - `--analyst`。
   - `--outline`。
4. 将测试索引显式标注为 `entrypoint` 层。
   - 区分 `unit`、`scenario-subchain`、`entrypoint-scenario`。

## 验收标准

- `main()` 启动顺序有测试护栏。
- 新增启动门禁若未接入入口，测试会失败。
- `.env.local`、锁、暂停/停止、progress 等运行态控制不再只靠真实运行发现问题。
