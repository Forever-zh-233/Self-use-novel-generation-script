# 主流程调度层场景测试补强计划

## 结论

现有测试覆盖了不少零件函数和局部链路，但对 `main()` 章节循环、规划角色触发、`ensure_beat()` 全链路、best_of_n、fresh editor、summarizer 调用、Archivist 失败停机等主流程调度层覆盖不足。之前 `previous_arcs_summary` 这类主流程错误能漏出来，根因之一就是测试主要咬住局部函数，没有咬住“角色是否真的按顺序接上、失败是否停在正确位置、产物是否落到约定路径”。

## 当前证据

- `scripts/run_pipeline.py:1700-1768`：`main()` 负责恢复、章节循环、VolumePlanner、StoryDirector、ArcPlanner、BeatPlanner、run_one_chapter、重试和停机。
- `tests/scenario_test.py:384-418`：现有恢复场景直接调用 `run_one_chapter()`，没有覆盖 `main()` 启动恢复入口。
- `tests/scenario_test.py:820-865`：ArcPlanner 相关测试覆盖 `needs_arc_planning()`、`build_arc_input()` 和手写合并，但不调用 `run_arc_planner()` 的角色调用、JSON 解析、落盘、side_characters 摄入。
- `scripts/run_pipeline.py:284-373`：`ensure_beat()` 包含 beat 调用、方向复查、重生成、hook 自查、debug 留档，现有测试只覆盖部分方向检查/留档，不覆盖完整接线。
- `scripts/run_pipeline.py:559-588`：best_of_n 会多次调用 writer、评分、写赢家到 `draft.md`，缺少主流程回归。
- `tests/scenario_test.py:468-485`：Editor 只覆盖已有 `edited.md` 时跳过 editor，没有覆盖 reviewer/gate 触发新 editor 调用和 `editor_input` 落盘。
- `tests/scenario_test.py:1732-1810`：Summarizer 测试覆盖解析和消费函数，没有覆盖 `run_one_chapter()` 成功后会调用 summarizer、失败时如何容错。
- `tests/scenario_test.py:422-488` 覆盖了 draft/review/editor 缓存复用和太短重跑，但没有覆盖 beat、state、ledger、prompt、run_cfg 变化后旧阶段产物失效。
- 现有 FactChecker 测试主要覆盖输入构造/语义边界，没有覆盖 `generate_chapter_final()` 内事实核查、自动修复、二次验证残留后的阻断策略。
- 现有 Archivist 测试覆盖结构化 merge 和正文已存在时补台账，但没有模拟 `apply_archivist_update()` 连续失败后必须停章、不推进状态、下次可恢复。
- 现有 Editor 测试覆盖缓存跳过，但缺少 `needs_revision=true` 必须调用 editor、`needs_revision=false` 必须跳过 editor、editor 输出过短/污染后 final_gate 处理的主流程场景。
- `scripts/run_pipeline.py:1676-1694`：`--analyst`、`--outline`、`--count`、`--beat`、`--dry-run` 等 CLI 分支只在 `main()` 内接线，现有 scenario 主要直接调用内部函数。
- `scripts/run_pipeline.py:355-360`：hook 自查非 `OK` 时会尝试解析 JSON 并覆盖钩子，现有测试只测 `recent_hooks_digest`，没有覆盖 hook 自查返回坏 JSON、空 JSON、改写后方向再校验。
- `scripts/pipeline/planning.py:1195-1200`：ArcPlanner JSON 解析失败只日志返回；`scripts/run_pipeline.py:1735-1751` 未检查规划是否成功，仍可继续生成 beat 和正文。
- 第 101 章跨卷样本暴露了调度层缺少边界场景：
  - `scripts/run_pipeline.py:1731-1743` 虽然在 `ensure_beat()` 前安排了 VolumePlanner/ArcPlanner 触发点，但没有一个场景测试断言“当前章超过卷纲 end 时，必须先提交目标卷纲，否则不得生成 beat”。
  - `scripts/run_pipeline.py:284-286` 已有 beat 直接返回，绕过卷纲水位、故事总监、active_arcs 与当前章的使用前校验。
  - `scripts/pipeline/planning.py:1394-1408` `build_beat_input()` 直接读取 `卷纲/10-卷纲.md` 和 `active_arcs_for_beat()`，没有调度级 guard 阻断“第101章目标 + 第一卷97-100卷纲 + ARC-007旧弧线”的组合。
  - `输出/评审/第101章_review.md:91` 对错位上下文评价为“完全服务卷纲，没有偏航”，说明下游 reviewer 不能替代调度层边界门禁。
- `tests/scenario_test.py:181`：测试 run_cfg 可设 `sleep_seconds_between_calls=0`，但没有 monkeypatch `time.sleep` 断言固定等待确实受配置控制。
- `TESTING.md:27` 仍把 `scenario` 描述为“写手上下文、衔接检查、记录员结构化合并、评审输入”等真实小链路，没有把 `main()`/CLI/规划角色触发/章节循环列为明确子类。
- `测试文件维护指南.md:86` 对 run pipeline 长流程的建议是“先拆成 quick/scenario 小链路”，容易被理解成永远不用覆盖主调度入口。
- `scripts/run_tests.py:27-31` 只有 `check/quick/scenario/all` 四类，`scenario` 全塞进一个模块，没有显式区分 `context/planning/dispatch/commit/analyst/cleanup`。
- `TESTING.md:99-103` 与 `测试文件维护指南.md:265-272` 的覆盖索引仍停在旧的 writer/reviewer/archivist 小链路，没有反映 `tests/scenario_test.py` 中已有的大量 planning/analyst/cleanup 场景，也没有标出 `main()` 仍未覆盖。
- 卷切换章还缺少 StoryDirector 强制复评场景：
  - `scripts/run_pipeline.py:1731-1734` 顺序上会在 BeatPlanner 前调用 VolumePlanner 和 StoryDirector。
  - `scripts/pipeline/planning.py:244-254` 但 StoryDirector 是否真实调用模型取决于缓存/间隔/弧线触发，未命中时会沿用 previous。
  - `scripts/run_pipeline.py:1734` 没有在卷纲刚提交或卷号变化时传 `force=True`。
  - `beats/_debug/第101章/beat_input.md:19-23` 显示第 101 章 BeatPlanner 前仍是“尚未触发故事总监审核”；`runtime/story_director.md:3,7,10` 到第 102 章才事后解释第 101 章是第二卷过渡。

## 根因判断

- 测试分层是合理的，但缺少一组“调度层 scenario”：mock LLM、真实临时工作区、跑主流程入口/准入口，断言角色调用顺序和产物契约。
- 高风险分支多数在主流程 glue code 中，不在纯函数里。
- 测试维护指南已经规定“角色链路/上下文/台账写入变了 -> 改 scenario”，但当前新增角色和分支没有都补齐 scenario。
- 测试维护文档和覆盖索引没有把“主流程调度层”作为一等测试面，维护者后续仍可能只补局部函数测试，继续漏“写了但没接上”“接错顺序”“失败后仍推进”。

## 影响

- 角色名错、函数未导入、调度顺序错、产物路径错、失败后继续推进等问题容易漏测。
- 高成本 best_of_n 分支可能一打开才发现赢家没落到标准 draft。
- 自动规划角色可能“写了 prompt/代码但没有真实接上主流程”。

## 治本方案

1. 增加主流程调度 scenario 分组。
   - 使用 `isolated_workspace()`。
   - mock `call_role()`、`time.sleep()`、必要的 gate 函数。
   - 不调用真实 LLM。
2. 覆盖 `main()` 最小章节循环。
   - StoryDirector/ArcPlanner/ensure_beat/run_one_chapter 顺序正确。
   - 某章失败重试后仍失败，断言停在本章、不跳章。
   - 覆盖 `--count`、`--beat`、`--dry-run`、`--analyst`、`--outline` 的路由、锁释放和不误跑章节循环。
   - 当前章超过现有卷纲 end 时，VolumePlanner 输出未覆盖目标章范围必须停机，不得进入 `ensure_beat()`。
   - 卷纲 hash 或卷号变化后，StoryDirector 必须 `force=True` 重新审核；失败时不得进入 `ensure_beat()`。
   - 当前章超过 active arc span/end 时，旧弧线必须归档/隐藏并触发新弧线规划，不得作为 beat 硬约束。
   - `needs_arc_planning(chapter)=true` 但 ArcPlanner 输出坏 JSON 时，章节循环必须停在规划阶段，不得继续 beat/writer。
3. 覆盖规划角色落盘。
   - VolumePlanner 触发与不触发。
   - VolumePlanner 错卷输出、旧卷输出、缺章节范围输出都不得提交到正式卷纲，也不得让下游使用旧卷纲继续。
   - StoryDirector 状态保存、过期复用、模型失败默认态。
   - ArcPlanner JSON 解析、补线合并、side_characters 摄入。
   - ArcPlanner 解析失败、schema 缺字段、空数组等失败类型要分别断言阻断或重试策略。
4. 覆盖 ensure_beat。
   - 缺 beat 自动生成。
   - 生成 beat 前校验目标章落在当前卷纲范围内；不满足时抛出明确错误。
   - 已有 beat 复用前也要跑卷纲水位、active_arcs 过期、story_director 纠偏等使用前校验。
   - 方向失败后重生成。
   - hook_self_check 改钩子后落盘。
   - hook_self_check 返回坏 JSON/空 JSON 时不应污染 beat；改写后要再过必要方向/格式校验。
   - dry_run 不调用 API。
5. 覆盖正文生成高风险分支。
   - POV writer 路由和 reviewer 授权。
   - 普通 writer 真实输入包含【本章写作要点】时，断言该 section 落到 `writer_input.md`，不是只测 `writer_focus_modules()` 局部返回。
   - POV writer 真实输入覆盖：安全写作模块接入、知识边界保留、主角专属盲感官/秘密不泄露。
   - writer module 触发正负样例：对话、潜台词、张力、角色别名。
   - best_of_n 多候选、评分、赢家 `draft.md`。
   - fresh editor 被触发并写 `editor_input.md`。
   - reviewer `needs_revision=true` 必须触发 fresh editor；`needs_revision=false` 且 gate 通过必须跳过 editor。
   - review 缓存 hash 不匹配时必须重跑 reviewer；edited 缓存 hash 不匹配时必须重跑 editor。
   - 最终 `writer/final.md` 和 `输出/文章` 路径都落盘。
6. 覆盖提交尾段。
   - Summarizer 被调用但失败不吞掉阻断问题。
   - Archivist 提交失败两次后停机，不推进完成状态。
   - 恢复路径中完整 Archivist 报告应复用，不完整报告应重调；重调仍失败不得 apply。
   - `sleep_seconds_between_calls=0` 时 writer/reviewer/editor/archivist 固定等待不发生；非 0 时等待有清晰调用点。
   - summary 先成功、Archivist 后失败时，下一章消费者不得读到未提交 summary。
   - final/summary/archive 的 hash 和提交状态必须一致，不允许不同输入版本拼成一章。
   - final_gate hard issue 默认阻断，不得进入 summarizer / archivist。
   - fact_checker 最终验证仍有残留时默认阻断，不得“接受现状”后提交。
   - 断点续跑已有正文时，必须有 final commit 凭证；只有长正文无凭证不得补台账。
   - 已有 beat 文件也必须跑使用前校验；未吸收 story_director severity>=2 纠偏时不得直接复用。
7. 更新测试维护文档。
   - `TESTING.md` 和 `测试文件维护指南.md` 明确 `scenario` 下有 `dispatch` 子类。
   - 凡改 `main()`、CLI 路由、规划角色触发、章节重试/停机、章节提交尾段，都必须补 mock LLM + 临时 workspace 的调度场景。
   - 覆盖索引必须列出已覆盖与明确未覆盖的主流程面。

## 验收标准

- `scripts/run_tests.py scenario` 能覆盖主流程调度层的角色调用顺序和关键产物路径。
- 新增主流程角色或分支时，测试会要求补调度场景。
- 类似 `previous_arcs_summary` 未导入、角色写了未接、失败后仍推进等问题能在测试中暴露。
- 测试维护说明不再把主流程调度风险降级成局部小链路风险。
