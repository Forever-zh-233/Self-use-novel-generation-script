# 主角实体与能力体系配置化计划

> 状态：待审核。本文件只记录治本方案，未修改代码或运行产物。

## 结论

主角境界/能力窗口读取逻辑写死 `ledger.entities.沈安`，境界 fallback 又写死“叩门”。能力体系则默认存在 `config/power_scaling.json` 和修炼境界序列。换书后，真实主角不叫沈安或作品没有修炼体系时，规划与写作仍会按当前书的玄幻能力窗口运行。

## 当前证据

- `scripts/pipeline/context.py:614-628`：`current_mc_realm()` 注释和代码都以 `ledger.entities.沈安.realm` 为权威源，最后 fallback `叩门`。
- `scripts/pipeline/context.py:1138-1157`：`power_scaling_for_chapter()` 再次读取 `ledger.entities.沈安`，并以 `叩门` 为默认境界。
- `scripts/pipeline/context.py:1147`：能力窗口固定使用 `REALM_ORDER_WITH_MORTAL[:7]`。
- `config/power_scaling.json` 当前内容包含沈安特有能力和当前书修炼体系。

## 根因判断

- “主角实体”没有从 profile 解析，而是散落在人名硬编码里。
- “能力体系”被默认视为所有书都有修炼境界，缺少 enabled/type/schema。
- fallback 选了当前书境界，掩盖了配置缺失；正确行为应是缺 profile 时显式告警或跳过该模块。

## 影响

- 换主角名后读不到真实主角境界，能力窗口可能长期卡在“叩门”。
- 非修炼体系作品仍会收到修炼/境界/战力限制提示。
- 当前书主角特有能力可能进入新书 Writer 输入。

## 治本方案

1. 在 book profile 定义主角实体。
   - `protagonist.name`
   - `protagonist.entity_id`
   - `default_realm` 或 `default_power_state`
2. 能力体系显式配置化。
   - `power_system.enabled`
   - `power_system.type`: cultivation / magic / tech / none
   - `realm_order` 或 `stages`
   - `scaling_file`
   - `fallback_policy`
3. 统一读取入口。
   - `current_mc_realm()` 改为 `current_protagonist_power_state()`。
   - 所有能力窗口、门禁、规划提示都走同一函数。
4. 禁止当前书 fallback。
   - profile 缺失时返回空并记录告警，不返回“叩门”。
   - `power_system.enabled=false` 时不注入能力窗口。
5. 增加测试。
   - 主角名为“林岚”时，读取 `ledger.entities.林岚`。
   - 非修炼书 profile 下，writer/beat 输入不含境界窗口。
   - 当前书 profile 下仍能读沈安境界并注入能力边界。

## 验收标准

- 主角实体和能力体系只有一个 profile 权威源。
- 非修炼作品不会被默认注入修炼境界。
- 缺配置时显式告警，不静默回退到当前书设定。
