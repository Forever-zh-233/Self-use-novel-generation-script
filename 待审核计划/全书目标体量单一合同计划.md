# 全书目标体量单一合同计划

> 状态：待审核。本文件只记录治本方案，未修改代码或运行产物。

## 结论

全书目标体量在不同入口中互相矛盾：`master_outline` prompt 建议 3-6 卷、200-400 章，骨架 skill 和当前骨架按 8 卷、约 800 章设计。未来新书跑 `--outline` 时，模型会从第一步就收到错误体量先验，影响卷数、卷长、伏笔跨度和 token 预算。

## 当前证据

- `prompts/master_outline.md:17`：示例 `total_volumes=5`。
- `prompts/master_outline.md:68`：约束建议 `3-6 卷`、每卷 `40-80 章`、总章数 `200-400 章`。
- `skill/骨架生成指南.md:93-99`：指南以 `800章`、`8卷`、每卷约 `100章` 为完整故事参考。
- `全书骨架.md:4-5`：当前骨架为 `total_volumes=8`、`total_chapters_target=800`。

## 根因判断

- 体量是 book/project contract，却散落在 prompt 示例、skill 指南和当前骨架中。
- `generate_master_outline()` 输入没有显式 `target_length/volume_count/chapter_budget` 合同。
- 下游卷纲、弧线、伏笔回收窗口、token 预算都依赖体量，但没有单一权威源。

## 影响

- 新书自动骨架可能被压缩成 200-400 章，不符合用户目标的长篇精品生成。
- 伏笔计划、角色成长、核心机制揭示会按短篇幅设计，后续 800 章规模会断档或拖空。
- token 预算和长线状态管理无法根据真实体量预估。

## 治本方案

1. 建立 `story_scale_contract`。
   - 字段包括 `target_chapters`、`target_volumes`、`chapters_per_volume_range`、`serialization_mode`、`arc_length_policy`、`foreshadowing_horizon`。
   - 来源为 book profile 或新书初始化配置。
2. MasterOutline prompt 参数化。
   - 删除固定 `3-6卷/200-400章` 默认建议。
   - 运行时渲染当前项目的目标体量。
3. Skill 和骨架脚本读取同一合同。
   - 指南只解释如何使用体量合同，不再给当前书专属数字作为默认。
4. 下游消费体量合同。
   - VolumePlanner 校验卷范围。
   - ArcPlanner 校验短弧长度和节点密度。
   - 伏笔窗口和 token 预算按目标体量分层。
5. 增加一致性测试。
   - prompt、skill、正式骨架中的体量字段必须一致或来自同一 profile。
   - 目标 800 章时，`--outline` 输入不得出现 200-400 章默认建议。

## 验收标准

- 全书体量只有一个机器权威源。
- 骨架生成、卷纲生成、结构参考和 token 预算使用同一体量合同。
- 换书时可配置短篇/中篇/长篇，不需要改 prompt 常量。
