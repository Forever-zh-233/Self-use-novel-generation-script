# 全角色结构化输出契约与重试分类计划

## 结论

当前每个 LLM 角色各自用 prompt 示例、手写解析器、局部 validator、局部 retry/fallback 来约束输出。问题不是某个字段单独漏了，而是系统缺少统一的 `StructuredOutputContract`：prompt 要了，parser 不保留；parser 保留了，consumer 不读；consumer 读了，validator 不拦缺失；validator 拦了，上层重试和降级口径又不一致。

## 当前证据

- `prompts/beat_planner.md:21` 要求只输出 JSON，`prompts/beat_planner.md:62` 附近列出多个必填字段；但 `scripts/run_pipeline.py:89`、`scripts/run_pipeline.py:119` 的 `normalize_beat()` 仍靠手写白名单保留字段。
- `prompts/archivist.md:328`、`prompts/archivist.md:337`、`prompts/archivist.md:412` 要求 Archivist 每章填写 `dominant_strand`、`cultivation_active`、`timeline_update`；`scripts/pipeline/archivist.py:975-991` 的校验只确认 `STRUCTURED_UPDATE` 存在且 JSON 可解析。
- `prompts/reviewer.md:219-233` 要求 reviewer JSON 含 `needs_revision`、`scores`、`blockers`；`scripts/pipeline/gates.py:547-558` 只要求能读到 `needs_revision`，解析失败还会回退关键词，关键词没命中即默认放行。
- `scripts/run_pipeline.py:599-612` 的 `resume_verdict()` 在 `review_verdict.json` 与 `review.md` 都缺失时返回 `source=resume_default` 且 `needs_revision=False`；恢复路径会把“缺判定”当“无需返工”。
- `scripts/pipeline/summarizer.py:63-83` 中 `_parse_summary()` 解析失败时使用空摘要并 setdefault，缺少 `_schema_error` 或不可信状态。
- `scripts/pipeline/api.py:298-314` 对超时/截断有统一重试和 partial fallback；但 `scripts/pipeline/planning.py:307-317` 中 StoryDirector 异常默认放行，`scripts/pipeline/planning.py:1198-1199` 中 ArcPlanner JSON 解析失败只日志后继续。
- `scripts/pipeline/planning.py:1195-1200` 中 ArcPlanner JSON 解析失败只打印“弧线未更新”，`scripts/run_pipeline.py:1735-1751` 随后仍继续 `ensure_beat()` 与 `run_one_chapter()`；这会让“需要新弧线规划”的章节无规划继续提交。
- `scripts/run_pipeline.py:1731-1743` 对 VolumePlanner/StoryDirector/ArcPlanner 的调用都没有统一返回 `PlanningResult`；主流程无法区分“未触发”“成功提交”“结构失败”“语义失败但沿用旧状态”。
- `scripts/pipeline/planning.py:637-644` VolumePlanner 输出会先写 runtime 再覆盖正式卷纲，但缺少目标章范围、卷号、连续性等语义校验结果供主流程阻断。
- `scripts/run_pipeline.py:811` 依赖 FactChecker 文本格式正则；格式漂移可能被当成 0 穿帮。
- `prompts/master_outline.md:12` 给出全书骨架 JSON 要求，但 `scripts/pipeline/planning.py:370` 生成后直接落盘；`prompts/volume_planner.md:91` 要纯 Markdown，`scripts/pipeline/planning.py:637` 直接替换卷纲，缺少落盘前结构校验。

## 根因判断

- schema 分散在自然语言 prompt、解析函数、消费者代码和测试样本里，没有单一契约源。
- “可解析”经常被当成“可提交”，字段缺失、类型错、语义冲突没有统一失败分类。
- 不同角色对同类失败采取不同策略：重试、默认放行、partial、空摘要、日志后继续。
- Reviewer “缺判定/坏判定”、ArcPlanner “结构坏”、FactChecker “格式漂移”都属于关键控制面失败，不应被解释成“无问题”。
- 关键规划/记忆角色的结构失败会污染后续章节，比普通表达角色风险更高。

## 治本方案

1. 建立 `StructuredOutputContract` 清单。
   - 每个角色声明 `role`、`output_format`、`schema_version`、必填/可选字段、类型、别名、未知字段策略、下游消费者、提交策略。
   - prompt 示例、parser、validator、测试样本从同一契约生成或对账。
2. 统一失败分类。
   - `transient_api`
   - `content_rejection`
   - `truncated`
   - `parse_error`
   - `schema_error`
   - `semantic_error`
   - `commit_error`
3. 每个角色声明失败策略。
   - 关键规划/记忆角色：schema/semantic/commit 失败默认阻断或重试，不得空降级。
   - reviewer/fact_checker/consistency：格式失败不得当作“无问题”。
   - arc_planner / volume_planner / beat_planner：结构失败或目标语义失败不得继续下游生成。
   - summarizer 可非阻断，但必须标记 `_schema_error` 或 `trust_level=failed`，后续消费者降权。
4. 关键规划角色返回统一结果对象。
   - `triggered`
   - `committed`
   - `artifact_hash`
   - `failure_type`
   - `failure_message`
   - `semantic_checks`
   - 主流程只在 `committed=true` 或明确“不需要触发”时进入下一阶段；`parse_error/schema_error/semantic_error` 都不得默默落到旧产物 fallback。
5. 落盘前统一预检。
   - Beat/MasterOutline/Arc/Volume 在覆盖正式产物前校验 schema 和必填小节。
   - Archivist 在任何文件写入前完成 JSON schema/type/semantic 预检；提交异常进入 `commit_error`，不得半提交。
6. 增加测试层。
   - quick：每个 contract 的 parse/schema/default/旧版本兼容。
   - scenario：格式失败、字段缺失、类型错、语义冲突分别触发对应策略。
   - scenario：ArcPlanner 返回坏 JSON 且 `needs_arc_planning=true` 时，主流程不得生成 beat/正文。
   - scenario：Reviewer 无 JSON 且无关键词命中、或恢复路径缺 review verdict 时，不得默认放行进入提交尾段。
   - check：prompt 中声明的字段与 contract、parser、consumer 对账。

## 验收标准

- 新增角色字段时，prompt/parser/validator/consumer 任一未同步都会测试失败。
- JSON/Markdown 可解析不再等同可提交。
- 同类失败在不同角色里的重试、阻断、降级策略一致且可审计。
- 关键规划和记忆产物不会因结构错误静默污染后续章节。
