# 关键章布尔字段类型契约修复计划

## 结论

`关键章` 在 prompt 和 `best_of_n_enabled()` 里是布尔语义，但 `normalize_beat()` 会把它转成字符串。若模型输出 `false`，会变成 `"无"`，在 Python 中仍为 truthy。当前 `best_of_n` 默认关闭，暂未造成额外 token 消耗，但这是明确的类型契约错误。

## 当前证据

- `prompts/beat_planner.md` 要求：
  - 当本章是弧线高潮爆发点时，加 `"关键章": true`。
  - 大多数章节不标。
- `scripts/run_pipeline.py:119-133`：
  - `关键章` 被放进字符串化字段列表。
  - `normalized[key] = str(beat.get(key) or "无")`
- `scripts/run_pipeline.py:532-555`：
  - `best_of_n_enabled()` 用 `bool(beat.get("关键章")...)` 判断。
  - 字符串 `"无"` 的布尔值为 True。
- `prompts/beat_planner.md:28-62` 的“字段必须包含”示例没有列出 `关键章`，但 `prompts/beat_planner.md:202-210` 又要求高潮爆发点加 `"关键章": true`；这是“条件必填字段”没有进入 schema 的契约漂移。
- `scripts/pipeline/planning.py:1323-1333` 会在弧线高潮窗口注入“必须标关键章”的硬约束，但 BeatPlanner 必填字段清单没有声明其类型与条件，增加模型漏标/错标概率。
- 第 78、89 章 raw 是布尔 `true`，正式 beat 变成字符串 `"True"`，见 `beats/chapter_78.json:44`、`beats/chapter_89.json:45`；更关键的是两章 `本章张力` 都是“小起伏”，见 `beats/chapter_78.json:24`、`beats/chapter_89.json:26`，并非 prompt 定义的弧线高潮爆发点。

## 影响

- 一旦 `best_of_n.enabled=true`，且章节落在高潮窗口内，`关键章=false` 被归一成 `"无"` 后可能被误判为 True。
- 当前 `in_climax_window()` 护栏降低了误触发概率，但没有修复类型错误。
- “关键章=true”还缺少语义一致性门禁：它应与 arc 高潮窗口、节点类型、张力档位匹配；“小起伏 + 关键章 true”至少应进入人工复核，不能只因为类型可转就当作高潮章。
- 这类问题属于“配置一打开就可能烧 token”的潜伏主流程风险。

## 修复建议

1. 在 `normalize_beat()` 中单独处理 `关键章`：
   - `true/True/"true"/"是"` -> `True`
   - `false/False/"false"/"否"/"无"/缺失` -> 不写字段或写 `False`
2. `best_of_n_enabled()` 不直接 `bool(value)`，改为显式布尔解析。
3. 增加关键章语义门禁：
   - `关键章=true` 必须满足 arc 高潮窗口或明确节点类型。
   - `本章张力` 应至少是 `小高潮` 或存在等价高潮标识；`小起伏/平 + 关键章=true` 进入复核。
4. 更新 BeatPlanner schema。
   - 把 `关键章` 定义为条件字段：默认缺失/false，仅当弧线高潮爆发点时 true。
   - 明确类型为 boolean，不得输出 `"无"`、`"True"` 等字符串。
5. 增加 quick 测试：
   - raw `{"关键章": true}` -> 正式 beat 为 True。
   - raw `{"关键章": false}` -> 正式 beat 为 False 或缺失。
   - raw `{"关键章": "无"}` -> 不触发 best_of_n。
   - raw `{"关键章": true, "本章张力": "小起伏"}` 且不在 arc 高潮窗口时，必须标语义不一致。

## 验收标准

- `关键章` 的正式 beat 类型不再是随意字符串。
- `best_of_n_enabled()` 只在明确 true 时触发。
- `关键章=true` 不再能和“小起伏/非高潮节点”静默共存。
- `scripts/run_tests.py quick` 覆盖 true/false/"无" 三类输入。
