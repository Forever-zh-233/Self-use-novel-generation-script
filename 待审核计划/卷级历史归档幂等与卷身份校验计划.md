# 卷级历史归档幂等与卷身份校验计划

> 状态：待审核。本文件只记录治本方案，未修改代码或运行产物。

## 结论

`volume_summaries.json` 和 `arc_history.json` 以 append-only 方式记录卷级历史，但缺少 volume id、幂等键、提交状态和源内容指纹。消费者又按数组下标推断“第几卷”，一旦重跑、续跑或旧书残留混入，就会把重复/错误历史当成权威上一卷回顾。

## 当前证据

- `scripts/pipeline/planning.py:596-598`：卷摘要只追加 `volume_end_chapter` 和 `digest`。
- `scripts/pipeline/planning.py:652-655`：弧线历史只追加 `archived_at_chapter` 和 `arcs`。
- `scripts/pipeline/state.py:504-510`：读取卷摘要时用 `enumerate(..., 1)` 生成“第 i 卷”，而不是读取真实 volume id。
- `scripts/pipeline/state.py:512` 后继续读取 `runtime/arc_history.json`，同样依赖历史文件作为规划回顾来源。
- 当前 handoff 已观察到 `runtime/volume_summaries.json` 含旧书残留/多个第一卷摘要，`runtime/arc_history.json` 重复出现 `ARC-007`。

## 根因判断

- 卷级历史没有“这条记录属于哪本书、哪一卷、哪次正式提交”的身份模型。
- 追加前没有检查同一 volume/source hash 是否已经归档。
- 消费端用数组顺序当业务身份，导致历史文件一旦污染，错误会被伪装成正常“上一卷/前几卷”。

## 影响

- VolumePlanner 的“上一卷实际走了什么”会混入旧书、重复卷或未来草稿。
- 同一卷多次运行 VolumePlanner 会重复归档 active arcs，后续 arc review 统计失真。
- 清理不完整或换书时，旧书卷史会继续影响新书规划。

## 治本方案

1. 定义卷级历史 schema。
   - 必填：`book_id`、`volume_id`、`volume_start`、`volume_end`、`archived_at_chapter`、`source_plan_hash`、`source_arcs_hash`、`commit_state`、`created_at`。
   - 禁止仅靠数组顺序表达卷身份。
2. 写入前做幂等 upsert。
   - 幂等键：`book_id + volume_id + source_plan_hash + source_arcs_hash`。
   - 同键重复运行不 append，必要时更新同一记录状态。
3. 消费端按身份读取。
   - VolumePlanner 只读取当前 book_id、目标卷之前、commit_state=committed 的历史。
   - 如果发现重复 volume_id 或 source hash 冲突，阻断并提示人工检查。
4. cleanup/换书流程清理或隔离。
   - runtime 卷级历史必须带 book_id；换书时默认不注入其他 book_id。
5. 增加测试。
   - 同一卷重复归档两次，断言只保留一条 committed 记录。
   - 历史文件插入旧 book_id 记录，断言不会进入当前 VolumePlanner 输入。
   - 数组顺序打乱时，卷号仍按 `volume_id` 显示。

## 验收标准

- 卷级历史记录可追溯、可幂等、可按书隔离。
- 消费端不再用数组下标推断卷身份。
- 重跑 VolumePlanner 不会重复污染历史回顾。
