# 卷纲生成时点元数据与运行水位隔离计划

## 结论

卷纲正文里混入了“当前已写到第 X 章、本卷纲覆盖第 Y-Z 章、下一周期生成下一卷”这类生成时元数据。后续 BeatPlanner 注入卷纲时把这段行政说明原样放进 `beat_input`，导致第 99 章仍看到“当前已写到第96章”。这不是要手改卷纲，而是卷纲消费管路没有区分“规划内容”和“生成时说明”。

## 当前证据

- `卷纲/10-卷纲.md:106`：卷纲正文末尾写“当前已写到第96章，本卷纲覆盖第97-100章”。
- `beats/_debug/第097章/beat_input.md:492`、`beats/_debug/第098章/beat_input.md:484`、`beats/_debug/第099章/beat_input.md:483`：同一句被完整注入连续三章 beat input。
- `runtime/state.json:2` 当前 `latest_chapter=99`，`runtime/progress.json:2-9` 已进入第 100 章 writer 阶段，说明“当前已写到第96章”早已不是运行事实。
- `runtime/volume_planner_output.md:106` 也含同一句，说明它来自 VolumePlanner 输出正文而不是独立 metadata sidecar。

## 根因判断

- VolumePlanner prompt 要求直接输出可覆盖 `卷纲/10-卷纲.md` 的 Markdown，缺少 metadata 分区。
- `build_beat_input()` 读取卷纲时直接注入整份文本，没有剥离生成说明。
- 系统没有把“卷纲生成时水位”和“当前运行提交水位”分别建模，导致历史说明可能被模型当成当前事实。

## 影响

- BeatPlanner 可能误以为当前仍在第 96 章之后的卷末规划起点，降低对第 98/99 已发生事实的信任。
- 卷纲覆盖范围、下一周期说明等行政信息会和 active_arcs/current state 竞争权威。
- 后续如果卷纲提前生成下一卷，也可能把旧生成时点当成当前章节进度。

## 治本方案

1. 卷纲输出拆分为正文与元数据。
   - 正文只保留世界、阶段、章节规划、硬承诺。
   - 元数据写入 `runtime/volume_plan_meta.json` 或卷纲 front matter：`generated_at_chapter`、`planned_range`、`source_state_hash`、`created_at`。
2. BeatPlanner 注入卷纲前过滤元数据。
   - “当前已写到”“下一周期生成”“本卷纲覆盖”等语句默认不进入 writer-facing 输入。
   - 如确需保留，只放入低优先 debug 区，并标注“生成时说明，不代表当前进度”。
3. 运行水位只从提交态读取。
   - 当前章节、水位、latest_chapter 只能来自 `runtime/state.json` / chapter manifest / progress，不从卷纲自然语言推断。
4. 增加元数据漂移门禁。
   - 若 beat_input 中出现 `当前已写到第\d+章` 且数字小于 `state.latest_chapter`，直接报 `stale_volume_plan_metadata_injected`。
   - 若卷纲 planned range 与当前章节不匹配，提示重生卷纲或重新仲裁，而不是静默注入。
5. 增加 scenario 测试。
   - 卷纲含“当前已写到第96章”，state latest=99，build_beat_input 不得把该句放入主输入。
   - 调试输入如保留该句，必须带“历史生成说明”标签。

## 验收标准

- beat_input 不再把旧卷纲生成时水位当作当前事实。
- 卷纲可追溯生成时点，但运行决策只信提交水位。
- 修改卷纲生成逻辑后有测试防止行政元数据再次污染规划语义区。
