# 卷纲硬承诺兑现追踪管路修复计划

## 结论

这是管路级兑现追踪问题：卷纲阶段表里有明确硬承诺事件，但系统没有把这些承诺拆成可追踪任务，也没有在阶段尾声检查“已兑现 / 已显式改写 / 未兑现”。因此 arc_planner 可以把直接盘问改成幕后等待，正文也可以按新弧线推进，但卷纲仍保留原承诺，下游同时相信两个版本。

## 当前证据

- `卷纲/10-卷纲.md:69`：71-80 阶段明确要求韩铮私下找沈安，问“你去过窑厂没有”，沈安否认。
- 71-80 的 beat 检索结果没有出现这个直接盘问：
  - `beats/chapter_77.json:24`：只写“韩铮在镇南街角看见沈安从窑厂回来但没拦”。
  - `beats/chapter_80.json:26`：信息差仍是陈寡妇不知道沈安去过窑厂，没有韩铮直接问话。
- 71-80 正文检索没有命中“你去过窑厂没有 / 韩铮私下盘问 / 沈安否认”的兑现片段。
- 当前 active_arcs 改为间接版本：
  - `runtime/active_arcs.json:16`：韩铮知道沈安去过窑厂外围，但没有上报，在等沈安自己送上门。
  - `runtime/active_arcs.json:21`：韩铮在东门哨所等沈安，沈安没来。
  - `runtime/active_arcs.json:167`：第 85 章节点变成“韩铮等了沈安三天没等到”。
- 第 85 章已经按新版本执行：
  - `beats/chapter_85.json:11`：韩铮等沈安三天没等来。
  - `输出/文章/第085章.md:147`：韩铮意识到自己在“等沈安来问话”。

## 根因判断

- 卷纲阶段表没有被转换成结构化 milestone / obligation。
- arc_planner 生成新弧线时，可以重排卷纲硬事件，但没有要求输出“对卷纲承诺的改写说明”。
- story_director 评估近期方向时没有收到“本阶段未兑现硬承诺列表”。
- summaries / ledger 能记录发生了什么，但没有反向对照“应该发生什么”。

## 影响

- 卷纲承诺的正面心理对峙没有发生，沈安没有经历“被韩铮当面逼问并撒谎”的压力。
- 灭门案前两人的关系变成“韩铮单方面知道 / 沈安不知道”，少了卷纲设计的互相试探层。
- 后续如果仍按卷纲理解“已经盘问过”，reviewer 或 story_director 可能误判韩铮线兑现度。

## 治本方案

1. 建立卷纲承诺账本。
   - 从阶段表中抽取硬事件：角色、动作、对话目标、阶段范围、可替代条件。
   - 写入 `runtime/outline_obligations.json` 或类似结构。
2. arc_planner 必须对承诺做三态处理。
   - `execute`：按卷纲兑现。
   - `transform`：改写形式，但保留功能，并说明替代方式。
   - `defer/drop`：延期或取消，必须生成正典变更提案。
3. story_director 在阶段尾声检查承诺账本。
   - 不是只看“近期是否顺”，还要看“阶段硬承诺是否被兑现或被合法改写”。
4. summaries / ledger 反向对账。
   - 每章提交后，用 summary/台账更新 obligation 状态。
   - 到阶段末仍未兑现且无 transform 说明的，进入 planning error。
5. 增加测试。
   - 构造卷纲要求 A 事件，active_arcs 改成 B 事件。
   - 若没有 transform 映射，测试应失败。
   - 若 transform 明确说明 B 保留了 A 的叙事功能，测试应记录为已改写而非未兑现。

## 验收标准

- 卷纲硬承诺有结构化账本和状态。
- active_arcs 改写卷纲事件时必须留下可审计的 transform/defer/drop 记录。
- 阶段结束前未兑现承诺会被 story_director 或 planning gate 点名。
- 测试能捕获“卷纲硬事件被弧线静默替换”的样本。
