# 弧线换批时机与上一批摘要边界修复计划

## 结论

当前弧线换批边界不够严：提示词说“最后节点被 beat_planner 消费后”才规划下一条弧线，但代码会在 `last_chapter <= 当前章 + 1` 时提前触发；同时 `previous_arcs_summary()` 读取的是当前 `active_arcs`，却命名为“上一批弧线收束摘要”。这会让新弧线规划误把仍在进行的弧线当成上一批已收束内容。

## 当前证据

- `prompts/arc_planner.md:10-12`：调用条件是“最后一个节点被 beat_planner 消费后”。
- `scripts/pipeline/planning.py:706-744`：
  - `needs_arc_planning()` 在所有弧线最后节点不大于 `chapter + 1` 时返回 True。
  - 这意味着最后节点前一章就可能触发新弧线。
- `scripts/run_pipeline.py:1733-1743`：
  - 每章先跑 `run_story_director()`，再在生成 beat 前跑 `run_arc_planner()`，最后才 `ensure_beat()`。
  - 所以提前触发时，旧弧最后节点尚未被本章 beat 消费。
- `scripts/pipeline/planning.py:877-889`：
  - `previous_arcs_summary()` 直接 `load_active_arcs()`。
  - 但 `build_arc_input()` 把它作为 `上一批弧线收束摘要(承上启下)` 注入。
- `tests/scenario_test.py:741-752` 现有测试只断言 arc input 能构造且包含活跃弧标题；在当前实现下，这反而会把“把 active arcs 当上一批收束摘要”的语义错误测成绿灯。
- 第 101 章现场样本显示“已过期弧线”仍作为硬约束进入下一卷：
  - `runtime/active_arcs.json:4-11` 当前 active arc 仍是 `ARC-007`“最后一程”，`span` 为 `[94, 100]`，摘要明确服务第一卷 91-100 收束。
  - `runtime/active_arcs.json:266-289` 该弧最后节点落在第 100 章，且 `cross_volume: true` 被保留。
  - `输出/章纲/第101章_beat_input.md:4-11` 在目标章节第 101 章时，仍把 `ARC-007` 作为“【硬约束】当前弧线走向”注入，并提示“本章已到/越过本弧最后节点……准备收束”。
  - `scripts/pipeline/planning.py:1223-1279` 的 `active_arcs_for_beat()` 对 `max(node.chapter) < chapter` 的过期弧线没有隐藏或降级逻辑；只要 `active_arcs.json` 还留着，就会继续输出为硬约束。
- `scripts/pipeline/planning.py:1195-1200`：ArcPlanner JSON 解析失败只打印日志并 `return`，不会向主流程报告“规划失败”。
- `scripts/run_pipeline.py:1736-1743`：主流程调用 `run_arc_planner()` 后不检查成功状态，随即进入 `ensure_beat()`；因此“需要新弧线但规划失败”的章节仍可能继续使用旧 `active_arcs`。
- `scripts/pipeline/planning.py:1406-1408`：只要 `active_arcs_for_beat()` 返回文本，就会被插入“【硬约束】当前弧线走向”，规划失败后的旧弧线仍可能继续压住 BeatPlanner。

## 影响

- 旧弧最后节点的 `narrative_ops`、`resolution_condition` 可能在消费前被新弧覆盖。
- 新弧规划师看到的“上一批弧线收束摘要”可能其实是当前活跃弧线，边界语义错误。
- 当触发原因是“缺主线补线”时，当前副线尤其不应被包装成已收束摘要。
- 跨卷章可能被上一卷已完成主线弧继续压住，导致新卷 opening beat 既要“第二卷开篇”，又被上一卷收束弧线要求“准备收束”。

## 修复建议

1. 区分触发类型：
   - `empty`：无活跃弧线。
   - `finished`：最后节点已小于当前章，且上一章 beat 已存在。
   - `nearly_finished`：只作为预警，不覆盖当前弧。
   - `missing_mainline`：补线模式，保留当前副线，输入名称也应写“当前仍在进行的弧线”。
   - `expired`：最后节点和 span end 均小于当前章，且没有未来 `chapter_drift` / future nodes；必须归档或隐藏，不能作为 beat 硬约束。
2. 将 `previous_arcs_summary()` 改为读取真实归档：
   - 新增或复用 `runtime/arc_history.json`。
   - `save_active_arcs()` 覆盖前，把已完成弧线归档。
3. `active_arcs_for_beat()` 增加使用前过滤。
   - 当前章大于 `span.end` 且大于最后节点章时，不输出到“硬约束”区。
   - `cross_volume=true` 只有在弧线存在未来节点或明确的 `continuation_condition` 时才可保留；否则视为错误标记。
   - 对过期但未归档弧线写审计告警，触发 ArcPlanner 重新规划，而不是让 BeatPlanner 自行消化。
   - 如果本章触发了 ArcPlanner 但规划失败，旧弧线不得作为 fallback 硬约束继续注入；必须停在规划阶段或进入人工复核。
4. `build_arc_input()` 分开注入：
   - “已收束弧线摘要”。
   - “当前仍在进行弧线”。
   - “即将收束但尚未消费的尾节点”。
5. 增加 scenario 测试：
   - 当前章是最后节点前一章时，不覆盖 active arcs。
   - 最后节点章 beat 已生成后，才允许归档并换批。
   - 补主线模式下，副线不出现在“上一批收束摘要”里。
   - 构造“当前章仍在旧弧最后节点前/节点章”的 active arc，断言 `needs_arc_planning()` 不覆盖 active arcs。
   - 构造已归档 arc，断言 `previous_arcs_summary()` 只读归档，不读 active arcs。
   - 构造 `span=[94,100]`、最后节点第 100 章、当前章 101、`cross_volume=true` 但无未来节点，断言 `active_arcs_for_beat(101)` 不输出该弧为硬约束，并触发重新规划/归档告警。
   - 测试名称和断言必须区分“当前活跃弧线摘要”和“上一批已收束弧线摘要”，不能只断言标题出现。

## 验收标准

- 最后节点不会在 beat 生成前丢失。
- “上一批弧线收束摘要”只包含已归档弧线。
- 补线模式保留活跃副线，不误标为已收束。
- 已越过最后节点和 span end 的弧线不会进入下一章 beat 的“硬约束”区。
