# 弧线逐章格子事实态去重与重定位计划

> 状态：待审核。本文件只记录治本方案，未修改代码或运行产物。

## 结论

BeatPlanner 第105章输入中，`active_arcs_for_beat()` 注入的“本章走向格子”仍要求写“捞到绣鞋、系统弹冷字”；但第104章 beat 和正文已经兑现了这个事件。这说明弧线 `chapter_drift` / 节点计划与已完成事实之间没有去重和重定位门禁，未来章节可能重复写已发生事件。

## 当前证据

- `runtime/active_arcs.json:80-85`：`chapter_drift` 中第104章是“遇到捞女老汉”，第105章是“捞到一只绣鞋、系统面板弹冷字”。
- `runtime/active_arcs.json:128-136`：第107章节点仍再次安排“系统拒认怨愿、拿着绣鞋站在泽边”。
- `beats/chapter_104.json:24`：第104章转折已经是“小满捡出绣花鞋，系统面板弹‘怨愿未解，了结方式不符’”。
- `输出/文章/第104章.md:171-175`、`输出/文章/第104章.md:213`：正文已经出现系统面板与“怨愿未解，了结方式不符”。
- `输出/章纲/第105章_beat_input.md:13`：第105章输入仍注入“本章走向格子：捞到一只绣鞋——系统面板弹冷冰冰一行字”。
- `scripts/pipeline/planning.py:1267-1276`：当前逻辑只按 `chapter_drift.ch == chapter` 取格子，没有检查该格子的核心事件是否已在前文兑现。

## 根因判断

- `chapter_drift` 是计划态，正文/beat 是事实态；当前系统只按章节号取计划，没有在进入下一章前校验计划是否被提前兑现。
- ArcPlanner 允许节点/逐章格子存在弹性，但下游没有“提前兑现后重定位”的机制。
- 已完成 beat/final summary 没有反向更新 active arcs 的节点状态，导致过期格子继续作为硬约束注入。

## 影响

- 下一章会重复上一章事件，尤其是“关键物件出现/系统提示/人物做出选择”等强事件。
- 弧线节点可能连续多次要求同一个爆点，造成读者感到原地打转。
- Writer/Reviewer 即使发现重复，也只能在正文层补救，规划层已经把重复当硬约束塞进来了。

## 治本方案

1. 为弧线格子建立 obligation / event key。
   - ArcPlanner 输出 `chapter_drift` 和节点时，为核心事件标 `event_key`、`expected_artifacts`、`can_shift`。
   - 例如“捞到绣鞋”“系统拒认怨愿”是可追踪事件，不只是自然语言 gist。
2. Beat 保存后提交事实态回写。
   - 从 beat/final summary 中记录已兑现 `event_key`。
   - 写入 `runtime/arc_progress.json`，标记 fulfilled_chapter。
3. BeatPlanner 输入前做重定位。
   - 如果当前章格子的 event_key 已在 earlier chapter fulfilled，则不再注入原格子。
   - 选择下一未兑现格子，或提示“上个格子已提前兑现，本章应处理后果/推进下一步”。
4. 节点与格子去重。
   - 同一 event_key 不应同时在 105 drift 和 107 node 中重复作为待执行硬事件。
   - 允许一个节点拆成“事件发生”和“后果发酵”，但必须用不同 event_key 标明。
5. 增加测试。
   - 构造第104章 beat 已兑现 `怨愿拒认`，第105章 active_arcs 仍要求该事件，断言输入改为后果处理或下一格。
   - 同一 event_key 出现在多个未来节点，断言 Arc/Beat preflight 告警。

## 验收标准

- 计划态弧线格子不会重复要求已在正文中发生的强事件。
- 提前兑现后，系统能把下一章重定位到“后果/反应/下一步”，而不是重写同一幕。
- BeatPlanner 的硬约束输入带有 fulfilled/next 状态说明。
