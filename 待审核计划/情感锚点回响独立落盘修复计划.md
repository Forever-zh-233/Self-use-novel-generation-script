# 情感锚点回响独立落盘修复计划

## 结论

`emotional_anchor_echoed` 在 prompt 中是独立字段，但实现把回响处理放在“本章新增情感锚点”的分支内部。也就是说，只回响旧锚点、不新增新锚点的章节，不会更新 `echo_count`、`last_echo_chapter` 或 `echo_status`。

## 当前证据

- `prompts/archivist.md:435` 定义 `emotional_anchor_event`：本章新增情感锚点。
- `prompts/archivist.md:436` 定义 `emotional_anchor_echoed`：本章回响旧锚点。
- `scripts/pipeline/archivist.py:755-780`：只有 `ea_events` 存在时才进入处理块。
- `scripts/pipeline/archivist.py:780-787`：`emotional_anchor_echoed` 的更新逻辑缩进在 `if ea_events:` 内。
- 当前 `runtime/ledger.json` 已有 22 个 `emotional_anchors`，大多仍为 `echo_status=活跃`、`echo_count=0`；由于中间记录员产物被清理，不能仅凭数量证明漏更，但实现路径已能证明“只回响旧锚点”不会落盘。

## 影响

- 旧情感锚点明明已在正文回响，账本仍显示未回响。
- 规划层会继续把已回响锚点当活跃素材推给 beat，造成重复消费。
- “意难平/牵挂/承诺”这类情感资产无法形成准确的回响次数和最近回响章。

## 修复建议

1. 将 `emotional_anchor_echoed` 处理移出 `if ea_events:` 块。
2. 无论本章是否新增锚点，只要 `emotional_anchor_echoed` 存在，就读取 `ledger.emotional_anchors` 并更新对应 ID。
3. 对不存在的 EA ID 做 warning，不要静默吞掉：
   - 可写入记录员报告校验 warning。
   - 或追加到 debug/门禁诊断。
4. 增加 scenario 测试：
   - ledger 预置 `EA-001`。
   - 记录员报告只含 `emotional_anchor_echoed:["EA-001"]`，不含 `emotional_anchor_event`。
   - 调用 `merge_ledger_update()` 后断言 `echo_count=1`、`last_echo_chapter=当前章`。

## 验收标准

- 回响旧锚点不依赖本章新增锚点。
- 不存在的 EA ID 有可追踪诊断。
- `scripts/run_tests.py scenario` 覆盖新增和只回响两种路径。
