# 断点恢复台账幂等提交修复计划

## 结论

断点恢复的提交标记是 `runtime/state.json.latest_chapter`，但 `apply_archivist_update()` 在推进该标记前已经写入 state/ledger、追加 `07-动态状态台账.md`、追加 `08-期待账本.md`、写 `台账版本/` 快照。如果进程刚好在这些写入之后、推进 `latest_chapter` 之前崩溃，下一次启动会把同一章记录员报告再应用一次，造成重复追加。

## 当前证据

- `scripts/run_pipeline.py:196-250`：
  - `recover_state_from_completed_articles()` 只看“正文已落盘但 `latest_chapter` 落后”的章节。
  - 对这些章节直接调用或复用 Archivist 报告，然后 `apply_archivist_update()`。
- `scripts/run_pipeline.py:221-250` 的恢复报告复用分支疑似把完整性校验语义写反：
  - `scripts/pipeline/archivist.py:975-995` 明确 `validate_archivist_report()` 返回问题列表，空列表代表通过。
  - `scripts/run_pipeline.py:224-225` 在 `archive_report and not validate_archivist_report(...)` 时打印“使用已存在且完整的记录员报告恢复台账”。
  - 但重调 Archivist 后的分支 `scripts/run_pipeline.py:244-247` 也使用 `if not validate_archivist_report(...): break`，说明空列表=完整。
  - 因此这段代码虽然当前分支条件能命中完整报告，但整体可读语义极易误维护；若未来有人按“validate 返回 True/False”的直觉改动，会把不完整报告复用进恢复写账。恢复路径需要显式封装 `problems = validate...`，避免双重否定。
- `scripts/pipeline/archivist.py:998-1037`：
  - `merge_state_update()`、`merge_ledger_update()`、`apply_character_arc_note()` 在 `latest_chapter` 推进前执行。
  - `07-动态状态台账.md` / `08-期待账本.md` 追加也在推进前执行。
  - `台账版本/chapter_N_*.md` 写入也在推进前执行。
  - 最后才写 `state["latest_chapter"] = chapter`。
- `README-使用说明.md` 与配置说明都把恢复定位为崩溃后自动补台账，因此这条路径属于主流程保障，不是边缘工具。

## 影响

- 崩溃窗口虽然窄，但一旦发生，会重复追加“第N章自动更新”。
- 重复的 recent ledger 会进入 writer/planner 上下文，造成同一事实被加权过高或反复提示。
- 如果 ledger merge 中有 append-only 字段，也可能重复增长。
- 恢复路径校验写法含双重否定，维护者很难一眼确认“完整报告复用、不完整报告重调”的行为，断点恢复这种高风险路径不应靠 `not validate(...)` 表达提交语义。

## 修复建议

1. 为每章记忆提交增加幂等标记，例如：
   - `runtime/commits/chapter_N.json`
   - 或 `state.applied_archivist_chapters`。
2. `apply_archivist_update()` 开始前检查该章是否已成功应用：
   - 若已应用，直接跳过，不重复 append。
3. 写入顺序改为两阶段：
   - `applying` 临时标记记录报告 hash。
   - 所有写入完成后写 `applied` 标记，再推进 `latest_chapter`。
4. 对根目录 Markdown 追加加幂等保护：
   - 追加前查找 `### 第{chapter}章自动更新`，存在则替换或跳过。
5. 增加 scenario 测试：
   - 模拟已写 07/08 但 `latest_chapter` 未推进。
   - 再跑恢复，断言不会重复追加同一章。
   - 报告 hash 不同但章号相同，应停下要求人工确认，避免覆盖冲突。
   - 构造完整 Archivist 报告，断言恢复路径复用报告且不重调 Archivist。
   - 构造缺 `STRUCTURED_UPDATE` 的不完整报告，断言恢复路径必须重调 Archivist；若重调仍失败，不得调用 `apply_archivist_update()`。

## 验收标准

- 同一章 Archivist 更新重复调用不会造成重复台账追加。
- 崩溃恢复可识别“已应用但提交标记未推进”的半提交状态。
- `scripts/run_tests.py scenario` 覆盖恢复幂等。
