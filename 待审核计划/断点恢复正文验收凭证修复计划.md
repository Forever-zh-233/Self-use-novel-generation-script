# 断点恢复正文验收凭证修复计划

## 结论

断点恢复把“正文文件长度足够”当成“生成阶段已经可信完成”的凭证。只要 `输出/文章/第NNN章.md` 已存在且汉字数达到阈值，主流程就会跳过 writer、gate、reviewer、editor、fact_checker，直接进入 summarizer 和 archivist。这个判断缺少“该正文确实通过最终门禁和事实核查”的提交凭证。

这不是要修某个已落盘章节，而是恢复路径的验收条件过弱。

## 当前证据

- `scripts/run_pipeline.py:931-939`：`have_final` 只看 `resume_partial_chapter` 和正文字数是否达到 `min_recover_article_chars`，成立后直接“跳过生成阶段，直接补台账”。
- `scripts/run_pipeline.py:926-940`：恢复路径虽然读取了当前 `beat_path`，但 `have_final` 不校验正文是否由当前 beat、writer prompt、ledger/state/chunks 生成；命中后会跳过 writer / reviewer / editor / fact_checker / final_gate。
- `scripts/run_pipeline.py:599-612`：`resume_verdict()` 在 `review_verdict.json` 与 `review.md` 都缺失时返回 `resume_default`，默认 `needs_revision=False`。
- `scripts/pipeline/gates.py:517-558`：Reviewer JSON 判定块解析失败时会回退关键词；关键词没命中即 `needs_revision=False`。恢复路径若只有坏 review 或无 review，也可能默认放行。
- `scripts/run_pipeline.py:949-955`：跳过生成阶段后仍会生成写手摘要。
- `scripts/run_pipeline.py:958-998`：随后仍会进入 Archivist，可能把未验证正文写入记忆。
- `config/配置说明.md:88`：默认 `artifact_retention=clean` 会清理 prompt/input/raw/draft/review/gate/editor/archive 过程副产物，只保留正式文章和 runtime 等文件。
- `config/配置说明.md:90`：恢复逻辑被明确描述为“正文已落盘但 state 尚未记录时补台账”，属于主流程保障路径。
- `scripts/run_pipeline.py:904-915`：标题规范化只在正常生成路径执行；恢复路径直接读取已落盘正文，不重新执行标题规范化。
- `输出/文章/第060章.md:1`：历史样本第一行是正文“日头偏西……”，不是标准标题；对应 `beats/chapter_60.json:3-4` 有标题和 POV，但 summary 已基于该正文生成。

## 根因判断

- 主流程没有给最终正文写一个不可清理的验收凭证，例如正文 hash、final_gate 结果、fact_checker 状态、review verdict、生成版本。
- `artifact_retention=clean` 会让恢复路径可用的证据更少，导致 `resume_verdict()` 可以默认放行。
- 恢复路径只保护“不要重复花 token”，没有保护“不要把未验收正文写入记忆”。
- 恢复路径没有做章节元数据再验收：标题、章号、POV、beat hash、final hash 都没有统一对账。

## 影响

- 如果进程在异常窗口、人工拷入、半自动调试或文件误留后出现一篇足够长的正文，恢复路径会把它当成可信终稿。
- gate/reviewer/fact_checker 的结果可能与正式正文版本不匹配，甚至完全不存在。
- 正文可能对应旧 beat / 旧提示词 / 旧状态，却被接到当前章节输入上继续提交。
- summarizer/archivist 会把未验证正文沉淀到后续上下文，污染后续生成。

## 治本方案

1. 增加正文验收凭证文件。
   - 例如 `runtime/final_commits/chapter_N.json`。
   - 内容包含正文 hash、标题、字数、final_gate passed、fact_checker passed/waived、review verdict、生成时间和 pipeline 版本。
2. `generate_chapter_final()` 在最终正文写盘后同步写凭证。
3. `cleanup_chapter_artifacts()` 不删除该凭证。
4. `run_one_chapter()` 的 `have_final` 判断改为：
   - 正文存在且长度足够；
   - 验收凭证存在；
   - 凭证 hash 与当前正文一致；
   - gate/fact 状态满足配置。
   - 标题/章号/POV 等章节元数据与 manifest/beat 一致。
5. 无凭证或 hash 不一致时不要直接补台账。
   - 保守方案：停机要求人工确认。
   - 或显式配置 `recover_unverified_article=true` 才允许只补台账，并写豁免记录。
6. 增加 scenario 测试。
   - 只有长正文、无凭证：不得进入 summarizer/archivist。
   - 长正文 + 凭证 hash 不匹配：不得进入 summarizer/archivist。
   - 长正文但缺标准标题或元数据不一致：不得进入 summarizer/archivist，除非人工显式豁免。
   - 修改当前 beat 后保留旧长正文：恢复路径必须因 `beat_hash` 不匹配而停止或要求人工豁免。
   - review 缺失、review JSON 坏且无关键词命中：恢复路径不得生成 `resume_default` 放行提交。
   - 长正文 + 有效凭证：可跳过生成并补台账。

## 验收标准

- 恢复路径不再只凭正文长度放行。
- 未通过或无法证明通过 final gate/fact checker 的正文不会进入记忆链路。
- 默认 clean 模式下仍保留恢复所需的最小验收凭证。
