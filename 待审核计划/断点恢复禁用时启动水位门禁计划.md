# 断点恢复禁用时启动水位门禁计划

> 状态：待审核。本文件只记录治本方案，未修改代码或运行产物。

## 结论

`auto_recover_on_start=false` 时，主流程会跳过从已完成正文补台账的恢复逻辑；但 `start_chapter=auto` 仍按 `输出/文章/` 最高章号继续往后跑，而不是按 `runtime/state.json.latest_chapter` 的提交水位启动。这会把“恢复被关闭”误解成“可以绕过恢复”，导致台账/状态未提交时继续生成后续章节。

## 当前证据

- `scripts/run_pipeline.py:196-198`：`recover_state_from_completed_articles()` 在 `dry_run` 或 `auto_recover_on_start=false` 时直接返回。
- `scripts/run_pipeline.py:274-278`：`determine_start_chapter()` 在 `start_chapter=auto` 分支解析自动起点。
- `scripts/run_pipeline.py:270`：自动起点返回的是已发现最高章号 `highest + 1`。
- `config/run.json:3,11`：默认配置同时启用 `start_chapter=auto` 和 `auto_recover_on_start=true`，说明二者目前被当作可独立开关。
- `config/配置说明.md:90-92`：文档把 `auto_recover_on_start` 解释为“启动时先补台账再继续”，但未定义关闭恢复时如何处理已落盘正文与 state 水位不一致。

## 根因判断

- 系统缺少“启动水位门禁”：正文最高章、beat 最高章、summary 最高章、state.latest_chapter、progress 当前阶段之间没有在启动前统一对账。
- `auto_recover_on_start=false` 目前只关闭“修复动作”，没有切换为“只读校验并阻断”。因此用户一旦关闭恢复，流程会继续相信文件系统最高章。
- “是否恢复”和“从哪一章开始”被拆成两个无约束决策，导致启动策略出现空洞。

## 影响

- 如果第 N 章正文已落盘但 Archivist/state 未提交，关闭自动恢复后会从第 N+1 章继续，后续规划输入缺失第 N 章事实。
- 台账、摘要、期待账本、弧线状态可能落后一章或多章，后续正文会在错误记忆上继续滚动。
- 用户以为关闭恢复只是“不自动修”，实际效果可能是“带着旧账继续写”。

## 治本方案

1. 增加启动前 `waterline_preflight`。
   - 读取 `输出/文章`、`beats`、`runtime/summaries`、`runtime/state.json.latest_chapter`、`runtime/progress.json`。
   - 计算 formal waterline 与 committed waterline。
   - 若正文最高章大于 state.latest_chapter，进入“未提交正文”状态。
2. 重新定义 `auto_recover_on_start=false`。
   - 关闭时只是不自动补账，不代表允许越过未提交章。
   - 发现正文水位领先 state 时应阻断并提示：开启恢复、指定 `--chapter`、或人工清理/提交。
3. `determine_start_chapter()` 不再单独相信正文最高章。
   - 自动起点必须来自通过 preflight 验证的 committed waterline。
   - 如果存在未提交正文，必须先处理恢复或显式人工选择。
4. 增加主流程场景测试。
   - state=104、正文到105、`auto_recover_on_start=false`、`start_chapter=auto` 时，断言不会直接启动 106。
   - state=104、正文到104 时，断言启动 105。
   - state=104、正文到105、恢复开启时，断言先进入恢复链路。

## 验收标准

- 关闭自动恢复不再允许绕过未提交正文继续生成。
- 启动日志明确打印正文水位、state 水位、恢复策略和最终起点。
- 测试能覆盖 `auto_recover_on_start` 开/关两种语义。
