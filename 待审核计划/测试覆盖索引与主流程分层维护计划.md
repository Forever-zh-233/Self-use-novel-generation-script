# 测试覆盖索引与主流程分层维护计划

## 结论

当前测试实际已经覆盖了不少 scenario，但测试入口和维护文档仍把 `scenario` 描述成 writer/reviewer/archivist/context/state 的“小链路”。这会让维护者低估 `main()`、CLI、规划角色触发、章节生命周期、提交尾段这些主流程调度风险。测试绿不等于主流程已覆盖，覆盖索引也没有告诉人哪些阶段仍是空白。

## 当前证据

- `scripts/run_tests.py:27-31`：测试入口只有 `check/quick/scenario/all`，`scenario` 全部映射到 `tests.scenario_test`，没有显式的调度层分组。
- `TESTING.md:27`：`scenario` 描述为“写手上下文、衔接检查、记录员结构化合并、评审输入”，未提 `main()`、CLI 路由、VolumePlanner、StoryDirector、ArcPlanner、BeatPlanner、章节循环、提交事务。
- `测试文件维护指南.md:86`：run pipeline 长流程的维护建议是“先拆成 quick/scenario 小链路”，但没有说明哪些主流程 glue code 必须用临时 workspace + mock LLM 覆盖。
- `TESTING.md:99-103`：当前覆盖索引只列 writer 上下文、writer 账本视图、相邻章节衔接、archivist 合并、reviewer 输入不读分数表。
- `测试文件维护指南.md:265-272`：覆盖索引同样停在旧范围，未反映 `tests/scenario_test.py` 中已有的 planning、analyst、cleanup、token 护栏等大量新增场景，也未标出 `main()` 仍未覆盖。
- `tests/scenario_test.py` 已有很多分散场景（如弧线、summarizer、analyst、cleanup），但没有一个维护索引说明它们属于哪一层、哪些主流程阶段仍缺。
- 主流程高风险缺口已能列出明确样本：
  - `scripts/run_pipeline.py:791-794`：final_gate 失败默认只在 `fail_on_final_gate=true` 时阻断。
  - `scripts/run_pipeline.py:887-891`：fact_checker 最终仍有残留时写 `residual_issues.md` 并“接受现状”。
  - `scripts/run_pipeline.py:284-286`：已有 beat 文件直接返回，绕过 `beat_direction_check()`。
  - `scripts/run_pipeline.py:936-939`：断点续跑只凭正文长度跳过生成阶段。
  - `scripts/pipeline/gates.py:548-553`：reviewer `scores` 被解析器丢弃。

## 根因判断

- 测试套件命名只有技术层级，没有业务调度层级。
- 覆盖索引靠人工维护，已经落后于真实测试文件。
- 维护指南强调“不要真实连跑几十章”是对的，但没有补上“用 mock LLM 跑主流程入口/准入口”的替代要求。

## 影响

- `previous_arcs_summary` 这类“函数存在但主流程接错”的问题会继续漏。
- 新增角色 prompt、规划触发分支、恢复/停机逻辑时，维护者可能只补局部函数测试。
- `run_tests.py all` 通过后，用户和维护者无法知道主流程调度层仍有哪些空白。

## 治本方案

1. 给 scenario 建立覆盖分层索引。
   - `context`：writer/reviewer/editor/fact_checker 输入。
   - `planning`：story_director/arc/beat/volume/master outline。
   - `dispatch`：`main()`、CLI、章节循环、角色调用顺序、失败停机。
   - `commit`：final/summary/archive/lifecycle/hash/waiver。
   - `analyst`：map/reduce/structure/cleanup。
   - `cleanup`：产物保留、debug 留档、运行态边界。
2. 更新 `TESTING.md` 和 `测试文件维护指南.md`。
   - 明确 `scenario` 不只是小链路，也包含主流程调度场景。
   - 凡改 `scripts/run_pipeline.py:main()`、`run_one_chapter()`、规划触发、恢复逻辑、章节提交尾段，必须补 `dispatch` 或 `commit` 场景。
3. 让覆盖索引可维护。
   - 可以先用文档手写表，但每条测试 section 要标分类前缀，如 `scenario(dispatch): ...`。
   - 后续可用脚本扫描 `h.section()` 生成覆盖清单，发现未分类 section 时 `check` 报警。
4. 标明未覆盖项。
   - 当前至少应显式列出：`main()` CLI/章节循环、规划触发顺序、Archivist 失败后 summary 可见性、final/review/beat hash 一致性等。
   - 还应显式列出：final_gate hard failure 阻断、fact residual 阻断、已有 beat 使用前校验、断点续跑正文凭证、review scores 落盘。

## 验收标准

- 维护者能从文档直接看出哪些主流程阶段已有 scenario，哪些仍缺。
- 新增主流程分支时，不会只补纯函数测试就误以为覆盖完成。
- `previous_arcs_summary` 这类“写了但接错/语义接反”的问题有调度层测试防线。
