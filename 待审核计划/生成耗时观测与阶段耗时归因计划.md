# 生成耗时观测与阶段耗时归因计划

## 结论

当前已经能从代码推断“慢主要来自多角色 LLM 串行与隐藏重试”，但运行时没有稳定的阶段耗时归因。用户想知道慢在哪里时，只能读日志片段或猜 provider、FactChecker、Archivist、压缩器哪个拖慢。需要给每次角色调用和每章生成建立轻量耗时观测。

## 当前证据

- `config/run.json`：`sleep_seconds_between_calls=1`，固定等待本身很小。
- `scripts/run_pipeline.py:663-671`、`714-722`、`758-766`、`811-886`、`950-974`：writer、reviewer、editor、FactChecker 修复链、summarizer、Archivist 串行调用。
- `scripts/pipeline/api.py:276-277`：`call_role()` 内部还有 `reject_retries=3`、`truncate_escalations=2`。
- `scripts/pipeline/api.py:121-125`：单请求 hard timeout 为 `timeout * 1.5 + 30`。
- `scripts/pipeline/context.py:819-827`：超预算压缩会额外调用 compressor。

## 根因判断

- 当前进度只记录当前阶段和总 elapsed，不记录每次 LLM 调用的输入规模、输出预算、实际耗时、重试/截断/压缩原因。
- 章节完成日志只有总耗时和评分，缺少 breakdown。
- 没有机器可读的 slow trace，难以判断优化优先级。

## 影响

- 用户会把慢误判为固定 sleep 或某个单一角色。
- 想调模型、timeout、fact_checker、archivist 或压缩阈值时没有数据依据。
- 隐藏重试、截断扩容、压缩器调用可能只在终端一闪而过，事后不可追。

## 治本方案

1. 为每次 LLM 调用记录 trace。
   - role
   - chapter
   - attempt
   - input_tokens
   - max_output_tokens
   - timeout
   - provider/model
   - elapsed_seconds
   - finish_status
   - retry_reason / truncate_escalated / compressed
2. 每章生成一个耗时 breakdown。
   - planning
   - beat
   - writer
   - gate
   - reviewer
   - editor
   - fact_checker
   - summarizer
   - archivist
   - cleanup
3. 章节完成日志追加简短归因。
   - 例如“本章 159s：writer 48s / reviewer 31s / archivist 42s / fixed sleep 3s”。
4. 与 audit manifest 联动。
   - clean 模式删除大产物后仍保留耗时 trace。
5. 增加测试。
   - fake `call_role()` 耗时，断言 trace 文件写出。
   - fake truncate/retry，断言 reason 被记录。
   - `sleep_seconds_between_calls=0` 时 fixed_sleep 归因为 0。

## 验收标准

- 每章能机器读取耗时 breakdown。
- 能区分 provider 慢、角色重试、FactChecker 修复链、Archivist 重试、压缩器触发和固定 sleep。
- 优化速度时有数据依据，不靠猜。
