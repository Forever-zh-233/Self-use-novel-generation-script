# 眼盲系统代价正典冲突仲裁计划

## 结论

眼盲/系统代价属于故事核心设定，但当前故事核、卷纲、全书骨架给出了互相冲突的口径，并且同一份 BeatPlanner 输入会同时注入这些口径。下游角色无法判断“眼盲是否会因了愿/修炼/负愿债变糟、是否永久不可逆、最终是否会解除”哪一个是正典。

## 当前证据

- `09-故事核.md:46`：眼盲是“入门费”，不是每次使用系统才扣。
- `09-故事核.md:52`：了愿不会让眼睛更瞎。
- `09-故事核.md:84-86`：眼盲是入门费，不是使用费；眼盲本身永久不可逆。
- `卷纲/10-卷纲.md:38,41,59,69,75`：第 99 章要求负愿债状态下强行突破可能加速眼盲封印松动，并明确“变强会瞎得更厉害”。
- `全书骨架.md:212,245`：后期写眼盲封印随系统消散而解除，眼睛能看了。
- `beats/_debug/第097章/beat_input.md:114-155` 注入故事核旧口径；同一文件 `beats/_debug/第097章/beat_input.md:424-461` 又注入卷纲新口径。

## 根因判断

- 故事核、全书骨架、卷纲都能表达核心设定，但没有 `canon_fact_id`、版本和覆盖关系。
- 卷纲可以新增“修炼/愿债影响眼盲”的机制变化，但没有显式声明这是对故事核的改写、细化还是例外。
- 输入构造层把相反设定并列注入，缺少正典仲裁。

## 影响

- Writer 可能写成“了愿会让眼睛更瞎”，违背故事核。
- 也可能完全忽略第 99 章卷纲要求的系统警告，导致卷末代价线失效。
- 长线 LF-001 的推进会在“永久不可逆”和“最终解除”之间摇摆，后期回收成本变高。

## 治本方案

1. 建立核心设定事实表。
   - `canon_fact_id`
   - `domain=system_cost/blindness/realm`
   - `current_rule`
   - `allowed_exceptions`
   - `planned_evolution`
   - `source`
   - `status=active|superseded|proposal`
2. 区分规则层级。
   - “普通了愿不会让眼睛更瞎”可作为 active rule。
   - “负愿债 + 强行突破触发封印松动警告”若成立，应登记为 exception，而不是覆盖成所有了愿都会变瞎。
   - “最终封印解除”应登记为 planned evolution，并说明与“永久不可逆”的关系：是旧认知被推翻、封印解除但视觉代价仍在，还是故事核需要更新。
3. BeatPlanner 输入只注入仲裁后的规则。
   - 相互冲突的原文可放入 debug，但 writer-facing 输入必须给 `chosen_rule` 和 `discarded/superseded_rule`。
4. 增加正典冲突门禁。
   - 同一输入同时出现“不会更瞎”和“变强会瞎得更厉害”时，必须要求 exception 解释。
   - 同一输入同时出现“永久不可逆”和“最终解除”时，必须要求 planned evolution 解释。
5. 增加 scenario 测试。
   - 故事核写普通了愿不加重眼盲，卷纲写强行突破触发警告，系统应输出例外关系而非冲突文本并列。
   - 没有例外说明时，beat_input 生成失败。

## 验收标准

- 核心设定不再以多份 Markdown 自然语言互相覆盖。
- 眼盲/系统代价规则进入 writer 前已仲裁。
- 长线 LF-001 能区分“普通规则、例外触发、后期演化”。
