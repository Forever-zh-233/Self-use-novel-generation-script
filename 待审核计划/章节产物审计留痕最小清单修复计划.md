# 章节产物审计留痕最小清单修复计划

## 结论

默认 `artifact_retention=clean` 会清理 writer final、archive_input、review/gate/editor 等大部分过程产物。节省空间是合理的，但当前没有保留一份最小审计清单来证明“哪些输入进入了哪个角色、哪个正文版本通过了哪些门禁、Archivist 基于哪个正文更新了记忆”。长期只剩正文、summary、台账，很多跨产物问题无法事后互证。

## 当前证据

- `config/配置说明.md:88`：默认 clean 会删除 prompt/input/raw/draft/review/gate/editor/archive 过程副产物，只保留终稿和 runtime 等。
- `scripts/pipeline/core.py:364-392`：`cleanup_chapter_artifacts()` 在 clean/reports 模式都会删除不在保留名单中的阶段产物。
- `scripts/pipeline/core.py:371-377`：reports 模式保留名单只有 review、archive_update、final_gate、final_style_gate、final_continuity 等，不包含 `archive_input.md`、`writer/final.md`、`editor_input.md`。
- `scripts/run_pipeline.py:969-973`、`992-996`：主流程原本会把 archive_input 写到角色产物路径，但成功清理后可能被删。
- 审计时第 89 章可从 beat 看到标题/POV/张力等字段，但 archive_input 已不可用，无法证明记录员当时是否拿到这些材料。
- 第 94 章样本：`runtime/state.json` 已推进到 94，`beats/chapter_94.json` 和 `输出/文章/第094章.md` 仍可对上标题/POV，但 clean 后 writer/gate/reviewer/editor/fact_checker/archivist 的过程输入大多不可追，只剩 beat debug、正文、summary、07/08、runtime 台账。
- `scripts/pipeline/core.py:371-377`：reports 模式也只保留少数报告，不保留 `archive_input.md`、`writer/final.md`、`editor_input.md`，排查“记录员到底吃了什么”仍会断证。
- `scripts/pipeline/context.py:40`：`make_section()` 只记录 title/body/priority/compressible/tokens，没有 `source_path/source_type/trigger_reason/exclusion_reason`。
- `scripts/pipeline/context.py:767`、`scripts/pipeline/context.py:779`：上下文预算选择/压缩只返回最终文本，不保留哪些 section 被纳入、压缩、排除及原因。
- `scripts/pipeline/context.py:856`：`select_chunks()` 的必选、场景、关键词、角色命中逻辑只存在于代码执行时，没有把命中关键词、cap 截断或被排除 chunk 写入审计清单。
- `scripts/run_pipeline.py:412`、`scripts/run_pipeline.py:461`：FactChecker 手工拼输入，不走统一 section 元数据与预算 manifest。
- `scripts/run_pipeline.py:811-890`：FactChecker 会产生 `fact_check.md`、`verify_*.md`、`residual_issues.md` 等多版本判断；若 clean 后只剩最终正文，就无法证明最终落盘正文是哪次 fact 修复后的版本。
- `scripts/run_pipeline.py:779-787`：final_gate 会落盘 `final_gate.json`，但当前审计清单没有强制记录 final_gate 校验文本 hash 与最终落盘正文 hash 是否一致。
- `scripts/pipeline/summarizer.py:16,32`：summary 写在 `runtime/summaries/chapter_NNN.json`，不走统一角色产物路径；审计清单需要单独记录 summary 输入/输出 hash 和提交态。

## 根因判断

- clean 策略是按“后续脚本是否读取”决定删除，而不是按“审计最小证据”决定保留。
- 没有轻量 manifest/hash 取代大文件留存。
- reports/debug 模式适合排错，但默认长跑需要的是低成本、长期可保留的证据链。
- 角色输入被当成一整段文本保存/删除，没有拆成可审计的“记忆注入 section”。因此无法回答“这次 writer/reviewer/archivist 到底看见了哪条台账、为什么看见、哪条因为预算没看见”。

## 影响

- 产物链错位只能在 cleanup 前排查，成功章节反而缺证据。
- 无法证明 final_gate 校验的是最终落盘正文同一版本。
- 无法证明 Archivist 输入基于哪份正文、是否包含必要上下文。
- 近章现场只能证明“某些结果落账了”，无法证明“哪个角色基于哪个输入做出该结果”，尤其是 clean 成功章节。

## 治本方案

1. 定义最小审计留痕清单。
   - 每章一个 `runtime/audit/chapter_N.json`。
   - 保存各阶段输入/输出 hash、路径、字数、标题、POV、调用角色、模型名、通过状态。
2. clean 模式删除大文件前写 audit manifest。
3. 对关键输入保留短摘要或 hash。
   - `writer_input_hash`
   - `review_input_hash`
   - `editor_input_hash`
   - `archive_input_hash`
   - `final_text_hash`
   - `final_gate_checked_text_hash`
   - `fact_check_input_hash`
   - `fact_check_result_hash`
   - `fact_verify_hashes`
   - `summary_input_hash`
   - `summary_output_hash`
   - `archive_commit_hash`
4. 增加角色记忆注入 manifest。
   - 每次 LLM 调用记录 `role`、`chapter`、`input_hash`、`budget`、`threshold`、`final_tokens`、`model`。
   - 记录 `sections[]`：`title`、`source_path`、`source_type`、`trigger_reason`、`priority`、`compressible`、`tokens`、`status=included|compressed|excluded`、`reason`。
   - chunks 选择必须记录命中关键词/角色/场景、cap 截断、未命中或被排除原因。
   - FactChecker、summarizer、consistency mapper 要么纳入同一 manifest，要么写明豁免原因。
   - FactChecker 修复链必须记录每一次 final 文本版本的 hash，避免“fact_checker 改写后没有再门禁”无法追溯。
5. reports 模式扩大保留名单。
   - 至少保留 `archive_input.md` 和 `writer/final.md`，或保留对应 hash+节选。
   - reports 模式还应保留 `editor_input.md`、`fact_check.md`、`verify_*.md` 的 hash/短节选，方便定位修后版本。
6. 增加测试。
   - clean 后 audit manifest 仍存在。
   - manifest hash 与正式正文一致。
   - archive_input 被清理时仍能通过 hash 证明调用过。
   - reports 模式下至少可追溯 archive_input/final 文本对应 hash。
   - 预算裁剪导致某个 high/normal section 被排除时，manifest 必须记录排除原因。
   - chunk 因关键词命中被注入时，manifest 必须能追到触发关键词和来源文件。

## 验收标准

- 默认 clean 不再等于证据链消失。
- 每章成功后仍能追溯关键角色输入输出版本。
- 审计和恢复可以依赖轻量 manifest，而不是要求保留全部大文件。
- 每次角色调用都能解释“看见了哪些记忆、为什么、哪些没看见”。
