# 章节元数据清单与产物链对账修复计划

## 结论

系统缺少章节级 canonical metadata 清单。beat 里有标题、POV、张力、关键章等字段；summary 只存写作手法摘要；ledger/state 也不保存章节标题/POV/beat 关键字段。结果是正文、summary、ledger、beat 之间没有一个稳定的章节元数据对账点。

## 当前证据

- `scripts/pipeline/summarizer.py:18-29`：summary schema 只包含 chapter、动作、动词、句式、意象、情绪手段、plot_digest，没有标题、POV、beat_key_fields。
- `scripts/pipeline/summarizer.py:44`：summary 输入只传第 N 章定稿正文，不传 beat 元数据。
- `scripts/pipeline/state.py:43-51`：默认 ledger 是实体、资源、债务、约束、关系，没有 chapter manifest。
- `scripts/pipeline/archivist.py:1009-1013`：`_chapter` 只注入结构化更新供 merge 使用，没有持久化标题/POV/beat 字段。
- `scripts/pipeline/archivist.py:1028-1036`：台账版本快照只写 07/08 并推进 latest_chapter，没有章节元数据对账记录。
- `beats/chapter_60.json:2-4`：第 60 章 beat 有标题“官道”、POV“韩铮”。
- `runtime/summaries/chapter_060.json:1-8`：对应 summary 只有 chapter 和写作摘要字段，没有标题/POV。
- 第 94 章样本：`beats/chapter_94.json` 有标题“最后一味”和视角“沈安”，正文第一行标题一致；但 `runtime/summaries/chapter_094.json` 仍只有 `chapter` 与写作特征，没有 title、pov、final_hash、commit_status。
- 当前没有第 94 章 `final_commit/chapter_manifest/hash` 证据，无法机器证明 summary、Archivist、正文和 beat 属于同一提交版本。

## 根因判断

- 章节级事实散落在 beat、正文标题、summary、ledger 中，没有一个权威 manifest。
- summary 被设计为“写手表达防重复”，不是章节目录/元数据记录；但后续越来越多管路依赖 summary 作为事实回顾。
- Archivist 提交只推进 `latest_chapter`，没有保存该章产物链的 canonical hash 与元数据。

## 影响

- 无法自动发现标题、POV、关键章字段在 beat/final/summary/ledger 之间错位。
- 断点恢复只能看正文和 state 水位线，很难确认当前正文对应哪份 beat。
- 目录、索引、POV 分布、关键章统计等都需要临时从不同文件猜。

## 治本方案

1. 增加章节 manifest。
   - `runtime/chapter_manifest.json` 或 `runtime/chapters/chapter_N.json`。
   - 字段：chapter、title、pov、beat_hash、final_hash、summary_hash、archive_report_hash、key_fields、commit_status。
   - 增加 `archive_input_hash`、`review_verdict_hash`、`final_gate_hash`、`fact_check_status`。
2. 在 beat 生成、final 落盘、summary 生成、Archivist 提交四个节点更新 manifest。
3. summary 输入加入章节元数据。
   - 不要求 summarizer 判断，只要求原样保存 title/pov/chapter。
4. Archivist 提交后写入 committed manifest。
5. 增加对账测试。
   - beat 标题/POV 与 manifest 一致。
   - final 第一行与 manifest title 一致。
   - summary chapter/title/pov 与 manifest 一致。
   - hash 不一致时恢复路径不得默认放行。

## 验收标准

- 每章有一个机器可读的权威元数据记录。
- 正文、summary、ledger、beat 可以自动对账。
- 恢复、索引、统计不再靠临时解析多个产物。
- clean/reports 模式下仍能用 manifest 证明各产物版本一致。
