# 章节摘要提交态隔离修复计划

## 结论

summary 在 Archivist 提交前生成并落盘，但章节的正式提交水位线由 Archivist 最后推进 `state.latest_chapter`。因此存在“summary 已可被后续读取，但 ledger/state 尚未提交”的中间态。摘要产物缺少提交态/可见性屏障，会让后续防重复、reviewer 上下文或规划管路读到未正式提交的章节摘要。

## 当前证据

- `scripts/run_pipeline.py:949-955`：`generate_chapter_summary()` 在 Archivist 之前执行，失败也不影响正文。
- `scripts/run_pipeline.py:957-998`：之后才进入 Archivist 更新和提交。
- `scripts/pipeline/archivist.py:1032-1036`：`latest_chapter` 只有在 Archivist 全部写完后才推进。
- `scripts/pipeline/summarizer.py:36-52`：summary 直接写到 `runtime/summaries`，没有 pending/committed 状态。
- `scripts/pipeline/summarizer.py:105`、`scripts/pipeline/summarizer.py:155`：后续 writer/reviewer 会读取近期 summary，summary 已经是主链上下文供给者。
- `scripts/pipeline/core.py:101`、`scripts/pipeline/core.py:355`、`scripts/pipeline/core.py:379`：summarizer 没有纳入统一角色产物路径、空目录清理和章节过程清理，提交态也无法通过角色产物体系追踪。
- 审计时曾出现第 89 章 summary 已存在、progress 停在 Archivist、state 仍停 88 的窗口；该窗口由上述顺序决定，不依赖具体章节。
- 该风险属于主流程提交尾段：`run_one_chapter()` 先生成 summary，再进入 Archivist。若 summary 成功而 Archivist 连续失败，下一章 writer/reviewer/规划层可能读到“正文未提交但摘要已可见”的幽灵历史。

## 根因判断

- summary 是实现态派生产物，但没有和章节提交标记绑定。
- 后续消费者按文件存在读取 summary，没有判断该章是否已经 Archivist commit。
- “摘要失败不影响正文”是合理的，但“摘要成功即可对后续可见”缺少事务边界。

## 影响

- 崩溃或 Archivist 连续失败后，未提交章节的摘要会留在 runtime。
- 下一次恢复或并行工具读取 summary 时，可能把未提交章节当作历史。
- summary 与 ledger/state 不一致时，防重复和近期回顾会出现幽灵章节。

## 治本方案

1. summary 写入 pending 区。
   - `runtime/summaries_pending/chapter_N.json`
   - Archivist 成功提交后移动到 `runtime/summaries/chapter_N.json`。
2. 或在 summary 内写提交字段。
   - `commit_status: pending | committed`
   - `committed_after_latest_chapter`
   - `final_text_hash`
   - `beat_hash`
   - `summary_input_hash`
   - `model/provider/prompt_hash`
3. summary 消费函数过滤未提交章节。
   - 只读取 `chapter <= state.latest_chapter` 且 committed 的摘要。
4. 恢复流程处理悬空 summary。
   - 正文和 Archivist 补齐成功后提交。
   - 正文无效或验收失败时删除/隔离 pending summary。
5. 增加 scenario 测试。
   - summary 已写但 Archivist 失败，下一章 writer/reviewer 不应读到该 summary。
   - Archivist 成功后 summary 才变为可见。
   - 并入主流程调度层场景：mock `generate_chapter_summary()` 成功、mock Archivist 连续失败，断言 `load_recent_summaries()`/reviewer 重复检查/下一章 writer 输入均不可见 pending summary。
6. 与 `Summarizer主链角色产物与提交态统一计划.md` 联动。
   - summary 即使失败非阻断，也必须在章节审计 manifest 中留下 `summary_status`、输入 hash 和失败原因。

## 验收标准

- summary 可见性与章节提交水位线一致。
- 未提交章节的摘要不会污染后续上下文。
- 恢复路径能处理 pending summary。
