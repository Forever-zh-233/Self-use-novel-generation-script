# 章节时间锚点门禁计划

## 结论

Beat 输出的时间锚点缺少与当前 state/story_director/input 的一致性门禁。第 93 章 debug 输入和 state 都是“第二十五天上午”，正式 beat 却输出“第十五天上午”，这种中文数字错位没有被拦截。

## 当前证据

- `beats/_debug/第093章/beat_input.md:522`、`beats/_debug/第093章/beat_input.md:549`：输入侧当前故事日是“第二十五天上午”。
- `beats/chapter_93.json:5`：正式 beat 输出为“第十五天上午”。
- `runtime/state.md` 同期也显示当前时间线在第二十五天附近。
- 第 95 章正式 beat 的 `时间锚点` 是空字符串，见 `beats/chapter_95.json:5`，但没有阻断进入正文生成。
- 第 95 章 BeatPlanner 输入内部存在多源冲突：`beats/_debug/第095章/beat_input.md:507` 写 `第17.1日·上午·未知`，同一输入稍后又在 `beats/_debug/第095章/beat_input.md:556`、`beats/_debug/第095章/beat_input.md:588` 写“第二十五天上午”；当前 `runtime/state.md:4` 则是“第二十五天上午至午后”。
- 当前 98 章正式 beat 中，`时间锚点` 为空的有 50/98；除第 93 章倒退、第 95 章空锚点外，还存在第 10、11、55、65、86 章这类非空但无法解析为故事日索引的相对锚点，例如 `beats/chapter_65.json:5`、`beats/chapter_86.json:5`。

## 根因判断

- BeatPlanner 被要求承接当前时间线，但 normalize/方向检查没有把时间锚点当硬字段校验。
- 中文数字、阿拉伯数字、相对时间缺少统一解析。
- 章节 manifest 计划覆盖元数据记录，但还需要专门的生成时门禁防止错误 beat 进入 writer。
- BeatPlanner 输入没有单一权威时间源；state、recent events、debug input 片段可以同时给出不同当前日，模型只能猜。
- 空时间锚点被当成普通缺省字段放行，没有 `required/non_empty` 级别的 schema 门禁。

## 影响

- 正文可能按错误故事日生成，造成全局时间线倒退。
- Archivist 可能把错误时间写入 state，污染后续章节。
- 人类很难从长输入中及时发现“十五/二十五”这种小字差。

## 治本方案

1. 定义时间锚点 schema。
   - `story_day_index`
   - `day_label`
   - `time_of_day`
   - `source_state_hash`
   - `anchor_kind`
   - `ref_chapter`
   - `relative_time`
   - `day_advance_reason`
2. Beat 输出后做时间锚点门禁。
   - 与 state/story_director/debug input 当前日一致或合理推进。
   - 禁止无解释倒退。
   - 中文数字和阿拉伯数字统一解析。
   - 禁止空字符串进入 writer；无法判定时进入重生成或人工复核。
   - 输入侧多源时间冲突时先阻断 BeatPlanner，而不是让模型在冲突材料里自行取舍。
3. 失败时要求重生成 beat，不进入 writer。
4. 将时间锚点写入章节 manifest。
5. 增加 scenario 测试。
   - 输入第二十五天，beat 第十五天，必须阻断。
   - 输入含 `第17.1日` 与 `第二十五天` 两个当前时间源时，必须在 beat 前报错或生成 `time_source_conflict`。
   - beat 时间锚点为空字符串时，必须阻断。
   - 合理跨日推进可通过，但必须有 `day_advance` 或明确说明。
   - 相对锚点必须能解析为 `ref_chapter + relative_time`，不能只保留自然语言。

## 验收标准

- Beat 时间锚点与当前故事时间线一致。
- 明显中文数字错位会在写作前拦截。
- 时间锚点进入章节 manifest 供后续对账。
