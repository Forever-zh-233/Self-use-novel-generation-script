# 章节标题规范化中文数字漏检修复计划

## 结论

这是章节标题格式契约没有集中化的问题。主流程最终清洗只识别阿拉伯数字标题 `# 第87章` 或模板占位符 `# 第{N}章`，不识别中文数字标题 `# 第八十七章`。模型一旦输出中文数字，正式正文就会绕过标题归一化，门禁也不会把它当成标题格式错误。

这不是第 87 章正文要不要改的问题；第 87 章只是证明这个漏洞真实发生过。

## 当前证据

- `scripts/run_pipeline.py:906-912`：最终清洗使用 `^#\s*第(?:\d+|\{N\})章` 查找和替换标题，只覆盖阿拉伯数字和 `{N}`，不覆盖中文数字。
- `scripts/pipeline/gates.py:258-262`：hard gate 豁免标题行时同样只剥离 `^#+\s*第\d+章`，没有统一检查“正式标题必须是阿拉伯数字标准格式”。
- `tests/scenario_test.py:1405-1417`：现有测试只覆盖 `# 第3章` 标题行豁免和正文内 `第2章` 泄露，没有覆盖中文数字标题。
- `输出/文章/第087章.md:1`：实际产物为 `# 第八十七章 打听`，说明模型输出已经绕过清洗进入正式文章。
- 现有产物里还出现过 `# 第一章`、`# 第七章`、`# 第二十三章`、`# 第五十五章`、`# 第六十七章` 等，说明不是单次偶发。

## 根因判断

- 标题规范散落在 prompt、清洗正则、gate 注释和测试样例里，没有一个统一的 `normalize_final_heading()` 契约函数。
- 清洗逻辑把“能找到标题行”与“标题行符合正式格式”混在一起；找不到可识别标题时只是原样放过。
- 测试只验证“阿拉伯数字标题不误报”，没有验证“中文数字标题必须被改成标准标题”。

## 影响

- 正式文章标题格式不统一，目录、索引、后续脚本和人工检索都会出现不稳定输入。
- 如果正文前有模型思考过程，且标题是中文数字，`chapter_heading` 找不到锚点，前置噪声也可能一起落盘。
- gate 报告和最终落盘正文的格式契约不一致，使用者会误以为标题已经被强制修正。

## 治本方案

1. 抽出统一标题规范函数。
   - 输入：`final_text/chapter/title`。
   - 识别：阿拉伯数字、中文数字、`{N}`、缺标题、标题前噪声。
   - 输出：永远以 `# 第{chapter}章 {beat_title}` 开头。
2. 将 run_pipeline 的最终清洗改为调用该函数。
3. gate 增加标题契约检查。
   - 正文第一行不是标准标题时报告硬问题。
   - 标题行豁免只豁免标准标题，不豁免中文数字错格式。
4. 增加 scenario/quick 测试。
   - writer 返回 `# 第八十七章 打听`，最终必须变为 `# 第87章 打听`。
   - writer 返回无标题正文，最终必须插入标准标题。
   - writer 返回标题前噪声，最终必须裁掉噪声并保留标准标题。

## 验收标准

- 所有正式文章第一行统一为 `# 第N章 标题`，其中 N 为阿拉伯数字。
- 中文数字标题不会再原样进入 `输出/文章/`。
- 测试覆盖中文数字、缺标题、前置噪声三种路径。
