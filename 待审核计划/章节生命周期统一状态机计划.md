# 章节生命周期统一状态机计划

## 结论

当前系统没有统一章节生命周期状态机。`runtime/progress.json` 更像 UI 心跳，`runtime/state.json.latest_chapter` 只表达 Archivist 记忆提交水位线；draft/review/edit/final/summary/archive/consistency 各阶段靠文件存在、长度、局部异常处理来判断能否继续。结果是断点续跑、重复提交、失败豁免、clean 后审计和下游失效传播都只能靠各处补丁。

## 当前证据

- `scripts/pipeline/core.py:192`：`write_progress()` 接收任意 dict，没有状态枚举或迁移校验。
- `scripts/pipeline/core.py:255`、`scripts/pipeline/core.py:270`：批处理只写 `running/done`。
- `scripts/run_pipeline.py:1759`、`scripts/run_pipeline.py:1770`、`scripts/run_pipeline.py:1774`、`scripts/run_pipeline.py:1777`：主流程临时写 `chapter_error/halted_on_chapter/finished/stopped`，没有每章阶段状态记录。
- `scripts/run_pipeline.py:196-214`：恢复按正文存在、章号大于 `latest_chapter`、字数阈值筛选。
- `scripts/run_pipeline.py:933-939`：正文续跑只看最终正文汉字数是否达标，然后可跳过前面阶段。
- `scripts/run_pipeline.py:949-957`：复用正文后仍继续 summary 和 Archivist，没有“已 committed 章节不得重复 apply archive”的统一守卫。
- `scripts/pipeline/archivist.py:1034-1035`：`latest_chapter` 在 Archivist 最后推进；但 `scripts/run_pipeline.py:1000-1022` 还会在之后修改 `impact_seeds` 状态并写 ledger，这类后置副作用不在章节提交边界内。
- `scripts/run_pipeline.py:889`：FactChecker 残留时会“接受现状”，但没有统一 `waived` 状态、原因、hash 和有效范围。

## 根因判断

- 系统把“进度显示”“章节提交”“阶段缓存”“人工豁免”“一致性缓存”混在不同文件和局部判断里。
- 没有合法迁移表，任何阶段都可能用旧产物继续推进。
- 没有全局 waiver registry，局部“接受现状”容易伪装成通过。
- consistency fact、summary、ledger、active_threads 没有统一纳入章节生命周期，清理/回退/重写时容易混入旧时代产物。

## 治本方案

1. 定义章节状态机。
   - 主状态：`planned -> beat_ready -> drafted -> gated -> reviewed -> edited_or_skipped -> final_checked -> fact_checked -> final_committed -> summary_committed -> archive_committed -> completed`。
   - 异常状态：`failed`、`blocked`、`waived`、`report_only`、`invalidated`。
   - 每次迁移必须声明 from/to、触发角色、输入 hash、输出 hash、时间戳。
2. 每章写统一 lifecycle manifest。
   - 推荐落点：`runtime/chapters/chapter_N.json`。
   - 保存 stage status、输入输出 hash、prompt/schema/model 版本、失败/豁免原因、clean 保留证据。
   - `progress.json` 只从 manifest 派生显示，不再作为权威状态。
3. 恢复和续跑只读 manifest。
   - 已 `completed` 的章节重复进入时默认跳过或校验，不得重复 apply archive。
   - 上游 hash 变化时自动将下游阶段标记 `invalidated` 并从变化点重跑。
   - 第 N 章必须能证明由 N-1 的 committed 状态迁移而来。
4. 建立全局 waiver registry。
   - 字段：`scope`、`reason`、`human_note`、`text_hash`、`stage`、`expires_or_chapter`。
   - `accepted_as_is` 不等于 `passed`，必须在 manifest 中可见。
5. 将 consistency 纳入章节生命周期。
   - fact sheet 指纹包含正文 hash、beat hash、map prompt/schema/model 版本。
   - check 阶段也校验鲜度；watermark 使用 scan generation + fact hash，而不是只看最高章号。

## 验收标准

- 每章状态只能按合法迁移推进。
- 断点续跑不会靠文件存在/长度误复用旧产物。
- 已提交章节不会被重复归档污染台账。
- 人工豁免有范围、原因、hash 和过期条件。
- consistency、summary、archive 都能随章节重写/回退正确失效。
