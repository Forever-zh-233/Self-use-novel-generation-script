# 终止性事件防回滚管路修复计划

## 结论

这是管路级连续性问题：系统允许“最后一次 / 收束 / 告别 / 离开 / 死亡”等终止性事件只以自然语言存在于弧线、beat、正文和台账里，没有结构化生命周期状态。后续 beat_planner / writer / archivist 看不到“这类事件已经关闭，不可按原模式重复”，于是已收束关系可以被下一章重新写成延续事件。

## 当前证据

- `runtime/active_arcs.json:36`：陈寡妇 hidden_agenda 写“她最后一次复诊时递出了丈夫名字和铜牌”。
- `runtime/active_arcs.json:41`：角色 beat moment 写“最后一次复诊……走了”。
- `runtime/active_arcs.json:86`：第 80 章节点写“陈寡妇最后一次复诊”。
- `beats/chapter_80.json:7`：第 80 章戏剧目的写“陈寡妇最后一次复诊并递出丈夫名字和铜牌信息”。
- `beats/chapter_80.json:11`：第 80 章冲突再次定义“陈寡妇最后一次复诊”。
- `输出/文章/第080章.md:83`：陈寡妇说出“我男人叫陈守义”。
- `输出/文章/第080章.md:141`：铜牌被收回，形成“说出但仍留着”的收束状态。
- `输出/文章/第082章.md:95`：后续正文又写“第二个是陈寡妇，复诊”。
- `07-动态状态台账.md:2655`：台账记录“沈安-陈寡妇：延续复诊”。

## 根因判断

- active_arcs 和 beat 使用“最后一次复诊”这种自然语言表达终止状态，但没有写入统一的 `closed_event / relation_lifecycle`。
- archivist 可以把后续正文继续登记为“延续复诊”，没有提交前校验“该关系是否已被终止事件关闭”。
- reviewer / fact_checker 的上下文主要检查事实穿帮和角色卡，没有专门的“终止性事件回滚”规则。
- story_director 只看整体节奏，可能认为线已收束，但它的判断不会反向约束台账与后续 beat。

## 影响

- “最后一诊”的情绪重量被削弱，原本第 80 章的收束不再成立。
- story_director 继续认为“陈寡妇线在 80 章自然收束”，但台账与正文已经变成未收束。
- 后续 beat_planner 可能继续把陈寡妇当作普通复诊角色使用，导致线索已经递出后还反复回到同一医患循环。

## 治本方案

1. 建立终止性事件注册表。
   - archivist 遇到“最后一次、收束、告别、离开、死亡、断绝、身份暴露不可逆”等事件时，写入结构化字段。
   - 建议字段：`closed_events[]`、`relation_lifecycle[角色/关系].status`、`closed_reason`、`allowed_future_forms`。
2. 在 beat_planner 输入中注入“不可回滚事件摘要”。
   - 不只注入最近台账，还要明确列出“已关闭的关系/事件”和允许的后续形态。
   - 例如：可“托人带话 / 消息余波 / 遗留物影响”，不可继续同类“普通复诊”。
3. 在 reviewer / fact_checker 增加终止性事件回滚检查。
   - 如果正文出现已关闭事件的同类重复，应作为连续性问题输出。
4. 在 archivist 提交前反查。
   - 如果 archivist 准备把已关闭关系写成“延续”，提交前阻断或要求标记为“重新开启，并说明依据”。
5. 增加测试。
   - 构造第 N 章注册 `closed_event`。
   - 第 N+1 章 fake beat 或正文重复同类事件。
   - 断言 beat/reviewer/archive 任一守门点能捕获，且台账不会静默写成延续。

## 验收标准

- 终止性事件不再只存在于自然语言，而有结构化 lifecycle 状态。
- 后续 beat/reviewer/fact_checker/archive 都能看到同一份不可回滚摘要。
- 已关闭事件被重复使用时会触发明确告警或阻断，不会静默进入台账。
- 测试覆盖“最后一次后再次同类出场”的样本。
