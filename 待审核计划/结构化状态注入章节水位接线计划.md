# 结构化状态注入章节水位接线计划

> 状态：待审核。本文件只记录治本方案，未修改代码或运行产物。

## 结论

`structured_state_text(chapter=0)` 已支持按章节水位裁剪旧角色和关系，但多个主流程入口调用时没有传入当前章节号，导致裁剪逻辑实际失效。未来章节的规划、卷纲生成和记录员输入可能继续混入过早、过旧或与当前章无关的角色状态。

## 当前证据

- `scripts/pipeline/state.py:163`：`_trim_state_for_context(state, chapter=0)` 只有收到章节号时才执行上下文裁剪。
- `scripts/pipeline/state.py:179-195`：角色和关系裁剪逻辑均受 `if chapter` 控制。
- `scripts/pipeline/state.py:254-255`：`structured_state_text(chapter=0)` 把章节号传给 `_trim_state_for_context()`。
- `scripts/run_pipeline.py:401`：Archivist 输入调用 `structured_state_text()`，没有传当前章节。
- `scripts/pipeline/planning.py:221`：StoryDirector 输入调用 `structured_state_text()`，没有传当前章节。
- `scripts/pipeline/planning.py:614`：VolumePlanner 输入调用 `structured_state_text()`，没有传当前章节。

## 根因判断

- 裁剪能力已经写在 state 层，但调用端没有形成“所有章节态上下文必须带 chapter”的契约。
- 默认 `chapter=0` 适合全局诊断，不适合主流程生成输入。
- 测试只验证了函数本身有界，没有覆盖主流程入口是否传入章节水位。

## 影响

- 结构化状态看似经过裁剪，实际仍按全局态注入。
- 旧角色、旧关系、旧事件可能污染未来章节规划。
- token 成本会随项目增长继续上升，和“按需注入”的目标相冲突。

## 治本方案

1. 明确调用契约。
   - 生成章节相关输入时必须传入当前 `chapter`。
   - 只有诊断命令或全局导出允许使用 `chapter=0`。
2. 为主流程入口补齐章节水位。
   - StoryDirector、VolumePlanner、Archivist 等入口统一传当前章节。
   - 新增入口必须经同一 helper 构造结构化状态 section。
3. 增加防呆。
   - 在非诊断模式调用 `structured_state_text()` 且未传章节时记录警告或测试失败。
4. 增加场景测试。
   - 构造一个 40 章前出现、当前章无关的角色，断言主流程输入不会注入。
   - 断言 `run_pipeline.py` 和 `planning.py` 的章节态入口均传入当前章号。

## 验收标准

- 章节相关角色输入不再被无章节水位的全量状态污染。
- 结构化状态调用点有明确的“章节态/全局态”分类。
- 测试能捕捉新增入口忘传 chapter 的回归。
