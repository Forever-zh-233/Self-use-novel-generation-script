# 自动精品小说生成系统治本总计划
首要目的是能挂机跑通宵，所有设计都得为总目的让路，冲突了就想办法优化，既能挂机又能写好文章的那种
> 状态：待审核（已按代码审计结论做减法校正——区分真 bug 与镀金、要求现网核实、对单机管线保持克制、补回被漏掉的高优先字段断链）。
> 本文件是本轮只读审计产出的散件计划收敛版；只记录未来生成系统的治本方案，不修改代码、配置、测试或已落盘文章。
> 本计划的目标不是“修好当前章节文本”，而是让未来章节在同一模型上尽量发挥上限：管路接对、状态可信、职责不污染、质量能拦截、token 不爆炸。
> 入口规则：从本次整理开始，`待审核计划/自动精品小说生成系统治本总计划.md` 是唯一总入口；目录里的其他散件只作为原始审计证据和细分执行说明，后续新发现先并入本文件，再决定是否保留独立散件。
> **分级原则（贯穿全文）**：每个 Epic 的"治本动作"不是平级待办。AI 批量产出的散件普遍"诊断对、方案过重"，落地时一律按"真 bug 先修 → 方案砍到最小子集 → 镀金删除"处理。`【降级】【裁剪】【删除·镀金套娃】` 标记即本轮校正结论。

## 一、总判断

当前系统不是单一 bug，而是“能跑的生成流水线”正在向“通用精品小说软件”过渡时暴露出的结构性问题：

1. 运行态水位不够可信：计划态、事实态、提交态、恢复态没有统一状态机。
2. 职责与输入契约不够硬：prompt 声称能做的事，实际输入未必给得到。
3. 通用职责被当前书内容污染：旧入口、配置、脚本、示例和安全版设定仍写死当前书。
4. 规划链没有事实回写：已经兑现的事件、部分回收的伏笔、节点提前消费等会继续污染未来规划。
5. 质量闭环还不够强：能生成“干净稿”，但不能稳定拦住平庸、假冲突、不可演 beat。
6. 一次性核心资产生成太弱：全书骨架、成长轨迹、结构参考、profile 播种还不够自动化和可审核。
7. token 治理必须作为一等约束：禁止靠疯狂塞提示词解决质量问题，必须按需注入、分层摘要、权威源检索。

## 二、硬边界

- **【第一性原则·高于一切】首要目的是挂机跑通宵不中断。一切设计为它让路,冲突了就优化掉冲突,不是牺牲通宵。**
  - 通宵跑的敌人按致命度排序:① 永久挂死(进程卡住)② 中途 raise 停机(停在某章,后面全没跑)③ 写出有瑕疵的正文(可事后清理)。**绝不允许用②去防③**——即"宁可标记瑕疵继续跑,也不为质量问题停机"。
  - 质量门禁检出问题时,默认动作是**标记 + 隔离 + 继续跑**(记入 `输出/_待复核章节.md`,早上人工复核重跑),**不是 raise**。只有用户显式开严格模式(`fail_on_final_gate=true`,白天盯着跑时)才停机。
  - 任何外部调用(LLM/HTTP)必须有总时长硬超时 + 重试,超时重试耗尽也只停当前章不挂死整体;正文先落盘、台账后记,停章不丢账、下次续上。
  - 新增的质量校验/门禁,先问:它会不会因一次 LLM 误报或格式抖动而停机?会 → 改成标记不停机。
- 不建议修改已落盘文章。
- 不把当前书具体内容写进通用职责。
- 当前书例子只能进入 book profile、fixture、story bible 或待审核样例，不进入通用 prompt 和通用代码默认逻辑。
- 所有治理优先面向未来章节和未来换书。
- 所有 LLM 输入必须有预算策略：critical 少而硬，high 有界，normal 可压缩，历史只按需检索。

## 二之一、反过度修补总闸（执行者是 AI，本节最高优先，违反即停手重想）

> 本计划的散件由 AI 批量产出，已确诊一个系统性通病：**诊断对、方案过重**——把一两行能修的问题膨胀成状态机/manifest/审计层。下一个执行的 AI 极易读完就照重方案施工。本节就是为了阻止这件事。

**落地任何一条之前，先回答这三个问题，写进该子项：**
1. 这个问题**最少几行 / 什么改动**能解决？(加一个字段 / 写前查重 / 一次 hash 比对 / 改一句 prompt / 删一份冲突文档 —— 这些是默认档位)
2. 我的方案有没有引入下面这些"重型零件"？
3. 如果有，我能不能举出一个**当前真实复现**(不是"将来可能")的失败场景来证明非它不可？举不出 → 砍掉，退回默认档位。

**重型零件黑名单**（问题本身是单机顺序管线时，默认全部砍掉）：
- 状态机 / 合法迁移表 / waiver registry
- session_id / owner 校验 / hostname / 分布式锁(单用户单进程不存在并发双开)
- book_id / 多租户 / 多会话并存(就一本书、一个人在跑)
- 两阶段提交 / commit marker / 幂等键(单机改用"写前查重"即可)
- 全链路 manifest / 每阶段 sidecar / hash 链 / DVC 式 lockfile(只存一两个关键 hash 就够)
- event_key 全套 / canonical 对账器 / 在已有 reviewer 之上再建独立审计层
- 为"未来换书 / 未来并发"预埋的字段(换书是独立目标，不许搭车进每个 bug 修复)

**三条结构性禁令：**
- **禁止元计划套娃**：不为其他计划做"统一抽象层 / schema 收敛层"。统一只在"两处以上已确认要做、且真实重叠"时才做。
- **一事一套**：同类记录机制(hash / manifest / 凭证)全项目只能有一套且最小，不许各 Epic 各造一套互相覆盖。
- **创作判断永不硬编码**：文风、可演性、张力、配角是否鲜活、该不该切多角度——交 LLM reviewer 看意图判断。代码只查"有唯一正确答案"的客观事实(配额有没有被消费、专名有没有写错、字段有没有缺)。

> 标记约定：下文治本动作中，`【最小】`=按默认档位做；`【裁剪】`=砍掉重型零件后保留核心；`【降级·远期】`=当前不做，仅留方向；`【删除】`=镀金套娃，不执行。

治本动作（按“先最小补丁、状态机只作远期方向”排序；不要为单机顺序管线一次性引入全套状态机/manifest/session）：

1. 章节水位可信——最小补丁优先，状态机是方向不是前置。
   - **真正要做的最小集**：台账/摘要落盘做幂等去重（同章不重复追加）；落一个轻量 `final_committed` 凭证（正文 hash + 已过门禁标记），恢复时按它判断而不是按文件存在猜测。
   - **不做**：`planned->beat_ready->...->archived` 七态状态机 + 全局 waiver registry + 每章全量 manifest。这是分布式系统级方案，对单用户顺序管线是过度设计；它解决的重复提交/缓存错位问题，上面的幂等去重 + 单一凭证已覆盖。状态机仅作为“若将来真出现并发/多入口”时的预留方向，落地前必须重新评估是否仍需要。
2. 运行锁——只补 owner 校验，不做会话化。
   - **最小集**：`release` 前校验锁的 owner（PID）再删，避免误删他人锁；保留已有的 stale-PID 探活。
   - **不做**：session id / requested_action 建模。单用户单机几乎不存在并发双开，会话化是镀金。
3. 建立产物证据链（轻量版）。
   - 关键阶段产物记录：chapter、input fingerprint、prompt 路径 + hash。够用即可，不必给每个产物都挂全套 source-files 清单。
   - 恢复时优先按 `final_committed` 凭证对账，而不是按文件存在猜测。
   - prompt 留档、input 留档、输出留档必须分文件分 hash；`beat_prompt.md` 不得写成 `beat_input.md` 的副本。
   - `beat.meta.json` 同时记录 prompt template path/hash、assembled input hash、role config hash；若 prompt/input hash 相同，直接判定证据链失真并阻断续跑。
4. 启动前做水位门禁。
   - 禁用断点恢复时，如果 runtime 显示上一章未完整提交，必须阻断或进入人工确认。
5. cleanup 只能按权威目录清理。
   - 不允许根据旧文件名模式误删或漏删。
   - 正式骨架、正式卷纲、chunks 等审核后核心资产不得被章节清理脚本当作可重跑产物删除；草稿和运行态必须另设目录。
6. Ledger 实体摘要事实鲜度对账。
   - `ledger.entities[*].summary / arc_core.want / status / first_seen / last_seen` 必须和 state、relationships、chapter summaries 对账。
   - 若实体摘要仍停在“未登场/计划态”，但 state 显示近期活跃或关系已推进，输入构造前必须报警或重写摘要。
   - 这类问题的疑似样本是：`runtime/ledger.json:1481`、`runtime/ledger.json:1518` 仍把小满写成未正式登场，而 `runtime/state.json:653` 显示 `_last_active=104`，`runtime/state.json:909` 已有沈安-小满关系推进，`输出/章纲/第105章_beat_input.md:495/540` 又注入了她的当前随行状态。**注意：以上行号/样本必须在动手前重新 grep 现网核实——当前已进入第二卷，部分引用可能来自已清理重跑的旧产物;若现网已不复现，本条降级或关闭。**
7. StructuredState 当前场景和近期活跃分域。
   - `当前可在场角色`、`当前地点/随行角色`、`近期活跃/近期相关角色` 必须分 section。
   - `_last_active` 窗口筛出的角色不得命名为“在场角色”，避免 BeatPlanner 把远处、已离场、死亡或未出场角色当作空间布局和多角度叙事素材。

验收：

- chapter 105 这种 in-progress 产物不会被当成已完成章节。
- 断点恢复不会从半提交状态继续污染后续。
- 任何章节可以回答：当前最终正文是哪份、摘要是哪份、账本提交是否完成。
- 同一输入中不会同时出现“角色未登场”和“角色已长期随行/关系已推进”的事实冲突。
- 章纲阶段能复核当时使用的 `prompts/beat_planner.md` 版本，不会把输入指纹误当 prompt 指纹。
- cleanup 后正式 `卷纲/10-卷纲.md` 不会被误删，也不会因误删触发自动卷纲重建。
- 第 105 章这类样本里，远处、已离场、死亡、未出场角色不会被放进“在场角色”栏。

## 五、Epic 3：Master / Volume / Arc / Beat 规划合同保真

目标：骨架、卷纲、弧线、beat 之间有可验证合同；计划态和事实态不混；已兑现、不该提前、部分兑现的内容都能被正确处理。

治本动作：

1. Beat 可演性门禁【裁剪·防硬编码】。
   - beat 缺"舞台/冲突/选择/代价/转折/尾钩"这类**字段为空**时退回重写——这是客观的(字段空=漏了)，可硬查。

2. Best-of-N 只在关键章或高风险章启用【最小·当前默认关闭，低优先】。
   - 默认关闭，无现网损耗；要做就只加"关键章降档罚分"，不建独立 LLM judge(昂贵新特性)。
3. Reviewer 评分结构化【最小】。
   - 分项分数、硬失败原因、是否返工必须落盘(`parse_review_verdict` 当前丢了 scores)。
   - 高分不能掩盖硬伤；评分与返工布尔一致性校验和"分项落盘"合并成一个改动做。
4. Editor/FactChecker/FinalGate 形成闭环【最小·第一阶段高优先】。
   - 所有正文改写后统一过 final_gate，hard issue 默认阻断(当前 `fail_on_final_gate` 默认 False、type_guard 恒 passed、残留穿帮"接受现状"继续推进——这是真 bug)。
   - editor 必须拿到 beat 摘要 + 知道 AI 腔约束，否则会把干净句改回"不是 X 是 Y"。
5. 负样本测试【最小】。
   - 建立"干净但平庸""任务清单式 beat""已有能力被无视""伪主题总结"等负样本。
6. **多角度退化专项【对症·新增，这是"经常不切多维度"的真正修复】。**
   - 病根：从 arc 填 angles → beat 吸收 → writer 落实 → reviewer 评判，**整条链是软请求，每一环都能合法地不干活，且没切反而因"单视角更稳"被 reviewer 奖励**。所以模型默认退化成单视角。
   - 修法按"客观硬查 vs 创作判断"分工：
     - **arc 层(客观配额，可硬查)**：规划完检查"配角与主角同场 ≥2 章的区间，`in_chapter_angles` 是否为空"，空则退回让它补。这不替它判断切哪，只确保它**给出配额**。
     - **beat 层(客观吸收)**：arc 下发了某章 angle，beat 的"多角度叙事"字段不许填"无"——要么落实、要么显式写明为何不适用。是"硬约束吸收"的最小版，不用 event_key。
     - **reviewer 层(创作判断)**：切得自不自然、配角活不活，交 reviewer；**并修掉它"全章单一视角更稳=高分"的标准冲突**(散件已列)，否则前面硬查到位了 reviewer 还在反向拉。
     - **writer 层**：保留章内短切实战指南，不加硬约束。
7. **落盘症状负样本专项【新增·高优先】。**
   - AI 腔禁令失效：`不是A是B` 从 Ch1 贯穿 Ch107，说明 prompt 禁令无效，必须进入确定性扫描 + Reviewer blocker + Editor 定向改写 + FinalGate 阻断。
   - 钩子失约与原地打转：Ch70-Ch72 钩子后，Ch73-Ch76 没承接反而退回早先情绪，需建立短期 hook ledger 和回应窗口。
   - 可删除弱章：Ch98-Ch100 氛围强但主线弱连接，需做删章影响检查，孤立氛围段必须绑定主线后果。
   - 机制欠解释：Ch104/Ch107 “了结方式不符”只给失败结果不给可行动依据，需区分留白和欠解释。
   - 压力线断供：Ch101-Ch107 换到雾南泽后清理指令/窑厂线紧迫感消失，需跨场景延续压力线。
   - 系统仪表盘化：系统不能长期只显示方向、等级、债务；系统事件必须改变选择、能力、代价、局势或后续任务。

验收：

- 不可演 beat 在写作前被拦住。

## 九、token 与成本治理

原则：质量提升靠“更准的信息和更强的门禁”，不是靠把所有材料塞进提示词。

治本动作：

1. 所有角色输入统一走预算层或显式豁免。
2. priority 语义硬化。
   - `critical` 必须少而不可压缩；`high` 是高优先级但仍有上限，不能等同无限保留。
   - `high + compressible=True` 不得承载会被改写后失真的硬事实；确需压缩时必须有事实回验。
3. 压缩后再次硬预算校验。
   - 压缩器返回后必须重新计算 token；仍超上限时降级、分段或阻断，不能把 `max_input_tokens` 变成软提示。
4. 所有 `call_role` 子路径都接预算层。
   - 正常 writer、FactChecker 修复 writer、第二轮事实修复 writer、内联 prompt、非根目录 prompt 都必须记录 role、预算、输入 hash 和豁免理由。
5. 长历史只走有界摘要 + 按需检索。
6. 正文全文只给必须看全文的角色；其余给片段、摘要、manifest。
7. 伏笔/关系/技能/角色卡按“本章相关 + 临期 + 最近变动 + 高强度”选择。
8. BeatPlanner 等规划角色对正式卷纲做窗口化，不把整卷未来事件全文塞入当前章。
9. Best-of-N 默认关闭，只在关键章、高风险章或质量连续低迷时启用。
10. 每章记录角色输入 token、输出 token、耗时和重试原因。

验收：

- 800 章规模下 planner/writer/reviewer 输入仍有上限。
- 调整 `max_input_tokens` 对所有配置角色真实生效。
- `high` section 不能绕过预算；压缩后的输入仍必须低于角色上限。
- FactChecker 修复链调用 writer 时也受 `writer` 预算控制，不再直接拼“穿帮报告 + 完整正文”旁路。
- 能归因“生成慢”是模型耗时、重试、压缩、Best-of-N 还是等待退避。

## 十一、外部研究增补区

本轮已查阅 Novelcrafter、Scrivener、Yarn Spinner、Twine、Ink、Plottr、Campfire、bibisco、World Anvil，以及 Re3、StoryWriter、PlotMachines 等长文本故事生成研究的公开资料。只吸收机制，不复制文本或代码。

可复用机制：

1. Codex / Story Bible 分层。
   - Novelcrafter 把 Codex 作为世界 lore、故事信息和 AI 可引用资料的集中区，并有 aliases、mentions、relations、progressions/additions、series/book codex 等概念。
   - 对本系统的启发：Book Profile、runtime canon、series canon、craft examples 必须分层；别名、关系、进展必须是可追踪字段，不是散落自然语言。
   - 对应 Epic：Book Profile、Ledger 实体事实鲜度、部分回收生命周期。
2. 场景卡 / 大纲 / 正文同源。
   - Scrivener 的 Binder、Corkboard、Outliner 是不同视图，但底层条目顺序、synopsis 和 metadata 彼此联动。
   - 对本系统的启发：章节 manifest 应成为同源实体；beat、synopsis、draft、review 状态、metadata 都挂在同一 chapter id 下，不再多文件猜水位。
   - 对应 Epic：章节生命周期、产物证据链、测试矩阵。
3. 节点与跳转合同。
   - Yarn Spinner 使用 node title、headers、body、options 管理叙事节点；node title 是跳转和运行入口，headers 可承载 location 等元数据。
   - Ink 用 knots/stitches/diverts 组织叙事片段，并用变量保存叙事状态。
   - 对本系统的启发：Master/Volume/Arc/Beat 应有稳定 node/obligation id、not_before、depends_on、fulfilled_by，而不是只靠自然语言“下一节点”。
   - 对应 Epic：规划合同保真、弧线节点事实水位重定位。
4. 变量状态外置与类型约束。
   - Yarn Spinner 明确变量有类型，并由外部 variable storage 管理；Twine 区分 story/global variables 与 temporary/local variables。
   - 对本系统的启发：runtime canon、session 临时态、草稿态、示例态必须分域；长期事实对象必须有 schema/type 校验。
   - 对应 Epic：运行水位、Upsert 非破坏式合并、Archivist schema。
5. 上下文按需引用。
   - Novelcrafter 的 Codex 资料有“何时给 AI 信息”的组织问题；其课程也强调 scene context、relations、research notes、series consistency。
   - 对本系统的启发：不要把全量资料塞给所有角色；用本章相关实体、临期债务、当前节点、关系链和高强度事实做检索注入。
   - 对应 Epic：token 与成本治理、角色输入预算统一接线。
6. 时间线生成大纲，而不是大纲和时间线双写。
   - Plottr 的 Timeline 是组织章节、情节线和场景卡的中心，Outline 由 Timeline 的内容和结构自动生成。
   - 对本系统的启发：骨架/卷纲/弧线/beat 应共享同一组事件对象；Markdown 大纲只是视图，机器合同应以事件对象和顺序/依赖为源头。
   - 对应 Epic：一次性核心资产自动生成、规划合同保真。
7. 场景标签驱动分析视图。
   - bibisco 强调给场景标注角色、地点、时间、POV 和叙事线，并据此做角色分布、地点分布、叙事线分布分析。
   - 对本系统的启发：beat/final/summary 落盘后应能反向更新“场景标签事实”，用于检查某条叙事线长期断档、某角色过久未活跃、某地点路线跳跃。
   - 对应 Epic：运行水位、三线节奏、质量闭环。
8. 手稿旁路的 lore 引用要按章显式关联。
   - Campfire、World Anvil 都把百科/世界观条目、时间线、地图或手稿放在同一工作流中，并支持在写作时关联或链接相关条目。
   - 对本系统的启发：Writer/Reviewer 不应读全量世界观，而应由章节 manifest 显式列出本章相关人物、地点、物件、秘密和时间线事件，再按需取摘要。
   - 对应 Epic：Book Profile、token 与成本治理、产物证据链。
9. 计划、当前状态、候选筛选和修订要分层。
   - Re3 用结构化总计划、当前故事状态注入、候选 rerank 和事实一致性编辑来提升长故事连贯性。
   - StoryWriter 使用 outline/planning/writing 多代理，并让 writing 根据当前事件动态压缩历史。
   - PlotMachines 的核心启发是 outline-conditioned generation 需要跟踪动态 plot state。
   - 对本系统的启发：Best-of-N、Reviewer、FactChecker 和 Archivist 不应互相替代；当前事件相关的历史压缩必须由事件 id 和动态 plot state 驱动，而不是整卷/整账本全量塞入。
   - 对应 Epic：质量闭环、token 与成本治理、规划合同保真。

## 十二、完成定义

这个系统达到“在不换模型的前提下尽量稳定生成高质量小说”的最低完成定义：

1. 换书不污染：通用职责不含当前书硬编码。
2. 水位可信：不会从半提交、旧草稿、错章号继续生成。
3. 规划可信：骨架、卷纲、弧线、beat 的合同可追踪。
4. 状态可信：已兑现、部分兑现、未兑现、终止性事件都能正确进入未来规划。
5. 质量可信：平庸稿、不可演 beat、假冲突、事实穿帮能被门禁拦住。
6. 成本可信：长篇规模下 token 不随历史无界膨胀。
7. 自动资产可信：全书骨架和 profile 可以自动生成草案，但必须审核后投入主流程。
8. 落地可信：所有治本方向都能裁剪成单机顺序管线的最小补丁，并有现网证据、消费端核验和回归测试。

结论：修完这些后，系统不能被承诺“必然一键精品”，但会从“会跑的流水线”升级为“有 profile、有合同、有水位、有质量闭环、有成本纪律的长篇小说生成软件”。这才是稳定逼近模型上限的治本方向。
