# 角色级重试退避配置化修复计划

## 结论

这是重试配置治理不统一：`config/run.json` 暴露了 `per_chapter_retries` 和 `retry_backoff_seconds`，但它们只控制整章失败后的重试。单个角色/API 调用内部还有独立的超时重试、风控拒绝重试、截断扩容重试，参数硬编码在 `pipeline/api.py`。实际耗时和失败策略无法由主配置统一治理。

## 当前证据

- `config/run.json:14-15`：配置 `per_chapter_retries=3`、`retry_backoff_seconds=45`。
- `scripts/run_pipeline.py:1761-1763`：整章失败后按 `retry_backoff_seconds * attempt` 等待。
- `config/run.json:50`：配置 `request_timeout_seconds=240`。
- `scripts/pipeline/api.py:276-277`：`call_role()` 内部默认 `reject_retries=3`、`truncate_escalations=2`。
- `scripts/pipeline/api.py:303`、`322`：角色级请求超时/风控拒绝使用硬编码 `time.sleep(min(5 * attempt, 20))`。
- `scripts/pipeline/api.py:121-125`：单请求还有 `timeout * 1.5 + 30` 的 hard timeout。
- `scripts/run_pipeline.py:673`、`scripts/run_pipeline.py:724`、`scripts/run_pipeline.py:768`：writer、reviewer、editor 之间会按 `sleep_seconds_between_calls` 固定等待。
- `scripts/run_pipeline.py:982`：Archivist 单章内重试也会按同一 `sleep_seconds_between_calls` 等待。
- `scripts/run_pipeline.py:796-902`：FactChecker 位于 final_gate 后，还可能触发 writer 修复、verify、二次 writer 修复、二次 verify，全部串行执行。
- `scripts/run_pipeline.py:1717-1751`：常规章节循环是强顺序串行；除一次性 analyst map 阶段外，章节生成没有并发。
- `scripts/run_pipeline.py:1069-1098`、`scripts/run_pipeline.py:1248-1268`：analyst 的 REDUCE/MAP worker 也有独立退避策略；MAP 虽有并发，但金丝雀、MERGE、REDUCE 仍有串行和硬编码等待。

## 根因判断

- 系统有“整章重试”和“角色调用重试”两层策略，但配置文件只暴露第一层。
- 用户看到 `retry_backoff_seconds` 容易以为控制全部重试等待，实际不控制角色内部退避。
- 不同模块（主生成、analyst、API）各有自己的退避策略，缺少统一说明和可调参数。
- 生成慢的来源不是单一 sleep：固定 sleep 只是每章数秒级；主要耗时来自多角色 LLM 串行、FactChecker 修复链、角色内部重试/截断扩容、整章失败退避。

## 影响

- 生成慢时难以判断是整章 backoff、角色内部重试、超时 hard timeout 还是 analyst 退避。
- 想降低等待或提高稳定性只能改代码，不能通过配置治理。
- 多角色串行时，隐藏重试会放大单章耗时，进度日志却不一定清晰显示所有等待来源。

## 治本方案

1. 在 run config 中新增 `role_retry` 配置。
   - `request_retries`
   - `reject_retries`
   - `truncate_escalations`
   - `role_retry_backoff_seconds`
   - `max_role_backoff_seconds`
   - `hard_timeout_factor`
   - analyst/map/reduce 的 retry/backoff/hard_timeout 也要纳入统一说明或单独命名的 analyzer_retry 配置。
2. `call_role()` 从 run_cfg 或统一 API config 读取这些参数。
3. 日志明确区分两层重试。
   - `[role_retry] writer 第 2/3 次`
   - `[chapter_retry] 第 N 章第 2/3 次`
4. 文档更新。
   - 说明整章重试、角色重试、analyst 重试分别控制什么。
   - 增加“单章耗时构成”说明：规划角色、beat、writer、reviewer、editor、summarizer、archivist、fact_checker 修复链分别可能增加一次或多次 LLM 调用。
   - 明确 `sleep_seconds_between_calls` 只控制少数阶段之间的固定间隔，不代表主要生成耗时。
5. 增加测试。
   - fake timeout，断言角色级重试次数来自配置。
   - fake 风控拒绝，断言退避参数来自配置。
   - 整章失败仍使用 chapter retry 配置。
   - analyst MAP/REDUCE 的退避边界在文档和配置中可解释。
   - `sleep_seconds_between_calls=0` 时主链固定等待可被测试确认。

## 验收标准

- 所有角色/API 重试参数可配置。
- 日志能看出等待来自哪一层。
- `retry_backoff_seconds` 不再被误解为控制全部等待。
