# 运行会话锁原子化与所有者校验计划

> 状态：待审核。本文件只记录治本方案，未修改代码或运行产物。

## 结论

主流程锁当前是“先检查文件是否存在，再写 PID”，不是原子互斥；释放锁时也不校验锁是否属于当前进程。两个流水线可能在竞态窗口同时进入同一工作区，或者一个进程误删另一个进程的锁，导致正式产物和运行状态交错写入。

## 当前证据

- `scripts/pipeline/core.py:275-290`：`acquire_lock()` 先检查 `LOCK_FILE.exists()`，清陈旧锁后再 `write_text()`。
- `scripts/pipeline/core.py:293-297`：`release_lock()` 无条件 `unlink()`，不校验 PID/owner。
- `scripts/run_pipeline.py:1676-1684`：Analyst 分支 acquire/release lock。
- `scripts/run_pipeline.py:1696-1700`：章节主流程 acquire lock 后继续运行。

## 根因判断

- 文件锁实现没有使用原子创建语义。
- 锁内容只有 PID，没有 session id、started_at、command、owner 校验。
- release 没有确认当前进程持有锁，存在误删有效锁的风险。

## 影响

- 两个流水线可同时写 `输出/文章`、`beats`、`runtime/state.json`、`ledger.json`、`07/08`。
- 提交水位、台账、章节正文可能互相覆盖，后续恢复也无法判断谁是权威。
- 长时间 Analyst 与章节生成并发时，chunk/index 或运行态可能互相影响。

## 治本方案

1. 使用原子锁创建。
   - Windows 可用 `os.open(..., O_CREAT | O_EXCL)` 或跨平台 filelock。
   - 创建失败时读取锁信息并判断是否 stale。
2. 锁文件写入 session manifest。
   - `pid`、`session_id`、`started_at`、`command`、`cwd`、`hostname`。
3. release 校验 owner。
   - 只有当前 session_id/pid 匹配时才能删除。
   - 不匹配时记录告警，不删别人的锁。
4. stale lock 清理需二次确认。
   - PID 不存在且 session 超时才允许清。
   - 清理写审计日志。
5. 增加并发测试。
   - 两个 acquire 同时执行，断言只有一个成功。
   - A 持锁时 B release，断言锁仍存在。

## 验收标准

- 同一 workspace 同一时间只有一个运行会话能进入写路径。
- 锁释放不会误删其他会话。
- 锁文件能用于诊断当前运行者和陈旧状态。
