# 运行控制请求会话化与启动收尾一致性计划

> 状态：待审核。本文件只记录治本方案，未修改代码或运行产物。

## 结论

`pause.request` / `stop.request` 是全局控制位，没有运行会话身份。章节入口会清旧 `stop.request`，Analyst CLI 不清，批处理又单独清。上一轮停止请求可能误停下一轮，也可能被某个入口静默吞掉，运行控制语义不一致。

## 当前证据

- `scripts/pipeline/core.py:26-27`：全局 `PAUSE_FILE` / `STOP_FILE` 固定在 runtime 根目录。
- `scripts/pipeline/core.py:229-241`：`wait_if_paused()` 只看文件是否存在，不看会话身份。
- `scripts/run_pipeline.py:1676-1684`：Analyst 分支 acquire lock 后直接运行，不清旧 stop。
- `scripts/run_pipeline.py:1696-1700`：章节分支 acquire lock 后如果 `STOP_FILE.exists()` 就直接 unlink。
- Mendel 审计还发现批处理脚本中也有独立清理 stop 的逻辑。

## 根因判断

- 控制请求是“对哪个运行会话发出的命令”没有被建模。
- 不同入口各自决定是否清理旧请求，缺少统一启动/收尾策略。
- 全局文件适合简单手动控制，但不适合多入口、多次运行、断点续跑和后台分析并存。

## 影响

- 上一次停止请求残留，可能误停下一次 Analyst。
- 章节入口清旧 stop 可能吞掉用户刚发出的停止请求。
- progress/lock 无法解释当前 pause/stop 属于哪次运行。

## 治本方案

1. 引入运行 session id。
   - 启动时生成 `session_id`，写入 lock/progress。
   - pause/stop 请求文件包含目标 session_id、created_at、reason。
2. 控制请求按会话匹配。
   - `wait_if_paused()` 只响应当前 session 或 `target=all`。
   - 历史请求不影响新会话。
3. 统一启动策略。
   - 所有入口共用 `prepare_run_session()`。
   - 对旧请求：过期清理、保留审计、或要求用户确认，不能各入口自定。
4. 统一收尾策略。
   - 正常结束清理当前 session 控制位。
   - 异常结束保留可诊断信息但不阻塞下一会话。
5. 增加测试。
   - 旧 stop 不影响新 session。
   - 当前 session stop 能在安全点停止。
   - `target=all` 可停止任意当前运行。

## 验收标准

- pause/stop 请求有明确归属，不再是无身份全局开关。
- Analyst、outline、章节入口启动/收尾行为一致。
- 控制请求不会被误吞或误作用到下一次运行。
