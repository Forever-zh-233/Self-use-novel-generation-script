# 近期事实回顾源切换修复计划

## 结论

`recent_beats_summary()` 的注释说给 arc_planner 看“已经发生了什么”，但它读取的是 `beats/chapter_*.json` 的计划字段，而不是 final/summary/ledger 的实现态。只要正文没有完全按 beat 走，下游弧线规划就可能把未兑现计划当作历史。

## 当前证据

- `scripts/pipeline/planning.py:747-758`：`recent_beats_summary()` 读取最近几章 beat 的标题、冲突、钩子，并返回给 arc_planner。
- `scripts/pipeline/planning.py:958`：`build_arc_input()` 把该结果作为“最近章节 beat 回顾”注入。
- `scripts/pipeline/planning.py:747-748`：函数注释写的是“给 arc_planner 看已经发生了什么”。
- `beats/chapter_89.json:28`：第 89 章 beat 钩子是继续朝南。
- `runtime/summaries/chapter_089.json:55`：第 89 章实际 summary 已经是与韩铮相遇后返回回春堂。

## 根因判断

- 管路把“计划回顾”和“事实回顾”混为一个输入。
- arc_planner 需要的是实现态历史，但当前使用的是 beat 计划态。
- 没有建立 final/summary/ledger 优先、beat 兜底的事实源优先级。

## 影响

- 新弧线会建立在错误历史上，安排已经被正文绕开的钩子或状态。
- 未兑现 beat 终点可能重复出现，造成原地打转。
- 实际正文新增的关键变化无法稳定进入 arc_planner 的“最近发生了什么”视野。

## 治本方案

1. 将函数拆分。
   - `recent_plan_summary()`：读取 beat，给“计划回顾”。
   - `recent_actual_summary()`：优先读取 summary/ledger/final，给“已发生事实”。
2. `build_arc_input()` 使用实现态事实源。
   - 优先：chapter delta。
   - 其次：runtime summaries。
   - 再次：Archivist recent ledger。
   - 最后才是 beat fallback，并明确标记“仅计划，未验收”。
3. 对旧 summary 缺字段情况做兼容。
4. 增加测试。
   - beat 与 summary 冲突时，arc input 中“已发生”必须采用 summary。
   - beat fallback 时必须带“计划态未验收”标记。

## 验收标准

- arc_planner 的近期历史不再默认来自 beat。
- 未兑现计划不会被包装成已发生事实。
- 事实源优先级有测试保护。
