# 阶段级续跑输入指纹修复计划

## 结论

【2026-06-07 进度】本项已开始落地最小版：`scripts/run_pipeline.py` 里已给 `draft.md`、`review.md`、`edited.md` 增加 sidecar 输入指纹校验（`*.meta.json`），只有 deps + output_hash 匹配才复用；无 meta/坏 meta/输入变化都会当缓存失效重跑。正文已落盘但缺 reviewer 判定时也会标记待复核，不再静默当作“无需返工”。

原问题：阶段级续跑复用 `draft.md`、`review.md`、`edited.md` 主要看文件存在和长度，不验证这些产物是否对应当前 beat、prompt、draft、gate 或 review。整章恢复已有验收凭证计划，但阶段内缓存仍可能错位复用。

## 当前证据

- `scripts/run_pipeline.py:592-596`：`read_cached_artifact()` 只按文件存在和最小长度判断缓存有效。
- `scripts/run_pipeline.py:699-704`：`review.md` 存在就跳过 Reviewer。
- `scripts/run_pipeline.py:738-743`：`edited.md` 存在就跳过 Editor。
- `scripts/run_pipeline.py:636-656`：`draft.md` 存在就跳过 Writer 初稿。
- `scripts/run_pipeline.py:676-695` 会基于当前 draft 重新计算 gate；但 `scripts/run_pipeline.py:699-728` 若旧 `review.md` 存在，会把旧 review 套到当前 draft/gate 上。
- `scripts/run_pipeline.py:738-766` 若旧 `edited.md` 存在，会把旧 editor 输出套到当前 draft/review/gate 上。
- `scripts/run_pipeline.py:284-286`：`ensure_beat()` 发现 `beats/chapter_N.json` 已存在就直接返回，不校验该 beat 是否对应当前 story_director、active_arcs、卷纲、台账、上一章正文等动态输入。
- `scripts/run_pipeline.py:310-327` 的 `beat_direction_check()` 只在自动生成 beat 后执行；已有/手写/缓存 beat 会绕过 story_director 方向校验。
- `scripts/pipeline/planning.py:1394-1484`：BeatPlanner 输入依赖故事总监、当前弧线、卷纲、当前状态、最近台账、上一章正文、节奏警告、威胁阶梯、配角议程等动态材料，缓存鲜度不能只看 beat 文件存在。
- `scripts/run_pipeline.py:599-612`：正文已落盘续跑时会重建 `resume_verdict()`；若 `review_verdict.json` 和 `review.md` 都缺失，则返回 `source=resume_default` 且 `needs_revision=False`，把“缺判定”当作“无需返工”。
- `scripts/run_pipeline.py:914-915`：最终会写 `writer/final.md` 与正式文章；如果恢复路径缺少 final_gate/fact_checker/archive commit 指纹，就无法证明这份正文属于当前 beat/prompt/台账版本。

## 根因判断

- 缓存复用只证明“某阶段曾写过东西”，不证明“这东西属于当前输入版本”。
- prompt、beat、gate、review 任一变化都可能让下游阶段产物失效。
- Reviewer / Editor 的缓存依赖不是单一上游文件：review 至少依赖 draft、beat、gate、reviewer prompt；edited 至少依赖 draft、gate、review、editor prompt。只按文件长度复用会把跨版本产物拼成一章。
- Beat 自身也是阶段产物；上游规划材料变化后，旧 beat 可能仍被无条件复用。
- “已有 beat”不等于“可用 beat”；使用前仍必须通过当前 story_director、active_arcs、卷纲硬承诺、时间锚点、字段 schema 的校验。
- “已有正文”也不等于“已提交正文”；恢复路径必须验证 final_gate、fact_checker、summary、archivist commit 的同源性。
- `artifact_retention=clean` 会让追溯输入更困难，因此更需要不可清理的阶段指纹。

## 影响

- 修改 beat 后，旧 review 仍可能被复用。
- 修改 draft 或 gate 后，旧 review 仍可能被复用；修改 review 后，旧 edited 仍可能被复用。
- 修改 story_director/active_arcs/卷纲/ledger 后，旧 beat 仍可能被复用，导致后续整章建在过期章纲上。
- gate 结果变化后，旧 editor 输出仍可能被复用。
- reviewer prompt 更新后，旧 verdict 仍可能被当成当前判断。
- 用户以为续跑省 token，实际可能复用错版产物。

## 治本方案

1. 为每个阶段产物保存 sidecar manifest。
   - `beat.meta.json`：beat input hash、beat prompt hash、story_director hash、active_arcs hash、volume plan hash、state/ledger relevant hash。
   - `draft.meta.json`：beat hash、writer prompt hash、writer input hash。
   - `review.meta.json`：draft hash、beat hash、gate hash、reviewer prompt hash。
   - `edited.meta.json`：draft hash、gate hash、review hash、editor prompt hash。
   - `final.meta.json`：final text hash、beat hash、draft/edited source hash、final_gate hash、fact_checker status/hash、summary status/hash、archive commit hash。
   - manifest 结构按 DVC-style lockfile 思路保存：每个 stage 声明 deps、params、outs、schema_version、input_hash、output_hash，不能只散落 sidecar 字段。
2. 续跑时校验指纹。
   - 全部匹配才复用。
   - 任一不匹配则从该阶段重跑并清理下游缓存。
   - 对已有 beat 增加 use-before validation：即使 hash 匹配，也要跑 `beat_direction_check`、schema 校验、卷纲/active_arcs 冲突门禁。
3. 指纹文件不被 `clean` 删除，或写入章节 manifest。
4. 增加 scenario 测试。
   - story_director 或 active_arcs hash 变化后不能复用旧 beat。
   - beat 文件已存在但未吸收 story_director severity>=2 纠偏时，`ensure_beat()` 不得直接返回。
   - 修改 beat hash 后不能复用旧 draft/review/editor。
   - 修改 draft hash 或 gate hash 后不能复用旧 review。
   - 修改 draft/gate/review/editor prompt 任一 hash 后不能复用旧 edited。
   - 修改 review 后不能复用旧 edited。
   - 正文已存在但缺 `final.meta.json` 或 archive commit 凭证时，不得用 `resume_default` 静默放行；必须重建缺失门禁或要求人工豁免。
   - hash 匹配时仍可复用，保证续跑收益。

## 验收标准

- 阶段缓存复用有输入指纹依据。
- BeatPlanner 章纲缓存也受输入指纹保护。
- 上游输入变化会自动使下游阶段失效。
- 断点恢复和阶段续跑共享同一套章节提交证据。
- 阶段指纹能表达“哪个上游变了导致哪个下游失效”，而不是只判断某个缓存文件是否存在。
