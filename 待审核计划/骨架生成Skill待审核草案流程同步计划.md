# 骨架生成 Skill 待审核草案流程同步计划

> 状态：待审核。本文件只记录治本方案，未修改代码或运行产物。

## 结论

`skill/骨架生成指南.md` 仍把骨架定义成“必须和用户讨论后确定、用户确认后写入正式 `全书骨架.md`”。这与总计划中“脚本自动生成高质量待审核草案，人工审批后再投入正典”的方向冲突。未来 agent 使用该 skill 时，会绕回手写锁定旧流程。

## 当前证据

- `skill/骨架生成指南.md:7`：写明“骨架不是脚本能自动决定的，必须和用户讨论后确定”。
- `skill/骨架生成指南.md:99`：层级规划体系写成“全书骨架(手写锁定,800章)”。
- `skill/骨架生成指南.md:115`：使用方式要求“用户确认后，将骨架写入 `全书骨架.md`”。
- `待审核计划/自动精品小说生成系统治本总计划.md:255-287`：当前治本方向要求全书骨架多阶段自动生成草案、schema 校验、进入待审核，不直接污染正典。

## 根因判断

- 旧 skill 是人工协作指南，未随系统目标升级到“自动生成 + 待审核 + 正典门禁”。
- Skill 对 agent 的影响很高；即使代码和总计划改了，agent 一旦读取 skill，会按旧流程建议直接写正式骨架。
- “人工讨论”应该成为审批/修订环节，而不是替代自动生成脚本。

## 影响

- 骨架自动生成脚本即使完善，也可能被操作指南绕开。
- 新书初始化仍依赖人工手写，无法稳定复制高质量骨架生成流程。
- 未经 schema/质量评审的骨架可能被直接写入 `全书骨架.md`，污染下游卷纲和弧线。

## 治本方案

1. 更新骨架生成 skill 流程。
   - 从“人工决定并写正式骨架”改为“脚本生成待审核草案，人工审批/修订”。
   - 明确 agent 不得直接把未审核草案写入正式 `全书骨架.md`。
2. 定义三层产物。
   - `draft`: LLM 多阶段生成草案。
   - `reviewed`: 人工或评审器通过的候选。
   - `canonical`: 正式投入主流程的骨架。
3. Skill 引导多阶段脚本。
   - 故事核诊断、体量合同、卷级骨架、伏笔/关系/能力表、风险审查、schema 校验。
4. 人工职责重新定位。
   - 用户确认的是草案是否晋升正典，不是从零手写全部骨架。
5. 增加文档一致性测试。
   - 扫描 skill 和 prompt，不得出现“直接写正式骨架”的旧流程描述。

## 验收标准

- 骨架生成 skill 与总计划的自动草案/待审核门禁一致。
- 未审核骨架不会被指南建议写入正式正典。
- 新书初始化能稳定走“生成草案 -> 审核 -> 晋升”的流程。
