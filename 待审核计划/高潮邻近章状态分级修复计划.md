# 高潮邻近章状态分级修复计划

## 结论

当前弧线管路把“距离高潮节点 ±1 章”直接升级为“必须关键章/必须写出爆点”。这会把高潮前铺垫章、高潮当章、高潮后余波章混成一个布尔状态，导致临近章被误当高潮章处理。

## 当前证据

- `scripts/pipeline/planning.py:1323-1333`：当 `tension` 是高潮/高且 `abs(nch - chapter) <= 1` 时，强制要求 beat 标 `关键章=true`，并写明“writer 必须写出爆点”。
- `scripts/run_pipeline.py:532-550`：`best_of_n_enabled()` 也把 beat 的关键章标记和 `in_climax_window()` 作为采样触发条件。
- `beats/_debug/第089章/beat_input.md:11`：第 89 章输入明确说“还差约1章到第90章高潮”。
- `beats/_debug/第089章/beat_input.md:18`：同一输入又要求第 89 章作为高潮节点 ±1 范围内章节“必须关键章/写出爆点”。
- `beats/chapter_89.json:8-9`：第 89 章实际 beat 是“酿 / 夜行过场”，不是高潮当章。

## 根因判断

- 管路只有 `关键章` 布尔，没有 `pre_climax / climax / aftermath` 这类阶段状态。
- 为 best_of_n 和 writer 加强质量的护栏，被复用成了叙事语义判断。
- “临近高潮要加压”与“本章就是爆点”没有分离。

## 影响

- 铺垫章会被迫写爆，抢走高潮章的情绪峰值。
- 余波章可能继续被要求爆点，导致节奏不能回落。
- best_of_n 可能在临近章误触发，额外烧 token，且选稿标准偏向过度爆发。

## 治本方案

1. 引入高潮邻近状态。
   - `climax_phase: pre_climax | climax | aftermath | none`
   - 或 `climax_distance: -1 / 0 / +1`
2. 改写 active_arcs_for_beat 文案。
   - `pre_climax`：要求加压、铺垫、收束线索，禁止提前写节点结果。
   - `climax`：要求关键章、爆点、不可逆后果。
   - `aftermath`：要求余波、代价、沉降。
3. best_of_n 触发策略分层。
   - 默认只在 `climax` 当章触发。
   - `pre_climax/aftermath` 需要显式配置才触发。
4. beat schema 增加阶段字段，Reviewer 也按阶段评判。
5. 增加 scenario 测试。
   - 第 N-1 章只能得到 `pre_climax`，不能被写成“必须爆点”。
   - 第 N 章才要求 `关键章=true`。
   - 第 N+1 章要求余波而非继续爆点。

## 验收标准

- 临近高潮章不再被误提升为高潮当章。
- best_of_n 不再因 ±1 粗窗口误触发。
- beat/writer/reviewer 对高潮前、中、后的职责一致。
