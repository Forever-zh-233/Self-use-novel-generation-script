Novel 1 测试文件维护指南
=========================

这份指南用于维护 `E:\Novel 1` 的测试系统。它仿照莲莲Bot的测试维护方式，但必须服从本项目 `agent.md` 的第一原则：**反 Goodhart**。

最重要的一句话：

```
测试可以咬住客观坏事，不能咬住创意审美。
```

代码测试只允许对这些内容有硬牙齿：客观事实错误、源文专名污染、内部编号/系统元信息泄露、无歧义 AI 腔坏词、字数硬底线、配置/索引/落盘契约破坏。句长、段长、比例、节奏、爽感、情绪强弱、人物声音好不好，只能作为数据交给 reviewer 或人类判断。


一、测试入口与职责
------------------

推荐使用项目固定 Python：

```powershell
& "C:\Users\Lenovo\AppData\Local\Programs\Python\Python312\python.exe" "E:\Novel 1\scripts\run_tests.py" all
```

| 入口 | 命令 | 职责 | 适用场景 |
|---|---|---|---|
| 语法与结构门卫 | `python scripts/run_tests.py check` | Python 语法、关键文件、JSON 配置、chunk 索引、`.gitignore` 隐私/运行态边界、Agent 守则 | 新增/移动脚本、prompt、chunk、配置后必须跑 |
| 快速契约 | `python scripts/run_tests.py quick` | 纯函数、硬门禁、Review JSON 解析、API 配置读取、状态默认值 | 改 `scripts/pipeline/*.py` 的函数、硬检查、模型配置逻辑后跑 |
| 行为链路 | `python scripts/run_tests.py scenario` | 临时工作区里的真实小链路：writer 上下文、衔接检查、archivist 合并、reviewer 输入 | 改 writer/reviewer/archivist/context/state 链路后跑 |
| 信任入口 | `python scripts/run_tests.py all` | `check + quick + scenario` | 交付前跑 |

如果 `python` 命令在 Windows 会话里不可用，使用上面的固定 Python 路径。不要用 `python -m py_compile` 当主语法检查；它会写 `__pycache__`，容易污染工作区或被权限挡住。本项目的 `check` 使用 `compile()` / `ast.parse()`，不生成 `.pyc`。


先看这个：什么时候要改测试
--------------------------

不要每次改文档或 prompt 都把测试翻一遍。先按这个表判断，能不改就不改。

| 你这次改了什么 | 要不要改测试 | 通常要动哪里 | 最少跑什么 |
|---|---|---|---|
| 只改普通说明文档、总结、计划草稿 | 通常不用 | 不动测试 | 可不跑；不放心跑 `all` |
| 只改 prompt 措辞，角色职责不变、文件名不变 | 通常不用 | 不动测试 | `check`，影响链路时加 `scenario` |
| 新增固定角色 prompt（如新 planner/reviewer 入口） | 要 | `tests/checks_test.py` 的 `REQUIRED_PROMPTS`；必要时 scenario | `check` |
| 新增 writer module | 通常要 | 若链路必须加载它，改 `scenario_test.py` 的临时素材；必要时加上下文断言 | `scenario` |
| 新增/删除/重命名 chunk | 要 | `chunks/index.json` 本身；`checks_test.py` 如新增必备类别/关键 chunk | `check` |
| 改 `chunks/index.json` schema | 要 | `checks_test.py` 的 chunk 索引断言 | `check` |
| 新增 `scripts/pipeline/*.py` 模块 | 要 | `checks_test.py` 关键文件（如是核心模块）；quick/scenario 按职责补覆盖 | `check && quick`，影响链路加 `scenario` |
| 只是从巨型脚本搬函数到新模块，行为不变 | 要维护结构测试，行为测试不一定新增 | `checks_test.py` + 已有 quick/scenario 保持通过 | `all` |
| 改 API provider/model/key 读取逻辑 | 要 | `quick_test.py` 的 API config helpers | `quick` |
| 改 `hard_gate` 客观硬规则 | 要 | `quick_test.py` 加“应拦截”和“应放行”样例 | `quick` |
| 改 `style_gate` 指标 | 视情况 | 如果只是 metrics，避免硬失败；只断言指标存在/诊断形态 | `quick` |
| 改 reviewer verdict JSON 解析 | 要 | `quick_test.py` 的 review parsing | `quick` |
| 改 writer 上下文注入 | 要 | `scenario_test.py` 的 writer context 场景 | `scenario` |
| 改 ledger/state/archivist merge | 要 | `scenario_test.py` 的 archivist merge 场景 | `scenario` |
| 改相邻章节衔接检查 | 要 | `scenario_test.py` 的 continuity guard 场景 | `scenario` |
| 改输出清理/锁/恢复逻辑 | 要 | 优先新增 isolated scenario；必要时 quick 覆盖纯函数 | `scenario` |
| 只是调试临时日志，之后会删 | 不建议改测试 | 不动测试 | 删除日志后跑相关入口 |

一句话判断：

```
项目结构变了 -> 改 check
客观规则/纯函数变了 -> 改 quick
角色链路/上下文/台账写入变了 -> 改 scenario
创意审美判断变了 -> 不写硬测试，把材料给 reviewer
```


各类行为应该改哪个测试
----------------------

| 行为范围 | 优先改这里 | 不要优先改这里 |
|---|---|---|
| Python 文件语法、关键脚本存在 | `tests/checks_test.py` | 真实跑自动写小说 |
| prompt 固定入口是否存在 | `tests/checks_test.py` | 扫 prompt 全文风格 |
| chunk 索引、chunk 文件存在 | `tests/checks_test.py` | writer 场景里靠报错发现 |
| `.env.local`、源文全文、运行态目录不进仓库 | `tests/checks_test.py` | 人工看 `git status` |
| API key/model/provider 读取 | `tests/quick_test.py` | 真实打 API |
| JSON 提取、Review verdict 解析 | `tests/quick_test.py` | reviewer 真实模型输出 |
| `hard_gate` 客观污染/泄露/坏词 | `tests/quick_test.py` | 源码字符串扫描 |
| `style_gate` 诊断指标 | `tests/quick_test.py`，只测 metrics 和无歧义坏词 | 让比例/方差成为硬失败 |
| writer 上下文、账本视图、禁止后期真相泄露 | `tests/scenario_test.py` | 全文大快照 |
| reviewer 输入包含诊断、不读取分数表 | `tests/scenario_test.py` | 真实 reviewer API |
| archivist STRUCTURED_UPDATE 解析与落盘 | `tests/scenario_test.py` | 只看函数返回、不看文件 |
| 相邻章节衔接规则 | `tests/scenario_test.py` | 风格评分 |
| run pipeline 长流程 | 先拆成 quick/scenario 小链路 | 真实连跑几十章 |


什么时候不要新增测试
--------------------

- 已有测试已经覆盖同一行为，只是实现方式换了。
- 只是改错别字、注释、标题、普通说明文案。
- 只是为了把某个审美判断“自动化”而写测试。
- 为了追求 100% 文案稳定而锁死整段 prompt 或整段模型输入。除非它是协议、JSON schema、硬约束标题，否则优先断言关键结构和 no-leak。
- 为了让某个源码字符串扫描看起来更安心。优先问：能不能用 quick 的真实输入或 scenario 的真实链路覆盖？


二、变更类型与必改项
--------------------

| 变更类型 | 必改 | 位置 |
|---|---|---|
| 新增核心 pipeline 模块 | 语法自动覆盖；若是固定核心模块，加入 `REQUIRED_FILES` | `tests/checks_test.py` |
| 新增固定角色 prompt | 加入 `REQUIRED_PROMPTS` | `tests/checks_test.py` |
| 新增 chunk 类别或必备 chunk | 确认 `chunks/index.json`；必要时加类别断言 | `tests/checks_test.py` |
| 改模型配置格式 | 调整 JSON/config 断言和 API helper 测试 | `tests/checks_test.py`、`tests/quick_test.py` |
| 改硬门禁词/正则 | 加正负样例，确保拦截和放行都覆盖 | `tests/quick_test.py` |
| 改 review JSON 输出协议 | 覆盖合法 JSON、markdown 围栏、尾逗号/容错、关键词 fallback | `tests/quick_test.py` |
| 改 writer 输入材料 | 用临时 workspace 构造 state/ledger/chunk/beat，断言注入与不注入 | `tests/scenario_test.py` |
| 改 reviewer 输入材料 | 断言有 diagnostics/正文/beat，且不读取 `输出/分数表/` | `tests/scenario_test.py` |
| 改 archivist merge 字段 | STRUCTURED_UPDATE 场景必须断言真实 `state.json` / `ledger.json` 落盘 | `tests/scenario_test.py` |
| 改运行态目录或隐私边界 | 更新 gitignore 检查 | `tests/checks_test.py` |


三、核心原则
------------

1. 行为变更优先用真实输入或临时工作区 scenario 覆盖，不要优先扫源码字符串。

2. `check` 守结构，`quick` 守纯函数和客观硬规则，`scenario` 守跨文件链路。不要把三者混在一个巨大测试里。

3. 所有会写文件的测试必须使用临时 `NOVEL_WORKSPACE`。不能写真实 `runtime/`、`输出/`、`beats/`、`.env.local`。

4. 测试不能调用真实 LLM API。需要覆盖 API 之前的输入构造、配置读取、错误解析；真实 API 连通性属于人工 smoke 或单独授权的线上验证。

5. 风格数字只做 diagnostics。任何会诱导模型“删内容、写短、写平、规避表达”的规则，都不许变成硬失败。

6. 每个硬规则都要有正负样例。只测“应拦截”会误伤，必须同时测“应放行”。

7. 断言要落到可验证事实：文件存在、JSON 可解析、字段落盘、字符串不泄露、返回 verdict 正确。不要断言“读起来更好”。


四、拆分 pipeline 文件步骤
--------------------------

拆 `scripts/run_pipeline.py` 或 `scripts/pipeline/*.py` 时，不要一口气大改到底。按这个顺序：

```
第 1 步：只创建目标模块，完整复制函数和必要常量，跑 check
第 2 步：源文件加 import / 调用接线，跑 check
第 3 步：保留旧逻辑但让新模块生效，跑 quick
第 4 步：删掉旧定义或旧分支，跑 quick
第 5 步：补/调 scenario，确认临时工作区真实链路通过
第 6 步：跑 all
```

如果某一步失败，先修这一小步，不要继续扩大改动面。

六、新增内容 checklist
----------------------

新增 `scripts/pipeline/xxx.py` 时：

- [ ] 文件能被 `check` 的 Python glob 扫到。
- [ ] 若是核心常驻模块，加进 `tests/checks_test.py` 的 `REQUIRED_FILES`。
- [ ] 有纯函数或配置解析，补 `tests/quick_test.py`。
- [ ] 会影响 writer/reviewer/archivist/context/state，补 `tests/scenario_test.py`。
- [ ] 跑 `check && quick`；影响链路再跑 `scenario`；交付前跑 `all`。

新增固定 prompt 时：

- [ ] 加进 `tests/checks_test.py` 的 `REQUIRED_PROMPTS`。
- [ ] 如果 prompt 会改变角色输出协议，补 quick 或 scenario。
- [ ] 检查 prompt 没有要求 writer 读源文全文、分数表或后期真相。
- [ ] 跑 `check`；影响链路跑 `scenario`。

新增 chunk 时：

- [ ] `chunks/index.json` 声明 `file`、`tokens`、`category`。
- [ ] `file` 指向真实 `chunks/chunk_*.md`。
- [ ] 如果是新类别，考虑 `checks_test.py` 是否要加类别断言。
- [ ] 如果 writer 必须使用它，scenario 临时 workspace 也要种同类素材。
- [ ] 跑 `check`。

改硬门禁时：

- [ ] 先问：这是客观错误，还是创意审美？
- [ ] 如果是创意审美，停止，改成 reviewer diagnostics 或 prompt 反面示例。
- [ ] 如果是客观错误，在 `quick_test.py` 加一个真实失败输入。
- [ ] 同时加一个安全放行输入。
- [ ] 不写巨型穷举正则；规则要小而明确。
- [ ] 跑 `quick`。

改 archivist/state/ledger 时：

- [ ] `scenario_test.py` 构造临时 `runtime/state.json`、`active_threads.json`、`ledger.json`。
- [ ] 输入 STRUCTURED_UPDATE。
- [ ] 断言真实文件里的字段更新。
- [ ] 断言旧字段未被无意清空。
- [ ] 跑 `scenario`。


七、测试报告四问
----------------

任何“已验证修复”的报告都必须回答：

1. 复现了用户哪条真实失败输入？
2. 断言了哪个失败现象不会再出现？
3. 哪些依赖被 mock 或隔离了？
4. 因为 mock/隔离，哪些真实链路仍未覆盖？

本项目常见回答模板：

```
复现输入：正文含“沈安翻开台账，看见 LF-001 写着第12章要回收”。
断言现象：hard_gate 必须报告内部编号/章节元信息泄露。
隔离范围：quick 测试不调用 LLM API，不写真实 runtime。
未覆盖链路：真实 reviewer 是否能解释该问题、真实自动写章节端到端输出质量。
```


八、常见陷阱
------------

1. 把 `style_gate` 的比例/方差指标升级成硬失败。这会违反反 Goodhart。

2. 场景测试 import pipeline 模块太早。`pipeline.core.BASE_DIR` 在 import 时读取 `NOVEL_WORKSPACE`，必须先进 `isolated_workspace()` 再 import。

3. 场景测试写真实 `runtime/` 或 `输出/`。所有写入测试都必须用临时 workspace。

4. 测试读取 `.env.local` 或 echo API key。测试只能验证 key 读取顺序，不能暴露密钥值。

5. 新增固定 prompt 后忘记加 `REQUIRED_PROMPTS`，导致文件丢了测试也不响。

6. 新增 chunk 文件但没更新 `chunks/index.json`，writer 实际找不到。

7. 用全文大快照锁 prompt 或 writer input。这样会让未来正常迭代很痛苦；优先断言关键标题、关键材料、no-leak。

8. 只测 happy path。硬门禁、解析器、merge 都要覆盖失败/容错路径。

9. 把 `输出/分数表/` 注入任何角色。分数表是 WRITE-ONLY，测试必须守住这个边界。

10. 为了测试通过改弱真实规则。测试红时先判断：是测试过窄、项目规则真的错、还是当前改动破坏了契约。不要为了绿而让客观防线失效。
