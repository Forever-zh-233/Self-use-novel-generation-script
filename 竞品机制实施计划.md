## 贯穿全程的四条铁律

1. **零文风/剧情硬编码** —— 风格、情绪、钩子、债务到期与否的语义判断一律归 LLM，
   代码只做"数标签 / 算章差"这类零语义统计。硬编码会让模型偷跑别的路子（Goodhart）。
2. **信号给对对象** —— **决策**给规划层（beat_planner 短期 / arc_planner 中期 / story_director 巡查）；
   **执行约束、连续性护栏**给 writer。把决策塞给 writer 单章视角 = 矫枉过正病根（三线/pacing/emotion 已踩过）。
3. **防 KPI 化** —— 一切分类（钩子型 / 情绪基调 / 三线）都是**诊断词汇**，不是**达标配比清单**。
   措辞软化（"有契机再带入，没契机继续挂着也行"），绝不要求"每章必须有某型 / 凑某比例"。
4. **token 成本按注入位置算** —— writer 基础卡 ×800章×N候选（最贵，一个 token 都不加）；
   reviewer rubric ×800×1（可控）；规划层 ×相关章（便宜）；维护文档 ×0（免费）。

## #6 reviewer 结构化输出 + 修打分 bug（地基，最先做，风险最高）

### 问题真相（读代码后确认，比"数字打架"更微妙）

- **真相①：分数根本没被代码读。** `parse_score_needs_revision`（run_pipeline.py:2295）只做关键词
  字符串匹配（"必须修改/不合格/建议重写…"），从不解析 `/60` 还是 `/30`、不读任何数字。
  所以 60/30 之争对"触发不触发 editor"**零影响**——真正的开关是 reviewer 有没有写出那几个字。
- **真相②：rubric 冗余注入且互相打架。** `reviewer.md` 这个 system prompt **本身就完整写着 12 项 /60**
  （带锚点，:41-196）；`06-验证打分表.md` 又作为 input 再注入一份 **7 项 /30**（make_review_input，:2282）。
  模型同一套标准收到两遍、数字还不一致 = 浪费 token + 污染判断。
- **真相③：关键词匹配脆弱。** reviewer 措辞稍变（写"需要修订"而非"必须修改"）→ 代码漏判，editor 不触发，
  烂章直接过；反过来"修改建议"里引用某些词又可能误触发。

### 决策（已与用户敲定）

- **统一 rubric：保留详细版（reviewer.md 的 12 项带锚点），删掉简要版（06-验证打分表.md 不再注入）。**
- **结构化输出走双轨**：reviewer 同时给 ①JSON 判定块（给代码读）②文字修改建议（给 editor 读）。
  不能纯 JSON——editor（:4384）靠文字建议修稿，需要"哪段哪行怎么改"。
- **代码读字段不猜词**：解析 JSON 里的 `needs_revision` 布尔 + 各项分数 + 阻断原因。
- **JSON 解析失败 → fallback 回退关键词匹配**（保留现有 parse_score_needs_revision 作兜底，白捡保险，零成本）。
- **分数落盘**：把结构化分数写进 artifact，供 story_director 看质量趋势（可选增益）。

### reviewer 输出的 JSON 判定块（拟定结构）

```json
{
  "needs_revision": true,
  "scores": {
    "叙事视角一致性": 5, "句式节奏": 4, "对话口吻": 4, "情绪推进": 4,
    "场景调度": 4, "原文复写风险": 5, "AI腔检测": 3, "读者体验": 4,
    "潜台词层次感": 3, "场景价值转变": 4, "内在推进": 4, "方向对齐": 5
  },
  "total": 49,
  "blockers": ["AI腔检测=3（低于4分必修）", "潜台词层次感=3"]
}
```
（代码读 `needs_revision`/`blockers`/`total`；文字修改建议另起段落给 editor。
注：钩子型不在此——#3 已改为 beat_planner 设计钩子，reviewer 不动，见下。）

### 分数的本职（澄清——用户追问"这分数本来干啥的"）

reviewer.md 名义上让分数当"返工开关"（≤22必重写等区间），但代码真相是 `parse_score_needs_revision`(:2295)
**只捞关键词、从不读分数**——那根线从来没接上。**#6 的本质就是把这根线接上**：让代码真读结构化分数+阻断项
来决定返工，而不是捞关键词。所以分数有它正经本职（返工判定），**不需要再发明第二用途**。

### 落盘（用户追问"给谁看"后收敛）

- 分数写进 reviewer artifact **纯存档**，供人事后查账（"第几章质量塌了"）。
- **绝不回灌任何角色输入**——尤其不给 story_director "看趋势纠偏"。那会踩第五次"统计信号喂错对象"的雷：
  AI腔分是散文执行层的盲评结果，规划层无力调控，只会瞎指挥；且分数变 KPI = 模型为分数写作（Goodhart）；
  且破坏 reviewer 盲评独立性（创作链里建反馈回路）。
- 原则：**验收结果是单向的，不该回头变成生产指令。**

### 边界情况与重试（用户追问后敲定，#6 在主链必须焊死）

现有 `call_role`(:671) 的 `reject_retries=3` **只对内容风控拒绝重试，不校验 JSON**。#6 面对的是"正常返回但非合法JSON"。

| 边界 | 处理 | 额外 token |
|------|------|-----------|
| ① JSON 合法 | 读 needs_revision/blockers/total | 0（主路） |
| ② markdown包裹/尾逗号 | 复用 `_sanitize_model_json`(:3046)+`extract_json_object`(:2940) 代码洗一遍再 parse | 0（纯代码，不重调） |
| ③ 洗完仍失败 | **fallback 回退关键词匹配**（现有 parse_score_needs_revision 捞原始文本） | 0（纯代码） |
| ④ 关键词也没命中 | **默认放行 needs_revision=False**（用户定），靠后续 final_gate(:4402) 兜硬伤 | 0 |

- **坏 JSON 不重调模型**（重调白烧 ~3000 输出token+网络往返）。架构哲学一致：beat 解析也是代码洗不重调(:3100)。
- **fallback 不是摆设是真第二条腿**：双轨输出里文字建议还在，关键词能从文字捞"必须修改"——这本就是现在唯一在跑的机制。
- **④默认放行的理由**：走到④需"JSON烂+文字里连'必须修改/重写'都没有"，此时 reviewer 极大概率本就想放行
  （有硬伤一定会写那些词，被③接住）；默认返工=宁可错杀，每次错杀白烧 ~7000 editor token。

### Token 净账（#6 是省钱改动）

- 新增：reviewer prompt 加"输出结构化JSON"指令 ≈ **+150** input/章。
- 省下：删 06-验证打分表.md 注入 ≈ **−600** input/章（那份冗余7项表）。
- 解析/兜底全纯代码 0 token；坏JSON不重调 0 token。
- **净：每章 input 反而少 ~450 token，输出不变。**

> 以下为权威标准，与 reviewer.md 正文一致。负向指标 = 任一项 <3 无论总分必修。

| # | 评分项 | 5分锚点 | 2分锚点 | 性质 |
|---|--------|---------|---------|------|
| 1 | 叙事视角一致性 | 全章第三人称有限视角，始终贴近沈安感知，不写他不知道的信息 | 视角混乱多处问题 / 突然切反派视角揭主角未知信息 | — |
| 2 | 句式节奏 | 长短句交替有起伏，紧张处短、平推处中句串流，有呼吸 | 通篇长段拖沓 **或** 满屏单句独段碎成节拍器（两者同扣2） | — |
| 3 | 对话口吻 | 每角色风格独特、符合声音表，蒙名能认出谁在说 | 风格混乱难区分 / 全用文绉绉古文 | — |
| 4 | 情绪推进 | 情绪靠动作细节自然传递、有感染力，章末张力高于章初 | 情绪混乱缺感染力 / 直白"他感到无比悲伤" | — |
| 5 | 场景调度 | 切换流畅、张弛有度、无冗余过渡 | 切换混乱 / 大段文字描述过渡 | — |
| 6 | 原文复写风险 | 完全无复写痕迹 | 明显复写（独特比喻/标志台词/专有桥段/语义高相似） | **负向** |
| 7 | AI腔检测 | 完全无AI腔 | 明显AI腔（点题/滥用仿佛/情绪总结/动作小尾巴/强行升华/不信任读者/碎句念经…见黑名单17条） | **负向** |
| 8 | 读者体验 | 有具体冲突/代价/小爽点或反转，章末钩子明确 | 明显水字数、解释多、爽点弱 | 低于4必修 |
| 9 | 潜台词/层次感 | 对话有表层+底层，靠岔开/沉默/借物传递真意 | 全直说，角色像汇报而非交流 | — |
| 10 | 场景价值转变 | 每场景结束情感状态明确不同于开头，靠行为体现 | 场景头尾情感状态基本相同 | — |
| 11 | 内在推进 | 角色内心有清晰变化、由具体瞬间触发 | 角色内心从头到尾没变化 | — |
| 12 | 方向对齐 | 自然服务总监批注/卷纲/短弧，不偏航也不打卡式纠偏 | 继续扩张偏航元素、挤占卷纲承诺、制造无意义债务 | 低于4必修 |

**总分 /60 区间**：54-60 优秀直用 ｜ 46-53 良好小修 ｜ 36-45 一般需较大修改 ｜ 24-35 较差建议重写 ｜ ≤22 必须重写。

**硬性必修触发（任一即 needs_revision=true）**：硬检查已报问题 ｜ AI腔<4 ｜ 读者体验<4 ｜ 方向对齐<4 ｜
原文复写或AI腔<3 ｜ 空钩子 ｜ 明显注水 ｜ 角色名/设定不一致 ｜ 解释型对话 ｜ 伪例照抄 ｜
**主角感知穿帮**（视力受限却看清精细细节，硬伤，无视总分）｜ **明显不信任读者** ｜ **碎句念经**。

## #3 追读力分型（最终形态：beat_planner 设计钩子，从"事后挑刺"变"事前设计"）

### 形态演化（三次推翻，记录决策轨迹）

1. 起初倾向"不用五型，只做假钩子布尔"——理由是空钩子检测已存在（reviewer.md AI腔#13、:250）。
2. 用户追问"你确定不用五型？我没读过竞品"——纠正方法论：不能拿"竞品有没有"当论据，且竞品仓库
   不在本地、原始 skill 描述已被压缩、无法重读原文。改从第一性推演 → 结论反转：五型该用（不同型有
   不同的"假"法，分型让诊断对症）。
3. 用户再追问"诊断完作用是什么？光摆样子？"——点破：诊断无下游 = 花瓶。
4. 用户定调"给章节规划师，让他建议本章用什么钩子"——**最终形态**：钩子型不进 reviewer，
   而是让 **beat_planner 带类型意识设计 `章末钩子`**。从反应式 QC 升级为前瞻式设计。

### 为什么给 beat_planner 比给 reviewer 对

- **reviewer 诊断 = 反应式**：钩子写完才发现假，浪费整个 writer+editor 周期才换。亡羊补牢。
- **beat_planner 设计 = 前瞻式**：写之前定好钩子型+内容，writer 一次到位。钩子是设计出来的，不是捞回来的。
- beat_planner 本就在写 `章末钩子` 字段（beat_planner.md:42），只是让它带类型意识，**不是加新活，是把现有活做精**。

### 妙用：钩子型 = 演奏弧线呼吸（pacing_shape）的乐器

接上上轮的弧线走向升级——arc_planner 产出 `pacing_shape`（"闷—紧—爆—沉"）是宏观战略，没说每章怎么实现。
**钩子型就是 beat_planner 逐章演奏这条呼吸曲线的乐器：**

| 弧线段 | 该用的钩子型 | 拉读者哪根弦 |
|--------|-------------|-------------|
| 闷（铺垫累积） | 悬念 / 渴望 | "这背后是什么"/"我想看到X" → 安静攒期待，让铺垫章也拽人 |
| 紧（逼近节点） | 危机 / 选择 | 抬高赌注 |
| 爆（高潮） | 危机 / 大兑现 | 一次性引爆 |
| 沉（余波） | 情绪 | 让冲击回响、沉淀 |

**多样性是自然副产品，不是 KPI**：钩子型跟着弧线位置走，而你永远在靠近/远离某节点，所以型自然在变，
不用"轮流用五型"强制配比——根除了"每章强制凑"的担忧。八百章不会全程危机钩把读者绷垮。

### 五型 + 各自的"假"法（写进 beat_planner.md 当设计词汇 / reviewer 现有空钩子检测继续兜底）

| 钩子型 | 真钩子 | 假钩子 |
|--------|--------|--------|
| 危机型 | 主角真陷险境、化解需代价 | 明明能轻松化解却硬撑危险感（虚假紧张） |
| 悬念型 | 抛出读者在意、且猜不透的谜 | 谜没人在意，或读者早已猜到 |
| 情绪型 | 有铺垫的情绪冲击 | 没铺垫硬煽，情绪悬空 |
| 选择型 | 两难，两个选项都合理都有代价 | 假选择，其实只有一个选项合理 |
| 渴望型 | 吊起读者真正想要的兑现 | 吊的胃口不是读者想要的那个 |

### 实施（极小改动）

- **beat_planner.md**：`章末钩子` 字段说明里加"先想本章在弧线哪段（看 pacing_shape + 进度定位），
  挑一个最贴合的钩子型，再写具体钩子内容"。附五型表 + 弧线段对应指引 + 各型的"假"法（让它避开假钩子）。
- **可选**：beat schema 加一个 `钩子型` 标注字段（writer 不必看，仅供落盘/回看），normalize_beat 保留。
  也可不加字段，纯靠 `章末钩子` 文字体现——倾向加，便于事后人工回看钩子节奏。
- **reviewer：完全不动**（用户定）。它故意盲评、看不到 beat（reviewer.md:5），不能告诉它 beat 意图否则
  破坏盲评独立性。现有"空钩子必修"（:250）继续独立把关假钩子，与 beat 设计互不污染。
- **红线（守死）**：代码**绝不写死"闷→悬念"映射**（Goodhart 陷阱）。映射只能是 beat_planner.md 文字指引，
  由 LLM 看弧线位置自己挑。代码侧零检测、零硬编码、零 token。**绝不进规划层做分布统计/轮用配比。**
- **"微兑现"不另造机器**：beat_planner 已有 `期待循环位置(埋/酿/收/缓冲)`+`本章爽点` 在管兑现节奏，不重复造轮子。
- token：beat_planner 每章一遍 prompt 加几行，×相关章，便宜；writer/reviewer 零新增。

## #5 债务合同（复用现成 ledger.obligations + 蹭 threads 管道，把"该还了"送规划层）

### 现状（读代码确认，已有三套"承诺/线索"机制，边界不同）

| 机制 | 装什么 | 字段 | 谁在读 |
|------|--------|------|--------|
| **threads (T/F-XXX)** | **剧情线索/支线**（开出去的故事问题、悬而未决的谜） | desc, owner, **last_advanced**, **plan_resolve_by**(收束计划), status活跃/休眠/已收 | ✅ **story_director 已读**（:2494），带"⚠已N章没推进"（≥6章）/"⚠无收束计划"告警 |
| **reveal_ledger** | 世界观大设定揭示节奏（揭到第几层） | topic, revealed_level, plan_next_level_in | ✅ story_director 已读 |
| **obligations (OB-XXX)** | **角色欠的债/承诺/因果**（人情、立的仇、答应的事） | desc, status悬空/已结, since_chapter | ❌ 只给 writer（:872-877），**规划层没人读** ← #5 缺口 |

### 核心区别（决策依据，用户拍板"并存"）

> **thread 问"这条线什么时候给读者交代"（叙事结构）；obligation 问"沈安什么时候兑现他的良心"（人物道德账）。**

- 同一事件常被两套同时跟踪、维度不同：「王老板女儿失踪案」作为读者追的悬念=thread；
  「沈安答应找回女儿」作为承诺=obligation。案子可以查清（thread 收），但沈安没能救回人
  （obligation 失败）= 沉重的人物代价 = 本书"深沉价值"的命根子。
- **沈安的人格就长在他怎么还这些债上**（答应治黑子娘的眼 OB-005 / 受王老板一袋米恩 OB-008 /
  立誓查师父死因 OB-011）。把它并进 threads 当普通剧情线会淹没这本"良心账"——故 **obligations 独立成账**。

### 最终设计：两套并存，但 obligations 复用 threads 已跑通的管道（不另造到期系统）

1. **obligations 并进 `threads_digest_for_director`**（现成函数 :2450）——story_director 一处看两本账：
   剧情线 + 人物债。obligations 算 `当前章 - since_chapter` 报到期（纯计数零语义）。
2. **beat_planner（短期）/ arc_planner（中期）新增** obligations 到期摘要——它俩现在连 threads 都没读，
   是真缺口。短期定"本章/本弧有契机可了结哪笔债"，中期把老债了结编进弧线节点。
