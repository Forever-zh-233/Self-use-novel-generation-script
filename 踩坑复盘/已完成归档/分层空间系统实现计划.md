# 分层空间系统实现计划（防穿帮·小地图）

## 设计哲学（与成长轨迹同源）

空间不硬编码、不全量注入。每一层在自己的**空间尺度**上生成和读取，层层递进、互相配合、动态落盘完善。低层不被高层的大尺度干扰，高层不管低层的细节走位。

```
全书骨架     → 世界地理骨架（大陆/区域/主要城邦的相对方位，北极星级）
卷纲规划师   → 本卷活动区域（这一卷在哪几个地方，相对位置，路线脉络）
弧线规划师   → 本弧场景群（这条弧用到哪些地点，它们的相对关系）
章节规划师   → 本章场景布局（这一章在哪、空间里有什么、谁在哪个位置）
写手         → 落笔时遵守本章布局 + 调出已知地点的既有布局
archivist    → 把本章新生成的空间事实落盘（地标方位/场景布局/走位变化）
门禁(gates)  → 按落盘的空间正典校验穿帮（走错/瞬移/凭空出现）
reviewer     → 对照 beat 的空间布局，判断写手有没有违背
```

## 四种穿帮 → 四个尺度的对应

| 穿帮类型 | 负责尺度 | 谁生成 | 谁校验 |
|---------|---------|--------|--------|
| 跨地点路线错（往东走到西） | 区域尺度 | 卷纲/弧线 | gates + travel_matrix |
| 地点相对方位错（镇东老井跑到镇西） | 聚落尺度 | 弧线/archivist 落盘 | gates 方位检查 |
| 场景内布局错位（左厨房变右厨房） | 场景尺度 | 章节规划师 + archivist | reviewer 对照 beat |
| 角色走位矛盾（凭空出现/消失） | 走位尺度 | 写手执行 beat | gates 走位检查 |

## 数据结构（落盘在 ledger，复用现有 entities 三档机制）

地点本来就是 `entities` 里 `type=地点` 的实体。给地点实体扩展空间字段，不新建独立文件（保持单一数据源）：

```json
"青石镇": {
  "type": "地点",
  "summary": "沈安南行第二站，暗案发生地",
  "scale": "聚落",                    // 区域/聚落/场景 三级
  "parent": "北砚县辖区",             // 上级空间（区域尺度）
  "bearing_from_parent": "北砚县以南40里",  // 相对上级的方位
  "landmarks": [                      // 聚落内地标的相对方位（聚落尺度）
    {"name": "镇东老井", "bearing": "镇东头，正街往东半盏茶"},
    {"name": "如意栈", "bearing": "镇中心正街"},
    {"name": "棺材铺", "bearing": "南巷，如意栈后巷绕行"}
  ],
  "layout": {                         // 场景内布局（仅重要场景填，场景尺度）
    "如意栈": "进门是堂屋，左手柜台，右手通后院；二楼三间客房，沈安住东头"
  }
}
```

走位（谁在哪）继续用 `state.characters[name].location`，但细化粒度到"场景内位置"（屋里/院外/门口）。

## 各层改动清单

### 1. 全书骨架（master_outline）— 世界地理骨架
- **prompt**：`master_outline.md` 输出里加「世界地理骨架」段：大陆/区域/主要地点的相对方位（只到大尺度，不写细节）
- **注入**：volume_planner 已读全书骨架，自动拿到

### 2. 卷纲规划师（volume_planner）— 本卷活动区域
- **prompt**：`volume_planner.md` 卷纲输出加「本卷活动区域」段：这一卷走哪几个地方、相对位置、路线脉络
- **注入**：已读全书骨架（拿到地理骨架）

### 3. 弧线规划师（arc_planner）— 本弧场景群
- **prompt**：`arc_planner.md` 约束加一条：弧线要标注本弧用到的地点群及其相对关系（写进 summary 或 approach_to_next）
- **注入**：新增——给 arc_planner 注入「地点正典摘要」（已登记地点的方位，从 ledger 提取，聚落级，~几百 token）

### 4. 章节规划师（beat_planner）— 本章场景布局
- **prompt**：`beat_planner.md` 加字段 `空间布局`：本章在哪个地点、场景里有什么（门/窗/关键物件位置）、出场角色初始走位
- **注入**：新增——给 beat_planner 注入「本章相关地点的既有布局」（从 ledger 调出，只给本章会用到的，按需）

### 5. 写手（writer）— 落笔遵守布局
- **注入**：新增 `build_writer_sections` 里加「本章空间布局」section（从 beat 的"空间布局"字段 + ledger 既有 layout 合并），**按需**：只在 beat 有空间布局或本章地点已有 layout 时注入
- **prompt**：`writer.md` 加一条：严格遵守空间布局，不擅自改地标方位和场景内陈设

### 6. archivist — 空间事实落盘
- **prompt**：`archivist.md` 扩展 `update_entities`，地点实体可填 `scale/parent/bearing_from_parent/landmarks/layout`
- 已有 `travel_update` 继续负责跨地点距离（保留）

### 7. 门禁（gates）— 空间一致性硬校验
- 现有 `continuity_check_adjacent` 已查时间/动作连续性，扩展加：
  - **走位检查**：beat 标注某角色在"院外"，正文却让他在"屋里"说话且没写进门 → 提示
  - **方位检查**：正文写"往东走"到了 ledger 里标注在西边的地点 → 提示
- 零 token（正则 + ledger 比对）

### 8. reviewer — 对照 beat 空间布局
- `reviewer.md`：新增检查点——正文是否违背 beat 的「空间布局」（地标方位、场景陈设、角色走位）

## Token 分层预算（核心：各层只拿自己尺度）

| 层 | 注入内容 | 尺度 | 预估 token |
|----|---------|------|-----------|
| master_outline | 无额外（自己生成） | 大陆/区域 | 0 |
| volume_planner | 全书骨架已含地理 | 区域 | ~0（已在骨架内） |
| arc_planner | 地点正典摘要（方位） | 聚落 | ~300-500 |
| beat_planner | 本章相关地点既有布局 | 场景 | ~200-400（按需） |
| writer | 本章空间布局 | 场景+走位 | ~200-400（按需） |
| reviewer | beat 已含空间布局 | 场景 | ~0（在 beat 内） |

到 800 章也稳定——因为每层只调"本章/本弧相关"的地点，不是全量地图。这跟 ledger 的实体三档是同一个抗膨胀原理。

## 实施顺序

1. 数据结构：archivist.md 扩展地点实体空间字段（先有写入）
2. 提取函数：ledger 里加 `spatial_digest_for_arc()` / `layout_for_beat()` / `layout_for_writer()`（按尺度提取）
3. 注入：arc/beat/writer 三处 build_*_input 加空间 section（按需触发）
4. prompt：master_outline / volume / arc / beat / writer 各加空间职责段
5. 校验：gates 走位/方位检查 + reviewer 对照点
6. 验证：编译 + 跑一轮确认各层空间 section 正确生成、空数据时静默

## 静默原则（避免噪音）

每个空间 section 都遵守：**没有空间数据时完全不注入**。开篇第一章没有任何已登记地点 → 所有空间 section 为空 → 不出现在输入里。只有当 ledger 真的攒下了地标/布局，下游才会读到。这跟成长骨架"弧线没标注就不提"是同一个克制原则。
