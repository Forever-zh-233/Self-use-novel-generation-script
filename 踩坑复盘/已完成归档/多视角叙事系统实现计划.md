# 多视角叙事系统实现计划（v2）

## Context

800章仙侠自动写作管线，当前运行到第14章。全书锁定沈安第三人称有限视角，但长篇需要多视角来：
- 展示主角行为对他人的真实影响（回响/涟漪）
- 制造信息差让读者窒息（暗流）
- 重新定义已发生事件的含义（镜像）
- 让配角在关键行为前被读者理解为活人（人物塑造）

## 核心设计原则

### 信息流方向：自上而下决策，自下而上执行

```
弧线规划师（决策层）
  → 决定：什么时候切视角、切谁、什么时间类型、什么叙事手法
  → 决定：什么时候埋伏笔、用什么载体、明线还是暗线
  → 输出：node 上标注完整的叙事指令

章节规划师（翻译层）
  → 读弧线给的标注，翻译成可执行的单章 beat
  → 不发明视角切换，不发明伏笔
  → 只负责"怎么把这一章写精彩"

写手（执行层）
  → 只看 beat，写正文
  → 不知道什么是伏笔，不知道为什么切视角
  → 伏笔对它来说只是"beat 要求场景里出现一把旧伞"
```

### 已确认的技术决策

- POV 章是**完整的正统章节**（2500-3500字），不是附在主线章末的小段落
- 每章只有一个 writer 调用——beat 里的 `视角角色` 字段决定走哪条路径
- 知识隔离靠**物理隔离**（POV writer call 的上下文里根本没有沈安的秘密/内心）
- 频率由 arc_planner 战略决定 + beat_planner 战术落地，不设代码硬编配额

## 一、时间线与叙事手法

### 五种时间顺序（中文叙事学）

| 手法 | 含义 | 对应场景 |
|------|------|---------|
| 顺叙 | 正常时间线 | 默认，80%+ 章节 |
| 平叙 | "花开两朵各表一枝"，同时发生的事分头讲 | 暗流型/铺垫型 POV 章 |
| 插叙 | 暂停主线，插入过去的完整场景，然后回到主线 | 回响型（"三天前那个晚上她怎么过的"） |
| 补叙 | 事后补充之前隐瞒的信息 | 镜像型（"原来当时他在想这个"） |
| 倒叙 | 先写结果再回头讲经过 | 极少用，特殊弧线开头 |

### 时间标记机制

**问题**：当 POV 章的故事时间 ≠ 当前主线时间，系统怎么告诉写手？写手怎么告诉读者？

**解法**：弧线规划师在 node 上标注时间类型，beat_planner 传递到 beat，writer 按规则处理。

beat 新增字段：
```json
{
  "视角角色": "小山媳妇",
  "时间锚点": "三天前，沈安离开后的当晚",
  "叙事手法": "插叙"
}
```

writer_pov.md 规则：
- **插叙/补叙**：章首第一段用一句话定位时间（注意要用抽象教学，避免模型无限使用同一个例句）
- **镜像型**：开头重复一个读者已经见过的具体细节作为锚（如重复沈安说过的那句话），读者立刻知道"这是那个时刻"
- **平叙**：用地点/角色切换即可，不需要时间标记（因为是同步的）
- 章末不需要特殊处理——下一章自然回到主线

### 读者定位的三层信号（调研结论）

1. **章首标记**（paratextual）：用相对时间而非绝对日期（"三天前"比"甲子年三月初七"好记）
2. **锚定细节**（diegetic）：在第一段放一个读者已知的具体细节，让读者自己定位
3. **一致性**：同一种手法每次用同样的信号方式，读者学会后就不再费脑

## 二、多视角类型

| 类型 | 作用 | 时间关系 | 触发条件 |
|------|------|---------|---------|
| 铺垫型 | 让读者先心疼这个人，主角出手时读者才有感觉 | 平叙/插叙 | 主角即将与某人产生重要互动 |
| 回响型 | 主角的善举真的改变了一条命 | 插叙（回到帮助后的那个时刻）或顺叙（几章后的现在） | 主角帮助某人后 3-8 章 |
| 涟漪型 | 主角不知道的地方，他的行为在发酵 | 顺叙/平叙 | 主角行为产生远端影响，20-50 章后 |
| 暗流型 | 读者知道危险在逼近，主角不知道 | 平叙 | 威胁酝酿期 |
| 镜像型 | 重新定义已发生的事 | 补叙（回到已叙述的那个时刻） | 某事件有隐藏的另一面 |
（批注：这里写的憧憬倒是不错，实现需要仔细）

## 三、伏笔机制修正

### 当前问题

伏笔职责分散在太多角色身上，导致：
- beat_planner 自己发明伏笔 → 没有全局视野，埋了收不回来
- writer 看到"伏笔操作"字段 → 越写越多越写越乱
- 长线伏笔资产库的"检查窗口"被 beat_planner 当 KPI → 强行外显

### 正确分工（伏笔的四层生命）

| 层级 | 关于伏笔该做什么 | 关于伏笔不该做什么 |
|------|----------------|------------------|
| 卷纲规划师 | 规划本卷的**长线伏笔**（跨卷/动人心魄/需要慢慢解开的）：埋什么、推到什么程度、预计哪卷回收、回收方式的大方向 | 不管短线伏笔（3-8章内收的小伏笔是弧线规划师的事） |
| 弧线规划师 | ①读卷纲的长线伏笔指令，作为**战术执行者**让长线伏笔在具体节点生动落地；②自主规划短线伏笔（埋+收必须成对出现在同一条弧线内） | 不能只埋不想收 |
| 章节规划师 | 读弧线给的伏笔任务，翻译成有灵魂的具体场景细节 | **不发明新伏笔** |
| 写手 | 按 beat 写场景，赋予伏笔血肉——它不知道这是伏笔，只知道"beat 要求这个场景里出现一把旧伞" | **不知道伏笔是伏笔** |

### 伏笔生命周期铁律：埋+收必须成对规划

**问题根源**：当前 ~40% 的伏笔回收率，根本原因不是"忘了收"，而是"埋的时候就没想过怎么收"。

**约束**：

**卷纲规划师**（长线伏笔）：
- 规划长线伏笔时，必须同时写明：预计回收卷、回收方式的大方向、回收时读者该有什么感受
- 格式：`"埋：XX伏笔 → 预计第N卷收 → 通过YY事件揭开 → 读者恍然大悟'原来那时候就……'"`
- 不允许只写"埋一条关于XX的伏笔"就完事

**弧线规划师**（所有伏笔）：
- `narrative_ops.foreshadowing` 里 `op: "埋"` 的条目必须带：
  - `resolve_by`：预计哪章回收（短线伏笔必须在本弧线内；长线伏笔按卷纲指令）
  - `resolve_hint`：怎么收的方向（一句话）
- 同一条弧线内埋的短线伏笔，同一条弧线内必须有对应的 `op: "收"` 节点
- 跨弧线的长线伏笔，必须标明交接给下一条弧线并带 deadline

**`overdue_foreshadowing_digest` 是兜底，不是主力**——如果上面的规则执行到位，过期警告应该很少触发。

### 伏笔的信息流：从骨架到血肉

```
卷纲规划师（骨架）
  → 读：长线伏笔资产库全貌 + 长线伏笔过期警告
  → 决定：本卷推进哪些长线伏笔、推到什么程度、预计何时回收
  → 输出：卷纲里的伏笔规划段落

弧线规划师（生动）
  → 读：卷纲的长线伏笔指令 + 长线伏笔资产库 + 全部伏笔过期警告
  → 长线伏笔：选择具体载体、场景、时机，让骨架变生动
  → 短线伏笔：自主埋+收（成对），服务于本弧线的戏剧张力
  → 输出：node 上的 narrative_ops.foreshadowing

章节规划师（灵魂）
  → 读：弧线给的 foreshadowing 列表
  → 把"周通枕头下的铜片"翻译成一个有情绪、有节奏的场景
  → 输出：beat 里的具体场景安排

写手（血肉）
  → 读：beat 里的场景要求
  → 用文字赋予这个场景温度、质感、呼吸
  → 它不知道铜片是伏笔，只知道"这个场景里沈安摸到了一片冰凉的铜"
```

### 伏笔不只是悬疑

伏笔的形态：
- 一个物件（后来引出一个人的出场）
- 一句闲话（后来变成关键线索）
- 一个场景细节（后来的事件让它有了新含义）
- 一个角色的习惯动作（后来揭示身份）

伏笔的可见度：
- **明线伏笔**：读者看到了，知道"这个东西以后会有用"，但不知道怎么用
- **暗线伏笔**：读者根本没注意到，回收时才恍然大悟"原来那时候就埋了"

### 明暗双线推进

暗线的节奏必须由弧线规划师控制：
- 什么时候露一点（让读者隐约感觉到）
- 什么时候完全潜水（读者忘了它的存在）
- 什么时候和明线交汇（爆发点）

这是结构决策，不能让章节规划师或写手自己决定"今天推一下暗线"。

### 现有 node 结构
```json
{
  "chapter": 18,
  "beat_hint": "逼近：沈安夜探周通后院",
  "tension": "高",
  "internal_shift": "沈安从怀疑变到确认",
  "approach_to_next": "..."
}
```

### 升级后 node 结构
```json
{
  "chapter": 18,
  "beat_hint": "逼近：沈安夜探周通后院",
  "tension": "高",
  "internal_shift": "沈安从怀疑变到确认",
  "approach_to_next": "...",
  "narrative_ops": {
    "pov": null,
    "foreshadowing": [
      {
        "op": "埋",
        "id": "F-018",
        "carrier": "周通枕头下的铜片",
        "visibility": "明线",
        "reader_should_feel": "这东西不对劲，但不知道是什么",
        "resolve_by": 26,
        "resolve_hint": "沈安在药铺认出同款铜片是某门派信物，反推周通身份"
      }
    ],
    "dark_thread": null
  }
}
```

当需要 POV 切换时：
```json
{
  "chapter": 20,
  "beat_hint": "回响：小山媳妇药起效后的第一个早晨",
  "tension": "低",
  "internal_shift": "从认命变到想活",
  "narrative_ops": {
    "pov": {
      "character": "小山媳妇",
      "type": "回响",
      "time_relation": "插叙",
      "time_anchor": "三天前，沈安离开后的第二个早晨",
      "purpose": "让读者看到沈安的义诊真的救了一条命",
      "seed_id": "IMP-001"
    },
    "foreshadowing": [],
    "dark_thread": null
  }
}
```

当需要暗线推进时：
```json
{
  "chapter": 25,
  "beat_hint": "...",
  "narrative_ops": {
    "pov": null,
    "foreshadowing": [
      {
        "op": "酿",
        "id": "F-010",
        "carrier": "村口老槐树上多了一道新刻痕",
        "visibility": "暗线",
        "reader_should_feel": "不注意就滑过去"
      }
    ],
    "dark_thread": {
      "thread_id": "DT-002",
      "action": "露一点",
      "how": "沈安路过时黑子对着某个方向低吼了一声，沈安没在意"
    }
  }
}
```

### `narrative_ops` 字段规则

- `pov`：null = 正常沈安视角。非 null 时 beat_planner 必须生成 POV 章 beat。
- `foreshadowing[]`：弧线规划师指定的伏笔操作。beat_planner 只能执行这里列的，不能自己新增。
  - `op`：埋/酿/收
  - `carrier`：具体载体（物件/对话/细节），不能是抽象描述
  - `visibility`：明线/暗线
  - `reader_should_feel`：读者读到这里该有什么感觉（给 beat_planner 翻译用）
  - `resolve_by`：（`op: "埋"` 时必填）预计哪章回收
  - `resolve_hint`：（`op: "埋"` 时必填）怎么收的方向（一句话）
- `dark_thread`：暗线推进指令。null = 本节点不推暗线。

## 五、章节规划师改动

### 删除的权限

- ~~自己发明伏笔~~（只能执行 arc 给的 foreshadowing 列表）
- ~~自己决定视角切换~~（只能执行 arc 给的 pov 指令）
- ~~自己决定暗线推进~~（只能执行 arc 给的 dark_thread 指令）

### 新增的输入

- `narrative_ops` 从弧线规划师传入
- 当 `pov` 非 null 时，beat 必须填 `视角角色`、`时间锚点`、`叙事手法`

### beat 输出新增字段

```json
{
  "视角角色": "沈安",
  "时间锚点": "",
  "叙事手法": "顺叙",
  "戏剧目的": ""
}
```

- `视角角色`：默认"沈安"。POV 章时填其他角色名。
- `时间锚点`：顺叙/平叙时留空。插叙/补叙时填相对时间描述。
- `叙事手法`：顺叙/平叙/插叙/补叙/倒叙。
- `戏剧目的`：POV 章必填——一句话说明这章存在的理由。

### 伏笔操作字段改为"执行确认"

现有的 `"伏笔操作": "埋[F-XXX]/酿[F-XXX]/收[F-XXX]/无"` 改为：
- 只能填 arc 给的 foreshadowing 列表里的操作
- 格式改为：`"伏笔执行": "埋[F-018]：周通枕头下的铜片，沈安摸到但没拿"`
- 不允许填 arc 没给的伏笔 ID

## 六、知识隔离（POV 章）

POV writer call 的上下文**只包含**：
- 该角色的 entity 卡（voice/facts/status）
- `ignorant_of[]`：这个人**不知道**的关键事实
- 本章主线的**公开事件摘要**（不含沈安内心/秘密）
- 世界观设定（通用）
- 本章 beat
- 时间锚点（如果是插叙/补叙）

**绝不注入**：沈安的 arc_core、self_deception、secrets、inner notes。

## 七、impact_seeds 数据结构

```json
{
  "id": "IMP-001",
  "who": "小山媳妇",
  "from_chapter": 14,
  "what": "沈安治她产后恶露，临走说'碗底有洞'",
  "pov_voice": "刚生产完、气血亏、怕拖累婆家的年轻媳妇",
  "directions": ["回响:三天后药起效后她怎么活", "镜像:她当时为什么攥着药包不说话"],
  "ignorant_of": ["沈安是盲人的秘密", "黄纸的真相", "了愿系统"],
  "best_window": "第17-22章",
  "status": "pending"
}
```

状态生命周期：pending → deployed / dropped

## 八、质量标准

一个好的 POV 章必须满足：
1. 删掉它后，读者对后续剧情的感受会变弱
2. POV 角色有自己的 want/内在矛盾（不是主角的摄像头）
3. 章内该角色发生内在位移（从A到B）
4. 这章给了读者一个主角视角永远给不了的情绪

## 九-A、过期伏笔主动推送（伏笔回收率修复）

### 问题

当前系统有期待账本记录"什么时候该收"，但没有机制在说"到时间了，必须安排回收"。F-003（愿录机制）到期 Ch10、F-004（指尖麻感来源）到期 Ch8，现在 Ch14 仍未回收。`obligations_due_digest()` 只推给 beat_planner 且语气是"有契机可考虑"——beat_planner 没有全局视野，它不敢自己决定回收。

### 解法：给弧线规划师主动推送过期警告

**新增函数 `overdue_foreshadowing_digest(chapter)`**：

```python
def overdue_foreshadowing_digest(chapter: int) -> str:
    """从期待账本提取过期/临期伏笔，生成警告文本给 arc_planner。"""
    ledger = load_ledger()
    expectations = ledger.get("expectations") or []
    
    overdue = []   # 已过期
    due_soon = []  # 3章内到期
    
    for exp in expectations:
        if exp.get("status") == "已收": continue
        deadline = exp.get("deadline_chapter")
        if not deadline: continue
        deadline = int(deadline)
        if chapter > deadline:
            overdue.append((exp, chapter - deadline))
        elif chapter >= deadline - 3:
            due_soon.append((exp, deadline - chapter))
    
    if not overdue and not due_soon:
        return ""
    
    lines = []
    if overdue:
        lines.append("⚠ 已过期未回收的伏笔（必须在本弧线安排回收）：")
        for exp, gap in sorted(overdue, key=lambda x: -x[1]):
            lines.append(f"- [{exp.get('id')}] {exp.get('desc')} — 过期{gap}章")
    if due_soon:
        lines.append("⏰ 即将到期的伏笔（本弧线内安排回收）：")
        for exp, remain in due_soon:
            lines.append(f"- [{exp.get('id')}] {exp.get('desc')} — 还剩{remain}章")
    
    lines.append("")
    lines.append("回收要求：(1)回报值得等待 (2)有代价 (3)方式出乎意料。")
    lines.append("如果判断当前剧情确实不适合回收，可以在 narrative_ops 里标注延期理由和新 deadline。")
    return "\n".join(lines)
```

**注入位置**：
- `build_arc_input()` 的 sections 里，priority = "critical"（全部伏笔的过期警告）
- `build_volume_input()` 的 sections 里，priority = "high"（只筛选长线伏笔的过期警告）

**分层逻辑**：
- 卷纲规划师只看长线伏笔（`expectations` 里 `scope: "长线"` 或 ID 在长线伏笔资产库里的）
- 弧线规划师看全部伏笔（长线+短线）的过期警告

**arc_planner.md 新增约束**：
> 看到「已过期未回收的伏笔」警告时，你**必须**在本弧线的某个 node 的 `narrative_ops.foreshadowing` 里安排回收（`"op": "收"`）。如果你判断当前弧线确实不适合回收（剧情冲突、时机不对），必须在输出里写明延期理由并给出新的 deadline 章节号——不能无视。临期伏笔优先安排，但允许顺延到下一条弧线。

### 与现有 obligations_due_digest 的关系

- `obligations_due_digest` 管的是"人物债"（沈安欠的承诺/因果），继续给 beat_planner 看
- `overdue_foreshadowing_digest` 管的是"叙事债"（埋了没收的伏笔），给 arc_planner 看
- 两者数据源不同：前者读 `ledger.obligations`，后者读 `ledger.expectations`（期待账本）
- 职责不重叠：人物债的回收时机由 beat_planner 在场景中自然安排；伏笔的回收时机由 arc_planner 在结构层决定

## 九-B、相邻章衔接一致性检查（事故率修复）

### 问题

Ch13→Ch14 出现衔接断裂：Ch13 末尾沈安已经"走了"，Ch14 开头又重新出发。这类问题 hard_gate 和 reviewer 都抓不到——它们只看单章内部一致性，不对比上下章。

### 解法：新增轻量 gate `continuity_check_adjacent`

**触发时机**：writer 生成初稿后、hard_gate 之前（成本低，只读文本不调 API）。

**检查内容**：

```python
def continuity_check_adjacent(chapter: int, draft_text: str) -> List[str]:
    """对比上一章末尾 vs 本章开头，检查衔接断裂。"""
    issues = []
    
    # 读上一章末尾（最后 500 字）
    prev_path = manuscript_path(chapter - 1)
    if not prev_path.exists():
        return []
    prev_text = read_text(prev_path)
    prev_tail = prev_text[-500:] if len(prev_text) > 500 else prev_text
    
    # 本章开头（前 500 字）
    curr_head = draft_text[:500] if len(draft_text) > 500 else draft_text
    
    # 检查 1：时间连续性
    # 上章末尾如果有明确的时间状态（天亮/天黑/几更天），本章开头不能矛盾
    
    # 检查 2：空间连续性
    # 上章末尾角色在哪里/往哪走，本章开头不能无故换地方
    
    # 检查 3：动作连续性
    # 上章末尾角色在做什么（走/睡/说话），本章不能无视
    
    # 实现方式：提取上章末尾的 state snapshot，与本章开头对比
    # 轻量版：用正则提取关键词（地点/时间/动作），检查是否矛盾
    # 完整版：调一次 LLM 做 5-point checklist（成本 ~500 tokens）
    
    return issues
```

**两种实现策略**：

| 策略 | 成本 | 准确率 | 推荐 |
|------|------|--------|------|
| 正则/关键词匹配 | 0 token | ~60%（只能抓明显矛盾） | 先上这个 |
| LLM 5-point checklist | ~800 tokens/章 | ~95% | 后续升级 |

**LLM 版 prompt（~200字）**：
```
你是衔接检查员。对比上一章末尾和本章开头，检查5项：
1. 时间：上章结束时是什么时辰？本章开头是否矛盾？
2. 地点：上章结束时角色在哪？本章开头是否无故换了地方？
3. 动作：上章结束时角色在做什么？本章是否无视了这个状态？
4. 在场人物：上章结束时谁在场？本章开头是否有人凭空出现/消失？
5. 物件状态：上章提到的关键物件（手里拿着什么/穿着什么），本章是否矛盾？

只输出有问题的项，格式：[时间/地点/动作/人物/物件] 具体矛盾描述。
没有问题输出"通过"。
```

**失败处理**：
- 发现矛盾 → 注入 writer 的修稿指令（和 hard_gate 失败一样的流程）
- 不阻断流水线，只是给 editor 一个额外的修改要求

### POV 章的特殊处理

当本章是 POV 章（视角角色 ≠ 沈安）且叙事手法是插叙/补叙时：
- 跳过与上一章的衔接检查（因为时间线本来就不连续）
- 改为检查：本章的时间锚点是否与已有正文矛盾（比如"三天前"但三天前的事件已经在别的章写过了）

## 十四、调研发现摘要

### 叙事时间（Genette 叙事学）
- 顺叙/倒叙/插叙/补叙/平叙 五种时间顺序
- 插叙 vs 补叙的关键区别：插叙暂停主线后必须回来；补叙是事后补充，不要求回到暂停点
- 镜像型 = Genette 的"重复叙事"（同一事件讲多次）
- 时间标记最佳实践：相对时间 > 绝对日期；锚定细节 > 章首标签

### 多视角（网文/Web Serial 惯例）
- Worm 模式：每条弧线结尾一个 interlude（侧面视角章）
- 网文主流：每 15-30 章一次支线视角，总占比 5-10%
- 连载读者对打断主线的容忍度低于实体书读者 → 标记要更明确

### 伏笔
- 知识隔离最佳实践：IGNORANT_OF 反向清单比正向知识清单更有效
- 伏笔 ≠ 悬疑：可以是物件/闲话/场景细节/习惯动作
- 明暗双线：暗线节奏是结构决策，必须自上而下控制

**系统能做的 vs 不能做的**：
- 能做：拉高下限（消灭不及格章）、保证伏笔不烂尾、防止衔接断裂、给多视角章提供正确的隔离上下文
- 不能做：保证灵感、保证每章都精品、替代写作本身的自然规律（高质量需要能量蓄积）
- 系统的真正价值：保证灵感来的时候不被 AI 腔污染、不被衔接 bug 毁掉、不被伏笔烂尾拖后腿
