# 项目三设计稿：analyst 结构维度 + 放开 map 死线

> 状态：**设计稿待审核**（2026-06-03）。开工前请审本稿。
> 上游计划见 [三项可优化项汇总-待审核.md](三项可优化项汇总-待审核.md) 项目三章节。

## 0. 一句话目标
让 analyst 从原文里**数出跨章/跨卷的结构区间**（弧长释放节奏、欠债还债距离、物件复现间距、反差窗口、闭环程度），
喂给 arc_planner / beat_planner，**不喂 writer**。因为 arc_planner 调用稀疏、每次决策杠杆大，
全量结构认知到位 = 一次昂贵分析摊到几十章用，边际成本趋近零。

## 1. 用户拍板的核心决策（不可推翻）
1. **不另起摘要层、不另起匿名管道。** MAP 本就逐批读完全文，其结构化产出**本身就是被压缩过的全书摘要**；
   REDUCE 把 61→41 份 MAP 并起来看，跨批的复现/回收自然浮出。一趟搞定。
2. **放开 MAP 死线**：MAP 可以随便记真名（专名/物件/桥段/章号）——因为要靠真名跨批拼接同一实体才数得出间距。
3. **消毒责任下移到 REDUCE/MERGE 出卡环节**：归纳成抽象规则时把真名洗掉，writer/规划层最终只拿到抽象区间。
4. **结构卡走独立第二通道**：独立 reduce 输出 + 独立注入路由，只进 arc/beat_planner 不进 writer，与 prose 卡分开，各自一套字数纪律。
   > **⚠ 2026-06-03 复查后推翻此条**（见 §5A）：核实消费端发现 arc_planner 是**填 JSON 模板**的、结构数字**已硬编码在 prompt 约束里**（节点隔7章/弧长20-30）。给它"贴自由文本卡"是项目二 C 的翻版（喂消费不了的格式）。**改为：analyst 数出真实区间 → 校准 arc_planner.md / beat_planner.md 的 prompt 数字与原理段落，不新增注入卡、不建注入路由。** 此条"独立第二通道"作废。
5. **MAP 批预算 40000 → 60000**（≈11章/批 → ≈17章/批，≈61批 → ≈41批）：更多短-中距埋-收在单批内闭环，跨批拼接压力更小。MAP 输入 token 不变（仍读全文），只是批数减、每批输出更重。
6. **"map 要精且好"**：MAP 每批多吐一份带章号锚点的结构台账（见 §3），REDUCE 才有东西可综合。

## 2. 现状与硬约束（已核实）
- 源文 271824.txt：575 万字符 ≈ 287 万 token，约 814 章，每章 ≈3526 token。
- 现有正式卡 3 张（情感高潮/铺垫/节奏控制）共 ≈8900 字符 ≈5900 token——**已超 reduce prompt 写的"≤2000 token"目标近 3 倍**。结构卡若混进同一份 reduce 会一起膨胀。
- `runtime/analyst/` 中间产物已清空（map/merge/reduce 全 0）→ **重跑必须从头跑满**。
- `build_arc_input`（planning.py:805）**当前完全不读 chunk** → 结构卡进 arc_planner 需新加 section。
- `split_and_write_technique_chunks`（run_pipeline.py:1247）给所有卡打 `category:"手法"`，`scan_chunk_for_contamination`（run_pipeline.py:1216）写盘前扫专名但**只警告不阻断**。
- `select_chunks`（context.py:779）是唯一注入入口，凡 index 里的卡都可能点给 writer → 结构卡必须靠 category 区分、走另一条路由，否则会漏进 writer。
- arc_planner **非每章跑**（仅弧线耗尽时），beat_planner 每章跑、max_input=50000 有余量、writer max_input=60000 是质量命门不可挤占。

## 3. MAP 改造：每批多吐一份"结构台账"（这就是"map 要精"）
现有 MAP 产出手法观察（保留）。**新增**一份带章号锚点的结构台账，附在观察之后：

```
## 结构台账（本批）
- 本批章节范围: 第NN–MM章
- 活跃弧线: [真名] 在本批处于(种子/憋/爆/收尾)哪一段；关键节点落在第几章
- 干了什么: 逐条，每条挂章号（"第48章 主角识破A的伪装"）
- 重点精彩: 本批最出彩的1-2处 + 靠什么出彩（动作/反转/沉默/物件）
- 出现的物件: [真名]@章号，逐个（铜铃@48、半块玉@52…）
- 本批悬置(开口未合): [真名] 本批抛出但本批没解释/没回收的东西 + 章号。【召回优先：宁滥勿缺，任何抛出来没解释的都记，哪怕不起眼】
- 本批回收(合上更早的口): [真名] 本批呼应/解释/回收了某个更早出现的东西 + 指向谁（"第91章 回收 铜铃@48 的来历"）
```

**铁律（写进 analyst_map.md）：**
- MAP **只记录、不下判断**。绝不准说"这是伏笔"——是不是伏笔由 REDUCE 用全书视角判定。MAP 不判断就不会误判。
- 章号锚点必须落到批内**具体章**，不能只说"这批干了X"。章号是 REDUCE 跨批拼接的钥匙。
- 物件/悬置/回收全挂**真名**（死线已放开），名字是跨批拼接同一实体的唯一钥匙。
- 悬置桶**召回优先**：假阳性可被 REDUCE 剪掉，假阴性（压根没记）不可逆。

## 4. REDUCE 改造：跨批综合 + 判定伏笔 + 消毒（结构卡专用 reduce）
独立于现有 prose reduce，新增**结构 reduce 通道**（独立 prompt + 独立输出 chunk）。
读全部 MAP 的结构台账（注意：只读台账段，token 可控），做四件事：

1. **拼接同名实体**：把"铜铃·悬置@48"与"铜铃·回收@91"按真名拼起来 → 确认伏笔、算间距=43章。
2. **判定三态（解决伏笔误判，§用户点名难点）：**
   - 悬置→有后续回收 = **确认伏笔**，记间距，进"伏笔间距分布"。
   - 悬置→全书找不到回收 = **不算进伏笔间距**（否则污染数据），单独归"刻意不回收/开放式留白"（也是合法手法）。
   - 回收→找不到更早悬置 = 埋点太隐 MAP 漏记，**唯一残余漏判**；靠 MAP 悬置桶"召回优先"把它压到最小。
3. **数结构区间**：弧长释放节奏、反差窗口、物件复现间距、欠债还债距离、闭环程度（彻底剪断 vs 留尾巴）。
4. **消毒自检（死线升级，写进 reduce prompt）**：输入故意带真名，**首要职责是逐条洗成抽象**——
   "某个不起眼随身物件，隔约40章后回收，回收时叠加新含义"。输出前逐条自查有无残留真名/原句。

## 5A. 结构 reduce 的产出：一份"校准报告"，不是注入卡（2026-06-03 定稿）
消费端核实结论：
- **arc_planner.md 是 JSON 模板填充器**，结构数字已硬编码在 prompt 约束：约束1「每弧20-30章/3节点」、约束2「两节点≥7章」、字段 `span`/`pacing_shape`/`chapter_drift`/`resolve_by`。给它贴自由文本卡 = 项目二 C 翻版（喂消费不了的格式，且与写死的数字打架）。
- **beat_planner.md 的 build_beat_input 已塞十几个 section 逼近 50000 预算**（钩子去重/场景装置去重/三线/债账/情感锚点…），其中 `recent_scene_devices_digest` 已在管"本章别重复招式"——再加"物件复现卡"职责重叠。

**所以结构 reduce 不产注入 chunk。它产一份给"人(我)"看的校准报告** `runtime/analyst/_structure_calibration.md`，内容是从原文数出的真实区间，对照当前 prompt 里的拍脑袋数字，列出建议改动：

| analyst 数出的 | 当前 prompt 写死的 | 校准动作 |
|---|---|---|
| 反差窗口真实区间（如示弱→证明底牌≈2-4章） | arc_planner 无此约束 | 新增一条约束/原理段 |
| 弧长释放真实分布（种/憋/爆各占比） | 约束1「20-30章」、`pacing_shape` 抽象描述 | 校准数字 or 补"憋占比"原理 |
| 节点真实间距分布 | 约束2「≥7章」 | 用真实分布校准这个 7 |
| 伏笔埋→收真实间距分布 | `resolve_by` 自由填、无参考区间 | 补一句"原文典型间距≈N章"供参考 |
| 闭环：剪断 vs 留尾巴比例 | `resolution_condition` 自由文本 | 补原理"几成弧留尾巴接跨卷" |
| 物件复现间距 + 叠义手法 | Phase2 motif kind/meaning（已有） | 补间距数字给 motif 机制；**原理**部分可进 beat_planner 已有的 motif 段，不新增卡 |
| 节奏轮廓（第6维度·情境化，**不下硬章数**） | arc_planner `pacing_shape` 自由文本 | 补"何种情境加速/减速/换线"的原理描述，进 pacing_shape 字段说明 |

**关键分野**：
- **能量化成硬数字的**（节点间距/弧长/反差窗口/伏笔间距）→ 校准 arc_planner.md 的**约束常量**。
- **只能当原理/手感的**（怎么让埋点回头才惊/闭环留尾巴/情境化换线）→ 补进对应 prompt 的**原理段落**（arc_planner 约束区 or beat_planner 已有相关 section），不新增注入卡。
- 所有数字都是**分布/参考**写进 prompt 当"原文典型值"，**不是 KPI 硬卡**（全局红线）；第7条节奏轮廓尤其只给"呼吸形状"不给"连跑N章必换"。
- 校准报告**含真名也无所谓**（只给我看、不进任何 build_*_input、不进 chunks/）——但我据它**手写**进 prompt 的最终文字必须洗净真名。这一步消毒由我做，不靠 LLM。

## 6. 安全前提（绝不漏进 writer / 绝不污染规划层）
1. **MAP 中间产物（脏，含真名）**只存 `runtime/analyst/`，结构 reduce 读完即归档，**绝不进 chunks/**。
2. **结构 reduce 产出的校准报告**只进 `runtime/analyst/_structure_calibration.md`，**不进任何 build_*_input、不进 select_chunks**——它是给我手动改 prompt 用的中间物。
3. **进入 prompt 的最终文字由我手写**（据校准报告），逐字洗净真名/原句——消毒责任在人，不交 LLM 自动出卡。
4. 因为不新增注入 chunk、不动 select_chunks，**writer 的输入完全不变**，零泄漏风险、零预算挤占。
5. prose reduce（现有 3 张卡）**完全不动**，本项目只动 MAP（加台账）+ 新增结构 reduce（产报告）+ 据报告改 prompt。

- **第一步(分离重构,已完成,536/0)**：
  - 新建 `config/structure_norms.json`：8 组结构数字(弧长分级/节点间距/伏笔窗口/憋占比/反差/物件/闭环/呼吸cadence)，全标"参考分布非KPI"。
  - arc_planner.md / beat_planner.md：**原理留下、具体数字删掉、改成指向输入里的「本书结构参考分布」**。修了 POV 约束里残留的"20-30章"(与新约束1打架)。
  - `planning.py` 新增 `structure_norms_digest(scope)`：读 JSON 格式化注入；arc 拿全量、beat 只拿呼吸cadence；文件缺失则优雅跳过。注入 section 可压缩(`compressible=True`)，不硬挤真约束。**纯注入、零校验/拦截/打分**(不把数字变 KPI)。
- **第二步(自动校准器,已完成,556/0)**：`scripts/calibrate_norms.py`。换书时 `跑analyst → 跑calibrate_norms → 全绿自动更新JSON`。
  - LLM 从校准报告的【可入prompt】段抽数字填固定 schema → **4 道确定性护栏**(真名扫描死线/schema齐全/范围sanity/digest可渲染) → 全过才**原子覆写**(旧文件存.bak)，否则**拿护栏报错当反馈重抽,最多3次**，3次不过保留旧JSON。
  - **agent 不被信任,护栏才是关口**——人无须审语义,只在护栏报警时介入。`--dry-run` 只抽+校验不写。
- **为何不做无人值守全自动改 prompt**：改 prompt = 改 agent 立法层，错误污染全局，且语义正确性无人(含双agent互检)能担保。D 把"换书要改的"缩到只剩可确定性校验的数字 JSON，绕开了这个无解问题。

## 8. 成本小结 + 复查揪出的误判修正（2026-06-03）
**模型物理边界（已核实，mimo-v2.5-pro）：**
- 上下文窗口 **1M token**（输入+输出共享池）；单次最大输出 **131K token**。
- 故"输出拉到 60w"做不到——会被 clamp/拒。`max_output_tokens` 是上限不是预付，设大不额外花钱（按实际生成计费），故 analyst 输出直接顶 131000。
- 输入侧 60w 在 1M 窗口内合法。**但 REDUCE 要一次吃全，则"全部 map 之和"必须 ≤ 窗口减输出余量**——靠 MAP 单批输出封 ~18000 来保证（见 §7）。

**成本：**
- **一次性**：重跑 MAP ≈287万 token 输入（不可压缩，必须读全文）+ 每批输出更重（多一份台账）。批数 61→约48（预算调大）。结构 REDUCE 一次性读全部 map ≈86万 token 输入。
- **长期每章**：**零**。结构知识以 prompt 文字形式静态写入 arc/beat_planner，不新增任何运行时注入、不碰 writer 预算。

**复查揪出的四处误判（已在上文修正）：**
1. **把结构卡当"贴给 arc_planner 读的散文"**——错。arc_planner 是填 JSON 模板的，结构数字写死在 prompt 约束里。改为"校准 prompt 数字"（§5A）。这是项目二 C 的翻版（喂消费端用不上的格式）。
2. **没读 build_beat_input 就写"新增 section 注入"**——错。它已逼近预算，且 `recent_scene_devices_digest` 已管"本章别重复招式"，物件复现卡职责重叠。改为不新增 section。
3. **"结构 reduce 必须走 hierarchical_reduce 分层"——错（最关键）。** 那个"喂不下"是基于代码里人为设的小预算（batch×0.7≈4万），不是模型物理限制。mimo 1M 窗口下全部 map 一次吃得下。**分层会切断跨批的埋-收拼接钥匙，是项目三头号敌人。改为全量一次喂、绝不分层。**
4. **"输出拉到 60w"——物理做不到**。mimo 单次输出顶 131K。改为顶格 131000（设大不花钱）；并补上"MAP 输出 × 批数 ≤ 1M"的联动约束。
