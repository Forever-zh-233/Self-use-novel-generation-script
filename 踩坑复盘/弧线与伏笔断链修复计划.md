# 弧线与伏笔断链修复计划

> 状态图例:⬜ 未开始 / 🔧 进行中 / ✅ 已完成并验证 / ⏸ 暂缓
> 纪律:每条改动先读对应代码锚点;禁硬编码 KPI(软提示交规划层/LLM 判断);禁跳层;改完跑 `python scripts/run_tests.py all` 绿了才算完。
> 工作区脏不许 `git add .`,只 add 本次相关文件。

## 背景(为什么修这些)

47 章实测暴露三大结构病,根因都在**代码层字段断链 / 消费端缺失**,不是模型写得烂:

1. **130+ 伏笔零回收**:`overdue_foreshadowing_digest`(planning.py:445)靠 `planned_resolution` 字段算 deadline,但实测 138 条伏笔**0 条带该字段**——archivist 的 upsert schema 根本不产出它,arc_planner 写的是平行字段 `resolve_by` 且从不回写。追债器结构性空转(planning.py:471-474 全部 `continue`)。
2. **主线弧缺失/全是副线注水**:`needs_arc_planning`(planning.py:646)只要"有任意活跃弧线"就不再规划。当前只剩 1 条副线 ARC-013(span 49→68)独撑 20 章,主线弧/修炼弧从第 49 章起整整 20 章无人补。
3. **POV 锁死 + 配角不灵动**:(a) `impact_seeds` 空数组 → POV 候选池空 → 几乎从不切视角;(b) `side_characters` 字段 arc_planner 产出了,但**全仓库 .py 零消费端**(grep `side_characters|independent_goal|hidden_agenda` 在 .py 中命中 0),writer 读不到配角议程。

### 对标调研结论(支撑修法,非照搬)
- 真实系统(autonovel 1068★/oh-story 1987★/dreampowers/Dramatica-Flow 等)横向去重,被 6/6/4 系统共采的三大核心手法:①中心化状态落盘 ②写审分离一致性 agent ③**叙事承诺台账+超期预警**。前两个我们已有且生效,坏的正是第三个。
- 权威理论 **Sanderson Promise→Progress→Payoff**:读者弃书首因是"承诺没**进展(Progress)**",非"没大事"。我们台账只记埋(promise)/收(payoff)两端,缺中间 Progress 追踪——这正是"注水"的本质。
- ⚠️ 红线:autonovel 用正则扫句式雷同,**违反本项目 [[anti-gaming-style-fix]] 教训,坚决不抄**;句式判断一律交 LLM reviewer。
- ⚠️ DeepSeek 调研里 Weavershift(虚构)、Pratilipi 那套架构(无据)不采用。

## 第一层:接断链(改动最小、收益最大、纯接线不造新机制)

### 任务 1:伏笔回收断链(planned_resolution 回写)✅
**根因**:arc_planner 在节点 `narrative_ops.foreshadowing[].resolve_by` 规划了回收章,但只存进 active_arcs.json,从不回写到 active_threads.json 的 F 条目 → 追债器读不到 deadline → 138/138 空转。

**改法**(planning.py `run_arc_planner`,~960 `save_active_arcs(arcs)` 之后):
- 新增 `_backfill_resolve_by_to_threads(arcs)`:遍历每条弧线 `nodes[].narrative_ops.foreshadowing`,对 `op in ("埋","酿")` 且带 `resolve_by` 的项,把 `resolve_by` 写进 active_threads.json 对应 F 条目的 `planned_resolution`(已有值不覆盖,除非新值更明确);同时把 `resolve_hint` 落进 notes 末尾(可选)。`load_active_threads` + `save_active_threads`(state.py:65,需确认有无 save 函数,无则用 dump_json(ACTIVE_THREADS_FILE))。
- 兜底放宽 `overdue_foreshadowing_digest`(planning.py:471-474):无 `planned_resolution` 时,按 `planted_chapter + 类型默认窗口`(从 config/structure_norms.json 的"伏笔回收窗口章数"取,按 strength/type 分层)推算**隐式 deadline**,使历史遗留的 138 条没 deadline 的伏笔也能被点名。隐式 deadline 仅用于预警排序,不写回文件。
- **不动 archivist 职责**(回收窗口归弧线规划师,符合现有架构分工;[[novel1-story-director-authority]])。

**验证**:跑一个 fixture——构造带 resolve_by 的 arc + 无 planned_resolution 的旧 F 条目,调用后断言①新弧线的 F 拿到 planned_resolution ②旧 F 被隐式 deadline 点名。scenario_test 加用例。

**如何做到**(已完成 2026/06/05):
- planning.py 新增 `_backfill_resolve_by_to_threads`(arc 存盘后调,把节点 `埋/酿` 项的 `resolve_by` 回写进对应 F 条目 `planned_resolution`,既有值不覆盖,`resolve_hint` 落 notes);`run_arc_planner` 在 `save_active_arcs` 后调用并打印回写条数。
- `overdue_foreshadowing_digest` 加隐式 deadline 兜底:无 `planned_resolution` 时用 `_implicit_deadline`(埋设章 + 类型窗口上界,窗口取自 config/structure_norms.json),标「估算窗口」提示规划师补真 deadline;隐式值只用于排序不写盘。`_foreshadowing_window_key` 按 type(身世/身份/命运/系统/记忆/异象 或 strength=大 → 信物命运级;情感/关系/角色/成长/托付 → 关系成长类;其余 → 任务即时类)分档。
- 实测对真实账本第51章:digest 从空串变 70 行催收报告,138 条堆债伏笔全被点名。scenario_test +10 断言,全量 679 绿。

### 任务 2:主线弧缺失(needs_arc_planning 按"线类型"判定)✅
**根因**:planning.py:646-659 只看"有没有任意弧线",不看"有没有主线弧"。1 副线就把整段判成"有弧线,不用规划"。

**改法**(planning.py `needs_arc_planning`):
- 判定升级:不仅"无弧线 / 弧线即将用完"触发,还要"**缺主线弧**"也触发——即当前活跃弧线里若没有 `type` 含"主线/主"的弧(全是副线),也返回 True。
- 配合 arc_planner.md 规则 4(最多 2 条 1主1副):让规划师在缺主线时优先补一条主线弧。需在 build_arc_input 里把"当前活跃弧线类型清单"喂给规划师,提示"你现在缺主线,本次必须产出一条主线弧"。
- 注意防抖:避免每章都判成缺主线导致反复规划(已有弧线即将用完才补)。用"无主线 **且** 距上次规划 ≥ N 章"或"无主线"单条件,实测取稳妥值。

**验证**:fixture 构造"只有 1 条副线"的 active_arcs,断言 needs_arc_planning 返回 True;构造"1主1副"断言 False。

### 任务 3:配角消费端(side_characters 落地)✅
**根因**:arc_planner.md 规则 29 产出 `side_characters[]`(independent_goal/hidden_agenda/knowledge_boundary/beat_moments),但 .py 零消费。arc_planner.md:217 白纸黑字承诺"archivist 会沉淀进 ledger 的 arc_core/secrets,writer 在主角章也能读到"——无代码兑现。

**改法**(两段接线):
- a) `save_active_arcs` 或 run_arc_planner 后:新增 `_ingest_side_characters(arcs)`,把每个 side_character 的 hidden_agenda/knowledge_boundary/independent_goal 沉淀进 ledger.entities[name] 的 arc_core/secrets(state.py:46-51 实体结构),不存在的实体先建卡。
- b) writer 输入:确认 `build_writer_input`(context.py)注入 active_cards 时,把配角的 arc_core/secrets(议程)带上——让 writer 在主角章里知道"这个配角心里揣着什么事"。POV 章(writer_pov)更要带全。
- c) beat_planner:`side_characters[].beat_moments` 指定的章,在 beat 输入里提示"本章 X 配角应基于其议程做可见动作"。

**验证**:fixture 构造带 side_characters 的 arc,断言①ledger 对应实体拿到 arc_core/secrets ②writer 输入文本里出现该配角议程关键词。

## 第二层:补真缺(挑性价比高的,严格软提示不 KPI)

### 任务 4:Sanderson Progress 追踪(治注水核心)⬜
台账从"埋/收两端"升级为三段:每条伏笔/线增加"最近一次 Progress(推进)是第几章"。借现成的 thread `last_advanced`(planning.py:148 已对 T-XXX 工作)思路,扩到 F-XXX。连续 N 章无 Progress → 软提示规划层"这条承诺读者快忘了,要么推一步要么收"。
**注**:不是新建机制,是把已生效的 thread 沉睡预警模式复制到伏笔。先做完任务1再评估是否合并。

### 任务 5:story_director "世界重量"维度 ⬜
story_director.md 现 9 维,加第 10 维:连续 N 章无任何角色面临真实生命危险 → **软提示**下一弧考虑安排生死/暴力选择节点。不滥杀、不设 KPI,只提示。一句话成本。配合骨架转折点类型(任务 6)。

### 任务 6:骨架转折点类型字段 ⏸(依赖任务 5,暂缓)
