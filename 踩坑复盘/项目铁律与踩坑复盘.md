# 自动写小说管线 · 项目铁律与踩坑复盘

> 读者对象:接手或协助这个项目的任何 AI / 工程师。
> 目的:把我们在这个管线上踩过的坑、为此立的规矩一次讲清。**动手改代码、改提示词、加约束之前,先读完本文。**
> 工程位置:`E:\Novel 1`。管线主代码 `scripts/run_pipeline.py` + `scripts/pipeline/` 包;角色提示词 `prompts/*.md`;设定文档在根目录(`01-`~`15-` 编号 md);测试 `python scripts/run_tests.py all`。
> 这是一个**单用户、单机、顺序执行**的本地管线。不要按分布式/多租户/高并发系统的标准来设计它——那是这个项目最常见的过度设计来源。

---

## 〇、一句话总纲

这是一条**层层递进的流水线**:全书骨架 → 卷纲 → 弧线 → 章节(beat) → 写手 → 评审 → 修稿 → 记录员。
每一层只做自己该做的事,信息**层层消化后**再往下传。两条命脉贯穿始终:
1. **代码只做有唯一正确答案的客观判断;一切创作/审美判断交给 LLM。**
2. **通用职责里不准写死当前这本书的内容。**
违反这两条,就是我们反复出事的根源。

## 一、最高铁律(违反必出事,无例外)

### 铁律 1:不准硬编码"创作判断",尤其不准用 code 抓具体句式/词表/魔法阈值
- 文风毛病(替读者拍板的裁决句、碎句念经、"不是 X 是 Y"成瘾、字幕式标注)**绝不能用代码去匹配具体句式或词表**。
- **为什么:** 这是典型的 Goodhart / specification gaming。你 gate 一抓"不是 X 是 Y",写手就换"并非 X 实为 Y""倒装句"等无数马甲——指标绿了、病没治、文章反而更做作。活的反面教材:曾经的 `style_gate` 只罚"短句比例 < 0.25",模型就拼命刷短句把指标刷绿,碎句念经反被系统**奖励**。
- **怎么做:**
  - **代码 gate 只做客观穿帮**:专名污染、境界倒退、时间线穿帮、库存矛盾——这类有唯一正确答案,不存在"好马甲"(李平安就是李平安),硬编码安全。
  - **文风/审美一律交 LLM reviewer**(它本来每章都跑):以资深读者视角判断"是否替读者下结论/碎句念经/节奏单一/描写超出主角感知"。LLM 看**意图**,抗马甲。低分触发 editor 软修。
  - **治本靠提示词写原则**,不靠枚举禁句:writer.md 写"判断和情绪由读者自己生成,你只摆事实和画面",**绝不列禁句式黑名单**。讲原则→所有马甲一起消失;列黑名单→逐个换马甲。
  - 唯一可保留的代码文风信号:句长方差这类**分布特征**(要刷绿就必须真写出长短起伏,无法换词规避),且只 warning、不判决。
- 同源反例:`chapter_satisfaction_check` 曾用"beat 关键词命中率 < 20%"判断"转折没写出来"——主角凭肌肉记忆先动了、转折其实写到了只是没照抄 beat 原词,就被误判,还被当成"脚本硬检查结果"注入 reviewer 污染它的独立判断。**已删,只保留无歧义的字数下限(正文 < 1800 字疑似截断)。**

### 铁律 2:通用职责里不准写死当前这本书
- 这是一个目标"换 config + 设定文档就能换书写"的通用软件。**当前书《沈安》的角色名、地名、境界词、系统名(了愿/愿录)、主角是盲人这件事,都不准写死在通用代码和通用提示词里。**
- **为什么:** 写死 = 耦合 = 换书即崩或静默污染。和"把文风规则硬编码成具体句式"是同一种病——换个上下文就失效甚至误导。
- **典型雷区(换书直接崩或静默出错,优先级最高):**
  - POV 路由用 `视角角色 != "沈安"` 来判断要不要走配角视角写手 → 换书后正常主角章会误走 POV 分支。
  - `current_mc_realm()` / 能力体系读死 `entities.沈安`,fallback 静默回退到当前书境界词("叩门")→ 掩盖换书故障。
  - writer 装配**无条件**注入"盲感官"模块(沈安专属)→ 非盲主角被强加盲人写作规则。
  - `safe_world_bible_for_writer()` 读完设定后又追加写死的"晏朝/巡夜司/妖祟"。
  - `_system_absence_warning()` 写死"了愿/愿录/青石镇"关键词。
  - `compact_state.py` 写死别名映射、`name == "沈安"` 才更新地点。
- **怎么做:** 这些常量应来自一份 `book.config.json`(地基,目前还没建)。**A 类(换书直接崩)必须抽离;B 类(不改也能跑,只是不优雅)可缓。** 注意分清哪些是真硬编码污染,哪些是"为遥远未来镀金"。
- **手法卡 / craft chunk 里也不准混入正典专名**:从参考书提炼的是"手法"(craft),不是当前书的"正典"(canon)。`chunk_情感回响手法.md` 这类卡里混进"沈安/长生者"就是污染。craft 与 canon 必须命名空间隔离。

### 铁律 3:层层递进,不跳层直塞;代码只搬运,不替角色做判断
- 每层只读上一层消化后的产物。**代码层只负责"把正确的文档喂给正确的角色",不负责"替角色做创作判断"。**
- **为什么(两次真实教训):** (1) 曾把成长阶段硬编码进 Python 直接注入 beat_planner,**绕过了弧线规划师的消化**;(2) 故事核把主角定义成"已经被磨平的成年人",导致所有下游角色都写出"满级心态"。根源都是不尊重层级分工。
- **怎么做:**
  - 链条:骨架/config 定大方向 → 卷纲规划师展开里程碑 → 弧线规划师细化到"这几章怎么走" → 章节规划师只看弧线产物 → 写手只看 beat。
  - 若某信息要被多层读到,正确做法是**让上层消化后写进自己的输出,下层读上层输出**,而不是代码同时塞给所有层。
  - **角色设定(故事核)定义的是"起点"不是"终点"。** 起点就写成终点,所有下游都会写出满级角色。
  - 笔锋随人长:前期少年心气、中期沉稳、后期飘忽遗憾。这不是角色设定,是文风本身的演变。

### 铁律 4:喂知识给某个角色前,先核实它的消费端怎么消费
- 设计"给 X 角色加点什么"之前,先 Read `build_X_input`(怎么拼输入)+ `prompts/X.md`(它是填 JSON 模板还是读自由文本、已经写死了哪些约束)。**不核实就假设它能用我给的格式 = 反复踩同一个坑。**
- **核心分野:注入 ≠ agent 指令。**
  - **长期恒定的结构知识**(节点间隔、弧长区间、境界顺序)→ 写进 agent 的 **prompt 文档**(约束常量 + 字段说明 + 原理段)。
  - **运行时注入的 chunk**(`select_chunks` 点亮的散乱 md)→ 只配"按章动态、需关键词点亮、因章而异"的东西(场景卡/角色卡/钩子卡)。
- **为什么(两次同源踩坑):** (1) 想把 reviewer 的 12 维文笔评分喂给 arc_planner——那些维度受众是写手,arc_planner 用不上,撤回;(2) 设计了 7 张"结构 chunk"运行时贴给 arc_planner——但它是**填 JSON 模板**的 agent,结构数字早已写死在 `arc_planner.md` 约束里,贴自由文本要么打架要么消费不了。用户原话:"注入根本不是散乱的 md 文档,你需要写到 agent.md 里面去。"

### 铁律 5:严禁用"已被清理/重跑的旧产物"当 bug 证据
- 当前 `active_arcs.json`、`卷纲/` 都已经是**第二卷(ch102-200)**。第一卷旧 runtime 里的冲突样本现网已不存在。
- **为什么:** 批量审计时多份"修复计划"拿旧第一卷的 beat_input / 旧卷纲冲突来"证明"有 bug,实际现网文档早已自洽。据此动手 = 修一个不存在的问题、甚至改坏已对的逻辑。
- **怎么做:** 任何"我发现 X 文件里有冲突"的结论,**先 grep 现网文件确认还在**,再谈修。计划自报的行号也常与现网不符,以 grep 实测为准。

## 二、"工程降噪" vs "替创作下结论":这条线必须分清

代码能不能管,取决于这件事**有没有唯一正确答案**。

| 可以硬编码(工程降噪,只发 warning 或修语法) | 必须交 LLM(创作判断) |
|---|---|
| 正文 < 1800 字 → 疑似截断 | 转折/承诺有没有在正文落地 |
| 专名污染(写错角色/地名)、境界倒退、时间线穿帮、库存矛盾 | 文风是否做作、是否替读者下结论 |
| 句长方差这类**分布特征**(刷绿必须真写出起伏) | 节奏是单一还是有起伏 |
| AI 腔黑名单做 warning、尾逗号/空钩子词净化语法 | 描写是否超出主角感知边界 |
| JSON schema 字段缺失/类型不对 | 配角是否"活着"、情绪出口是否单一 |

判断口诀:**这个检查能不能被"换个写法"绕过?** 能 → 它是可钻指标,交 LLM。不能(李平安永远是李平安)→ 硬编码安全。

## 三、按角色/模块分类的具体踩坑案例

### http / 网络层(命门级)
- **`http_post` 无总时长超时 → 通宵跑永久挂死。** `urllib.request.urlopen(timeout=)` 只约束单次 socket 操作,不约束整体响应时长。模型非流式(`stream:false`)时服务端要全生成完才回第一字节,只要 TCP 不死 timeout 永不触发——实测一次 archivist 卡死 16 分钟,进程卡在 recv 回不到 stop 检查点,通宵跑直接挂死。
  - **症状识别:** 某次 call_role 耗时远超 `request_timeout_seconds`(默认 240s)且 input 很小。之前所有"STRUCTURED_UPDATE 疑似被截断"的诡异,根子都在这,**不是 JSON 解析问题**。
  - **修法(已落地):** daemon `threading.Thread` 跑请求 + 主线程 `Event().wait(hard_timeout)` 兜底,`hard_timeout = int(timeout*1.5)+30`,超时抛专用 `RequestTimeout`(继承 RuntimeError)。**不能用 `concurrent.futures`**——它的 atexit handler 会 join 未完成线程,卡死的请求反过来卡死进程退出;裸 daemon 线程随主进程直接丢弃。
  - **超时必须重试,不能一次判死**(东西不能不记录):纳入 `call_role` 重试循环,退避 5/10/15s,连续 `reject_retries`(默认 3)次才抛"停在本章"。
  - **超时停章不丢账:** 正文在 archivist 之前就落盘,停章不推进 `latest_chapter`;下次启动 `recover_state_from_completed_articles` 发现"正文在但台账落后"会用正文重新调 archivist 补记。
  - **强杀卡死进程后必须手动清** `runtime/novel_pipeline.lock`、`novel_pipeline.bat.lock`(taskkill 不会自动清锁)。

### analyst(参考书提炼)
- **REDUCE / MAP 重试漏接:** 结构/merge/prose 三处裸调 `call_model` 没包重试,风控一拒就把"拒绝语"当报告写盘。已补统一重试包装 + calibrate 防线。
- **静默丢批:** 41 批只归并 38 批——计数与落盘脱钩 + 401 误判被永久 SKIP。已补 REDUCE 前完整性校验 + 失败按类型分流。
- **archivist 截断根因不是 token 不够:** 是"同预算空转重试 + 从不读 `finish_reason`"。识别截断后应加倍预算重试。
- **提炼产物只进"手法"命名空间**,不进当前书 canon(见铁律 2)。

### archivist(记录员)
- **JSON 净化用老正则猜引号会有重叠消费盲区** → beat/archivist 解析失败。已改**字符级结构状态机**,4 个消费端共享。
- **token 膨胀根因 = append-only 的伏笔/期待账本被全量 dump 进规划层。** 已改有界摘要,114k → 50k。**这是 token 治理的核心教训:长历史只走有界摘要 + 按需检索,绝不全量塞。**
- **STRUCTURED_UPDATE 命名空间不统一:** prompt 示例把 `technique_new`/`impact_seeds` 放 JSON 顶层,但 merge 只从 canon/ledger 读 → 模型照示例输出就**静默丢盘**。最危险的"假成功"。
- **`validate_archivist_report` 太弱:** 必填字段(`dominant_strand`/`cultivation_active`/`timeline_update`)漏填则三线计数器/时间线**静默停更**。校验应集中加强。
- **职责输入缺口:** prompt 让记录员读 `15-长线伏笔资产库.md`/弧线 JSON,但 `make_archive_input` 根本不注入 → 职责悬空,模型被要求做看不到依据的事。

### 规划层(volume / arc / beat)
- **`needs_arc_planning` 必须同时判两件事:** "上一节点是否已过" **且** "当前章是否落在某条弧线的 span 范围内"。只判前者会让 arc_planner 生成远期弧线后,中间几十章**裸奔无弧线覆盖**(曾经 ch31-71 共 37 章无弧线 → 无节点 → beat 永不标关键章)。
- **字段名断链(改动时高发):** prompt 写 `chapter`、数据存 `ch`、消费端读 `chapter` → 恒命中 0,功能完全失效(`in_chapter_angles` 多角度提示就栽在这)。三线节奏同理:写入端写 `dominant`、渲染端读 `strand` → 输入全是问号。**改字段名/加字段时务必三端对齐:prompt 定义、实际落盘、消费端读取。**
- **系统/金手指断档:** arc_planner 把系统耦合进"修炼"("修炼本弧按兵不动"),导致核心金手指 72 章只出现 9 次。修法:beat 层新增**独立于修炼锚点**的"系统了愿"字段——行医日常章正是了愿事件最自然的发生场景。
- **卷边界事故:** 目标卷身份/101 孤章/单槽覆盖/历史幂等多个问题其实是同一事故的切面,**应合并成一个"卷边界事务"处理**,不要各开一套 `volume_plan_hash`。

### writer / reviewer / editor
- **POV 分支当初为知识隔离复制了一条窄管路** → 防重复、写作模块、知识边界、标题兜底四处与主管路分叉。教训:复制管路 = 复制维护债,应抽"writer 通用安全层"。
- **editor 必须知道 AI 腔和 writer 的全部约束。** 否则修稿手会把干净句子改回"不是 X 是 Y"。editor.md 要同步 writer 的反例与修法。
- **reviewer 的待审正文不能标 `compressible=True`** —— 核心审查材料被摘要后,reviewer 就不是在评原文了。
- **反 Goodhart 在 reviewer 阈值上也适用:** 负向指标 `<3` 与 `<4` 两套阈值并存会让模型同分不同判。

### 主流程门禁
- **硬门禁不能退化成日志。** 曾出现:`fail_on_final_gate` 默认 False 且 config 没配、satisfaction 降成 warning、`type_guard_check` 恒返回 passed、FactChecker 残留穿帮"接受现状"继续推进。结果明知不合格的正文照样落盘、污染下一章记忆。**所有正文改写后应统一过 final_gate,hard issue 默认阻断。**

## 四、断点续跑 / 崩溃安全(通宵跑的前提)

- **正文先落盘,台账后记。** 任何阶段崩溃,要么这次重试成功、要么下次启动补记,不丢数据。
- **章内断点续跑:** 阶段产物落盘则复用,断哪续哪,不推倒重来(`resume_partial_chapter`)。
- **`latest_chapter` 是水位真值。** 半提交的章(正文在、台账未完成)不能被当成已完成章;消费摘要时按 `chapter <= latest_chapter` 过滤,避免"幽灵历史"。
- **强杀进程后手动清锁文件**(见上)。

## 五、token 治理是一等约束

**质量提升靠"更准的信息 + 更强的门禁",不是靠把所有材料塞进提示词。**
- 长历史只走**有界摘要 + 按需检索**,绝不全量 dump(append-only 账本全量塞曾把规划层撑到 114k)。
- 正文全文只给"必须看全文"的角色;其余给片段/摘要/manifest。
- 伏笔/关系/技能/角色卡按"本章相关 + 临期 + 最近变动 + 高强度"选择性注入。
- Best-of-N 默认关闭,只在关键章/高风险章启用。
- `archivist` 的 `max_output_tokens` 不要给最大值——它输入小、是抽取增量,给太大既拖慢又放大截断面。

## 六、工作流约束(怎么干活)

- **改任何 bug / 压缩恢复后,先读对应的设计/优化文档章节再动手**,不要凭记忆。
- **测试:** `python scripts/run_tests.py all`。改完必须回归(基线约 700+ 全绿)。Python 路径 `/c/Users/Lenovo/AppData/Local/Programs/Python/Python312/python.exe`。
- **环境:** Windows + bash shell。中文输出到控制台会乱码 → **写进临时文件再读**,别直接 echo/print 中文到终端。
- **git:** 工作区脏,**禁止 `git add .`**,只 stage 明确要提交的文件。
- **诊断真伪:** 任何"发现 bug"先 grep 现网代码/文件坐实(铁律 5),计划自报的行号不可信。
- **方案要配得上问题:** 这是单机顺序管线,真 bug 往往一两行能修。警惕"建状态机 / 全套 manifest / event_key / 会话化"这类对本项目的镀金——AI 批量产出的修复计划普遍**诊断对、方案过重**,落地时砍到最小可执行子集。
- **改动时三端对齐字段名**(prompt 定义 / 落盘 / 消费),这是改规划层最高发的断链来源。

## 七、动手前自检清单(每次改动过一遍)

1. 我要加的检查,**能被"换个写法"绕过吗?** 能 → 别硬编码,交 LLM reviewer。
2. 我要写的常量/逻辑,**换一本书还成立吗?** 不成立 → 它属于 `book.config.json`,不属于通用代码/通用 prompt。
3. 我要喂给某角色的知识,**它的 `build_X_input` 和 `prompts/X.md` 真能消费吗?** 恒定知识 → 写进 prompt;因章而异 → 才考虑注入。
